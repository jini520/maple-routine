## 철학
로컬 우선(local-first) MVP. 백엔드/로그인 없이 기기 안에서 완결되는 최소 구현을 우선 선택하고, 알림·백그라운드 사운드처럼 웹만으로 불가능한 기능만 네이티브 브릿지에 의존한다. Nexon Open API 연동([[ADR-007]])도 사용자 개인이 발급받은 API 키를 기기에 저장해 직접 호출하는 방식이라 이 원칙을 벗어나지 않는다. 게임 밸런스에 따라 자주 바뀌는 수치 데이터는 로직과 분리해 유지보수 비용을 낮춘다.

### ADR-001: Capacitor 기반 하이브리드 앱 채택
**결정**: React + Capacitor로 웹 코드를 Android/iOS WebView 앱으로 패키징한다.
**이유**: 웹 코드베이스를 거의 그대로 재사용하면서 `@capacitor/local-notifications`, Background Runner 등 공식/커뮤니티 플러그인으로 일간·주간 알림과 사냥 타이머 백그라운드 사운드를 구현할 수 있다. Cordova 대비 유지보수가 활발하고, 커스텀 네이티브 WebView 대비 두 플랫폼 코드를 따로 관리할 필요가 없다.
**트레이드오프**: 완전 커스텀 네이티브 UX/성능은 포기한다. 앱스토어 심사, 네이티브 빌드 파이프라인(Xcode/Gradle) 관리 비용이 순수 웹 배포보다 추가된다.

### ADR-002: Vite 기반 React SPA (Next.js 미사용)
**결정**: Next.js App Router 대신 Vite + React 기반 SPA로 개발한다.
**이유**: WebView 안에서 실행되는 앱이라 SSR/SEO가 불필요하다. Capacitor는 정적 빌드 산출물을 그대로 패키징하는 방식과 궁합이 좋고, 빌드/배포 파이프라인이 단순해진다.
**트레이드오프**: Next.js의 파일 기반 라우팅, API 라우트, 이미지 최적화 등은 사용하지 않는다. 라우팅은 React Router 같은 별도 라이브러리로 직접 구성해야 한다.

### ADR-003: 백엔드 없이 기기 로컬 저장소만 사용 (MVP)
**결정**: 별도 서버/DB 없이 기기 로컬 저장소(Capacitor SQLite 또는 Preferences)에 모든 데이터를 저장한다. 사용자가 openapi.nexon.com에서 개인적으로 발급받은 Nexon API 키도 이 로컬 저장소에 저장하고, 앱이 이 키로 Nexon Open API를 직접 호출한다([[ADR-007]]).
**이유**: 로그인·서버 인프라 없이도 개인 기록 관리라는 핵심 가치를 제공할 수 있어 MVP 범위를 최소화할 수 있다.
**트레이드오프**: 기기 변경/앱 삭제 시 데이터가 유실될 수 있고 멀티 디바이스 동기화가 불가능하다. 추후 백업(export/import) 또는 클라우드 동기화로 확장 가능하도록 저장소 접근을 `storage/` 레이어로 추상화해둔다.
**확정 — SQLite vs Preferences 하이브리드 채택 (2026-07-11)**: 단순 key-value 데이터(Nexon API 키, 선택된 계정, 일간/주간 화면의 추적 캐릭터 목록, 스케줄 동기화 캐시)는 `@capacitor/preferences`로 충분해 그대로 쓴다. 반면 앞으로 만들 보스 수익 기록(캐릭터+보스+난이도+주차 unique key로 upsert, [[ADR-008]] 멱등성 요구사항)과 물욕템 드랍 히스토리(날짜 범위로 "이번 주/월간 추이" 조회, PRD 핵심 기능 4)는 실제 쿼리(WHERE/날짜 범위/upsert)가 필요해 Preferences로는 비효율적이다 — 이 두 기능을 구현하는 시점에 `@capacitor-community/sqlite`를 도입하는 **하이브리드**로 확정한다. 지금 당장 기존 Preferences 기반 저장소를 SQLite로 전면 마이그레이션하지 않는다. SQLite 도입 시 웹 환경 테스트를 위한 `jeep-sqlite` 폴리필도 함께 검토해야 한다.
**검토했다가 기각한 대안**: Nexon Open ID(OAuth 로그인) 연동 시 authorization code → access token 교환에 서버가 필요해 이 ADR을 일시 수정하는 방향으로 검토했었다. 하지만 Nexon Open ID는 "메이플스토리 파트너스" 승인이 필요한 제한된 기능이고(사용자 미신청 상태), 실측 결과 스케줄러 Open API가 개인 API 키 + ocid만으로 호출 가능함이 확인되어([[ADR-007]]) 이 대안은 기각하고 원래 결정을 유지한다 (2026-07-09).
**진행 상황 — SQLite 도입 시작 (2026-07-11)**: 위 하이브리드 결정의 "이 두 기능을 구현하는 시점"이 도래해, 보스 수익 계산기(핵심 기능 4) 1차 구현의 첫 step으로 `@capacitor-community/sqlite` + 웹 테스트용 `jeep-sqlite` 폴리필을 도입한다. 이 시점에는 보스 수익 기록(`boss_profit_records`, 캐릭터+보스+난이도+주차 unique key upsert) 테이블만 만든다 — 물욕템 드랍 히스토리 테이블은 물욕 아이템 드랍(핵심 기능 5)이 별도로 구현될 때 추가한다.

### ADR-004: 서버 푸시 없이 로컬 알림만 사용
**결정**: 일간/주간 스케줄 알림은 Capacitor Local Notifications로 기기에서 직접 예약한다. FCM/APNs 등 서버발 푸시는 사용하지 않는다.
**이유**: 백엔드가 없는 로컬 우선 아키텍처와 일치하며, 리셋 시각·사용자 지정 알림 시각을 기기에서 미리 예약하는 것만으로 요구사항을 충족한다.
**트레이드오프**: 기기 재부팅, 강제 종료, OS 배터리 최적화 정책에 따라 알림이 지연되거나 누락될 수 있다. 특히 iOS는 예약 가능한 로컬 알림 개수에 제한(64개)이 있어 스케줄링 전략(예: 오래된 예약 정리, 앱 실행 시 재예약)을 별도로 설계해야 한다.
**미완료 판단 기준 — 확정 (2026-07-09)**: [[ADR-007]] 도입 이후 일간/주간 미완료 여부 판단은 알림 발송 예정 시각에 **백그라운드에서 Nexon API를 실시간 재호출**해 정확한 최신 상태를 확인한 뒤 결정한다(단순 로컬 캐시 기준이 아님, 정확도를 우선). 이는 순수 "예약 후 무조건 발송"이 아니라 "예약된 시각에 백그라운드 작업을 트리거 → API 재확인 → 조건부로 알림 표시" 패턴이 필요함을 뜻한다 — Android는 WorkManager로 네트워크 호출 후 조건부 알림 발생, iOS는 BGAppRefreshTask/BGProcessingTask를 활용하되 OS가 정확한 시각 실행을 보장하지 않는다는 플랫폼 제약이 있다. 재호출 시점에 API 실패(오프라인, 429 등)하면 마지막 캐시 기준으로 폴백한다([[ADR-008]]).
**트레이드오프(추가)**: 배터리·네트워크 사용량이 "예약 후 무조건 발송" 방식보다 늘어난다. iOS의 백그라운드 실행 시간 제약상 정확히 예약 시각에 재호출이 안 될 수 있어 완벽한 실시간성은 보장 못 한다(베스트 에포트).
**64개 한도 초과 시 우선순위 정책**: 캐릭터 수 × (일간 1개 + 주간 1개) 알림이 64개를 넘어설 수 있으므로, 초과 시 (1) 리셋 시각이 가장 임박한 항목 우선, (2) 동일 시각이면 마지막으로 앱을 연 캐릭터 우선 순으로 예약하고 나머지는 예약하지 않는다. 예약 못 한 항목이 있으면 설정 화면에 "일부 알림이 예약되지 않았습니다" 안내를 표시한다([[ADR-008]]).

### ADR-005: 사냥 타이머는 상시 표시 알림 + 주기적 사운드로 구현
**결정**: 타이머 시작 시 네이티브 타이머 앱처럼 "경과 시간이 표시되는 상시 알림"을 띄우고, 그 알림이 떠 있는 동안 지정 주기(1분/2분 또는 사용자 지정 N분)마다 알림음을 재생한다. 플랫폼별 구현은 다음과 같이 분리한다.
- **Android**: Foreground Service를 실행하고, `Notification.Builder.setUsesChronometer(true)`로 경과 시간을 OS가 자동 갱신하는 알림을 표시. 동일 서비스 내에서 주기 타이머를 돌려 알림음 재생.
- **iOS**: `Live Activity`(ActivityKit, iOS 16.1+)로 잠금화면/Dynamic Island에 경과 시간을 표시. 사운드 재생은 Live Activity와 별도로 로컬 알림 스케줄 또는 백그라운드 오디오 세션으로 트리거.
- 두 플랫폼 모두 Capacitor 공식 플러그인이 이 기능을 지원하지 않으므로, **Swift/Kotlin으로 커스텀 네이티브 플러그인을 직접 작성**해 `native/hunting-timer` 모듈로 노출한다.
**이유**: "네이티브 타이머 앱처럼 알림창에 항시 떠 있고 경과 시간도 표시"라는 요구사항은 단순 백그라운드 사운드 재생을 넘어, OS가 보장하는 상시 알림 UI(Android Chronometer 알림 / iOS Live Activity)가 있어야 신뢰성 있게 구현된다.
**트레이드오프**: 공식 플러그인만으로 해결되지 않아 커스텀 네이티브 개발·유지보수 비용이 발생한다. iOS Live Activity는 **iOS 16.1 이상**에서만 동작하므로 구버전 iOS 사용자는 동일 UX를 제공받지 못한다(대체로 로컬 알림 + 사운드만 제공하는 폴백 필요). 정식 구현 전 기술 스파이크(PoC)로 두 플랫폼 모두 실현 가능성과 배터리 영향을 검증해야 한다.

### ADR-006: 게임 레퍼런스 데이터는 코드와 분리된 데이터 소스로 관리
**결정**: 주간 보스 목록(`weekly-bosses.json`), 결정 가격표(`boss-crystal-prices.json`), 보스별 아이템 드랍 테이블(`item-drop-table.json`)은 `src/data/`의 JSON으로 분리하고, 실제 수치는 사용자(도메인 전문가)가 확정해 채운다.
**이유**: 게임 패치로 자주 바뀌는 정보를 로직과 분리해두면 다음 패치 때 코드 변경 없이 데이터만 업데이트할 수 있다. AI가 정확하지 않을 수 있는 실제 게임 수치를 임의로 하드코딩하는 위험도 피할 수 있다.
**트레이드오프**: 초기 출시 시 정확한 데이터가 채워지기 전까지는 계산기/드랍 기록 기능이 placeholder 데이터로만 동작한다.
**진행 상황**: 사용자가 데이터를 확정해 세 파일 모두 반영 완료. 가격은 사용자가 지정한 시세표 값을 채택(보스별 상세 정보 쪽 금액은 폐기), 갱신일은 주간/이벤트 주간 보스 `2026-06-25`, 월간 보스(검은마법사) `2026-07-01`로 일괄 표기. 힐라(하드)·핑크빈(카오스)는 일간 보스로 격하되어 세 파일에서 모두 제거. 벨로나(전 난이도)는 미출시 보스라 `status: "unreleased"`로 표시하고 가격/아이템은 `null`로 보류. 파티원 수별 가격은 `floor(priceMeso / 파티원 수)` 공식으로 확정. 메이린(이벤트 주간 보스)은 "강력한 힘의 결정"을 드랍하지 않고 대신 황금 메소 주머니(1개=1000만 메소)를 드랍하므로, 가격은 주머니 총 개수 × 1000만으로 `boss-crystal-prices.json`에 반영(`2026-07-09`)하고 `item-drop-table.json`의 황금 메소 주머니 항목은 제거함. 카이는 시즌이 종료된 레거시 보스라 `weekly-bosses.json`의 이벤트 주간 보스 목록에서 제외(`2026-07-09`).
**파티 입장 인원 제한 (2026-07-09 추가)**: 보스 입장 가능 인원은 기본 1~6인이나 예외가 있다 — 스우 익스트림 2인, 최초의 대적자·찬란한 흉성·림보·발드릭스·유피테르 3인, 메이린 1인. `boss-crystal-prices.json`의 `partySizeScaling.defaultMaxPartySize`(6)와 각 entry의 `maxPartySize`(예외가 있는 보스만)로 반영. 수익 계산기의 파티원 수 입력 검증에 사용.
**참고**: [[ADR-007]] 도입 이후 `weekly-bosses.json`의 역할은 "앱 내 보스 선택 UI용 목록"에서 "Nexon API 응답의 보스명/난이도 표기(영문)를 우리 데이터(한글)와 매핑하기 위한 참조 테이블"로 바뀐다. `boss-crystal-prices.json`·`item-drop-table.json`은 API와 무관하게 기존 용도(수익 계산, 물욕템 기록) 그대로 유지.

### ADR-007: 일간/주간 콘텐츠·보스 진행 상태는 Nexon Open API로 동기화
**결정**: 일간 스케줄러, 주간 스케줄러(퀘스트+보스)의 진행 상태는 사용자가 앱 안에서 수동으로 체크하지 않는다. 대신 Nexon Open API의 "스케줄러 Open API"(2026-06-25 출시)를 통해, 게임 클라이언트의 "스케줄러" 기능에 실제 등록된 항목과 그 완료 여부를 그대로 동기화해 읽기 전용으로 표시한다. 앱 자체의 보스 선택 UI(예: 기존에 검토했던 "최대 12마리 선택" 화면)는 두지 않으며, 게임 내 스케줄러 등록 상태가 유일한 기준이다.
**인증 방식**: 사용자가 [openapi.nexon.com](https://openapi.nexon.com)에서 개인적으로 애플리케이션을 등록해 발급받은 API 키를 앱 설정 화면에 직접 입력한다. NEXON Open ID(OAuth 로그인)는 사용하지 않는다 — 별도의 "메이플스토리 파트너스" 승인이 필요한 제한된 기능이고(2026-07-09 확인, `openapi.nexon.com/support/notice/2726294/`), 사용자가 이 프로그램에 신청하지 않은 상태다. 실측 결과 개인 API 키만으로 다음 흐름이 정상 동작함을 확인함(2026-07-09, 사용자 실측):
1. API 키 등록 직후 `GET /maplestory/v1/character/list`로 계정 소속 캐릭터 목록을 자동 조회(캐릭터명+월드 수동 입력 폼 없음, 2026-07-09 확인)
2. 각 캐릭터의 ocid → 스케줄러 Open API 호출로 `daily_contents`/`weekly_contents`/`boss_contents` 획득
**이유**: 사용자가 게임과 앱에 같은 내용을 이중으로 입력해야 하는 문제, 그리고 수동 입력이 실제 게임 상태와 어긋날 위험을 원천적으로 없앤다. OAuth 없이 개인 API 키만으로 동작이 확인되어 [[ADR-003]](백엔드 없음)을 그대로 유지할 수 있다.
**제약**:
- 응답에는 게임 클라이언트의 "스케줄러" 기능에 실제로 등록된 항목만 포함된다. 사용자가 게임에서 먼저 등록하지 않은 콘텐츠/보스는 이 앱에서도 보이지 않는다 — 온보딩 시 이 전제 조건을 안내해야 한다.
- 최근 14일치 기록만 조회 가능하다. 월간 수익 추이 등 장기 히스토리는 계속 로컬(`storage/`)에 누적 저장한다.
- 2026-06-25에 막 출시된 기능이라(확인 시점 기준 2주 차) 커뮤니티 문서가 거의 없고 스펙이 바뀔 수 있다.
- 개인 API 키는 사용자 본인의 Nexon 개발자 계정에 귀속되며(공용 비밀키 아님), 유출 시 영향 범위가 그 사용자 본인의 API 호출 한도에 한정된다 — 그래도 평문 노출은 피해야 하므로 `storage/`의 보안 영역(Keychain/Keystore 또는 암호화된 Preferences)에 저장한다.

**이용약관 준수 사항 (2026-07-09, `openapi.nexon.com/ko/support/terms/` 확인)**:
- **API 키 비공유**: 이용자는 발급받은 API 키를 타인에게 제공·공개·공유할 수 없다(약관 제5조①). 우리 앱은 각 사용자가 본인 키를 본인 기기에만 입력하는 구조라 이 조항과 자연히 부합한다 — 절대 앱에 공용 키를 내장하면 안 됨.
- **출처 표기 의무**: API 서비스 이용 결과물에 "NEXON Open API를 이용한 것임"을 명시해야 한다(약관 제6조④). 앱 footer에 상시 노출하는 것으로 확정(2026-07-09, 아래 "확정 — 온보딩 및 이용약관 표기" 섹션 참고).
- **영리 목적 이용 제한**: 회사 승낙 없이 API 데이터를 영리 목적으로 이용할 수 없다(약관 제6조⑥). 현재 MVP는 무료 개인용 도구라 문제 없으나, 추후 유료화/광고 등을 고려한다면 별도 승낙 절차가 필요하다.
- **데이터 재배포 제약**: 결과 데이터를 약관이 허용한 범위를 넘어 무단 복제·저장·가공·배포할 수 없다(약관 제5조⑤). 우리 앱의 로컬 저장은 사용자 본인의 개인 기록 목적이라 통상적 이용 범위로 보이나, 법률 자문이 필요한 영역은 아니라 확정적으로 단언하지는 않음.

**전제 재검토 (2026-07-09) — 검증 안 된 위험**:
- ~~멀티 Nexon 계정 문제~~ — 정정 및 재확인 완료(2026-07-09): 앞서 "사용자는 단일 계정만 써서 리스크 없음"으로 정리했던 건 **실측 데이터로 반증됨**. 실제로 `character/list` 응답을 보면 `account_list`에 서로 다른 `account_id` 2개가 존재하고 각각 별도의 캐릭터 목록을 가진다. 사용자 설명: "하나의 넥슨 계정에 여러 메이플 ID가 존재할 수 있음" — 즉 걱정했던 "서로 다른 Nexon 로그인 계정(별도 API 키 필요)" 문제가 아니라, **하나의 Nexon 계정·API 키로 여러 "메이플 ID"(게임 계정)의 캐릭터가 한꺼번에 조회되는 게 정상 동작**이다. 이건 이제 v1 범위 밖 리스크가 아니라 **반드시 설계해야 하는 핵심 흐름**이 됐다 — 아래 "계정(메이플 ID) 선택" 항목 참고.
- **게임 내 스케줄러 등록이라는 전제의 실효성**: "스케줄러" 기능 자체가 2026-06-25 출시라 대부분의 유저가 아직 써본 적 없을 가능성이 높다. 이 앱을 처음 켜는 유저 상당수가 "게임에서 아무것도 등록 안 해서 앱이 텅 비어 보이는" 상태를 겪을 수 있다 — 이건 버그가 아니라 예상된 첫 화면 상태이므로, 빈 상태(empty state) UI와 온보딩 안내가 필수다.
- ~~리셋 타이밍 정합성 미검증~~ — 확인 완료(2026-07-09): Nexon 서버는 한국시간(KST) 기준 목요일 00:00에 초기화되며, `lib/reset-clock`이 계산하는 일간 00:00 KST / 주간 목요일 00:00 KST 리셋 시각과 정확히 일치한다.
- **API 성숙도 리스크**: 2주 차 API라는 것은 단순히 "문서가 부족하다"를 넘어, 스펙이 실제로 바뀔 수 있다는 뜻이다. 이 기능을 "베타" 취급하고, 동기화가 계속 실패할 경우 "게임 내에서 직접 확인해주세요" 같은 폴백 안내 문구를 준비해야 한다 (에러 핸들링은 [[ADR-008]] 참고).

**실제 응답 스키마 예시** (2026-07-09, 사용자 제공, 캐릭터 "낟낟" 조회 결과 일부):
```json
{
  "date": "2026-07-09T00:00+09:00",
  "character_name": "낟낟",
  "world_name": "엘리시움",
  "character_level": 293,
  "character_class": "렌",
  "daily_contents": [
    { "content_name": "몬스터파크", "type": "contents", "registration_flag": "true", "now_count": 7, "max_count": 14, "quest_state": null }
  ],
  "weekly_contents": [
    { "content_name": "에픽 던전 : 악몽선경", "type": "contents", "registration_flag": "true", "now_count": 5, "max_count": 0, "quest_state": null },
    { "content_name": "[메이플 유니온] 주간 드래곤 퇴치", "type": "quest", "registration_flag": "false", "now_count": 0, "max_count": 0, "quest_state": "0" }
  ],
  "boss_contents": [
    { "content_name": "검은 마법사", "difficulty": "extreme", "cycle": "bossMonthly", "list_order_no": 53, "registration_flag": "true", "complete_flag": "true" }
  ],
  "weekly_boss_clear_count": 0,
  "weekly_boss_clear_limit_count": 0
}
```

**필드 주의사항**:
- `registration_flag`/`complete_flag`는 boolean이 아니라 문자열 `"true"`/`"false"`로 온다. 파싱 시 문자열 비교 필요.
- `difficulty`는 영문 소문자(`hard`, `extreme` 등)로 온다. `src/data/`는 한글 표기(`하드`, `익스트림`)를 쓰므로 매핑 테이블이 필요하다.
- `content_name`의 보스명이 `src/data/weekly-bosses.json` 등의 표기와 띄어쓰기가 다를 수 있다(예: API `"검은 마법사"` vs 우리 데이터 `"검은마법사"`, 반대로 API `"블러디퀸"` vs 우리 데이터 `"블러디 퀸"`). **공백 방향이 일정하지 않으므로, 조인 전 양쪽 문자열에서 공백을 전부 제거하고 비교해야 한다** — "API 쪽에 공백이 더 있다"처럼 한쪽으로 가정하면 틀린다(2026-07-09, 실제 응답으로 재확인).
- `cycle`은 확정: `"bossDaily"` / `"bossWeekly"` / `"bossMonthly"` 세 값 중 하나다(2026-07-09, 사용자 실측). 이 앱은 **`bossWeekly`와 `bossMonthly`만 사용**하고 `bossDaily` 항목은 무시한다 — 힐라(하드)·핑크빈(카오스)처럼 원래 주간 보스였다가 일간으로 격하된 보스들이 실제로 `bossDaily`로 온다는 게 이 응답으로도 확인됨([[ADR-006]] 데이터 정리와 일치).
- **이름 별칭 필요 — "시즌 보스 메이린"**: API는 메이린을 `content_name: "시즌 보스 메이린"`으로 반환한다(공백 제거만으로는 못 잡는 접두어 차이). `src/data/weekly-bosses.json`의 `메이린` 엔트리와 매핑하려면 별도 별칭 테이블 항목이 필요하다.

**확정 (2026-07-09)**:
- 일간/주간 화면은 완전 읽기 전용 — API 값만 표시, 앱 내 수동 체크 없음
- 알림 발송 판단은 마지막 캐시가 아니라 **알림 시각에 실시간 API 재호출**로 확인 (세부 사항은 [[ADR-004]] 참고)
- `weekly_boss_clear_count`/`weekly_boss_clear_limit_count`의 의미: 메이플스토리에는 "캐릭터당 주간 보스는 최대 12마리(난이도 조합 단위)까지만 처치 가능"이라는 게임 자체 제한이 있고, 이 두 필드가 정확히 그 카운트(현재 처치 수/상한 12)다. 단, "시즌보스"(현재는 카이·메이린뿐, 카이는 [[ADR-006]]에 따라 시즌 종료로 목록에서 이미 제외)는 이 12마리 제한에서 예외라 카운트에 잡히지 않는다. 샘플 응답에서 값이 `0`이었던 건 그 응답에 포함된 완료 보스(검은 마법사)가 월간 보스라 애초에 주간 카운트 대상이 아니었기 때문 — 필드 자체는 미스터리가 아니었고 샘플 데이터 선택의 문제였다(2026-07-09 확인).

**확정 — 캐릭터 목록 조회 및 응답 스키마 (2026-07-09, 사용자 실측)**: `GET /maplestory/v1/character/list`로 캐릭터 목록을 자동 조회한다. 캐릭터명+월드를 수동 입력하는 등록 폼은 두지 않는다.
```json
{
  "account_list": [
    {
      "account_id": "da9b2f2...",
      "character_list": [
        { "ocid": "50119a0...", "character_name": "내옆에최성일", "world_name": "베라", "character_class": "아크메이지(썬,콜)", "character_level": 211 }
      ]
    },
    {
      "account_id": "69e3525...",
      "character_list": [
        { "ocid": "23be5de...", "character_name": "낟낟", "world_name": "엘리시움", "character_class": "렌", "character_level": 293 }
      ]
    }
  ]
}
```
**확정 — 계정(메이플 ID) 선택 UI 필요**: 응답이 `account_list` 배열이라는 것 자체가 "하나의 Nexon 계정 안에 여러 메이플 ID가 있을 수 있다"는 게임 시스템을 그대로 반영한다. 따라서:
1. API 키를 처음 입력하면 `character/list`를 호출해 `account_list`를 받고, 항목이 2개 이상이면 **어느 메이플 ID를 쓸지 선택하는 화면**을 보여준다(예: 각 계정의 캐릭터 수·대표 월드명으로 구분 표시 — `account_id` 자체는 해시라 사람이 읽기 어려움).
2. 선택된 계정의 `character_list`만 앱에 캐릭터로 등록한다.
3. 설정 화면에서 **나중에 계정을 변경**할 수 있어야 한다(확정, 사용자 지시 2026-07-09).
4. ocid 문자열 길이가 계정/캐릭터마다 다르게 관찰됨(예: 32자 vs 65자) — 고정 길이로 가정하지 말고 불투명한 식별자 문자열로만 다뤄야 함.

**확정 — 스케줄러 API 엔드포인트 (2026-07-09, 사용자 제공)**: `GET /maplestory/v1/scheduler/character-state` (ocid 파라미터로 호출, 응답 스키마는 위 샘플 참고)

**확정 — API 호출 방식·제한 (2026-07-09, 공식 가이드 `openapi.nexon.com/ko/guide/`·FAQ 확인)**:
- 실제 API 호출 도메인은 문서 사이트(`openapi.nexon.com`)와 다른 `https://open.api.nexon.com/`이다. 예: `https://open.api.nexon.com/maplestory/v1/...`
- API 키는 HTTP 요청 헤더 `x-nxopen-api-key`에 담아 전송한다.
- 호출 제한은 애플리케이션 단계에 따라 다르다: **개발 단계** 초당 5건 / 일 1,000건, **서비스 단계** 초당 500건 / 일 20,000,000건. 서비스 단계로 전환하려면 API Key를 새로 발급받아야 한다(신규 발급, 기존 키 재사용 아님).
- 호출량 초과 시 에러 코드 `OPENAPI00007`, HTTP 상태 `429`("Too Many Requests")를 반환한다 — [[ADR-008]]의 429 처리 정책과 일치.
- **개발 단계(1,000건/일) 기준으로도 개인 사용에는 충분할 것으로 추정**: 캐릭터 수 개 × (일간/주간 동기화 + 알림 시각 실시간 재확인)을 감안해도 하루 수십 건 수준이라 여유가 크다. 다만 5건/초 제한이 있으므로, 여러 캐릭터를 동시에 병렬 호출하지 않고 약간의 간격을 두고 순차 호출하도록 구현해야 한다.

**확정 — 온보딩 및 이용약관 표기 (2026-07-09)**:
- 온보딩 흐름: API 키 발급 안내는 앱 내 단계별 위저드가 아니라, **openapi.nexon.com 링크 제공 + 간단한 샘플 이미지 첨부 + 설명 문구**로 안내한다(가벼운 안내, 전용 튜토리얼 아님).
- NEXON Open API 이용 출처 표기(약관 제6조④, 위 "이용약관 준수 사항")는 **앱 footer**에 넣는다.
- 계정(메이플 ID) 선택 UI: 각 `account_id`는 그 계정 내 **가장 레벨이 높은 캐릭터의 닉네임 + 직업 + 레벨**로 표기한다(예: "낟낟 · 렌 Lv.293"). `account_id` 해시 자체는 화면에 노출하지 않는다.

### ADR-008: 에러 핸들링 및 복원력 정책
**결정**: 이 앱이 의존하는 외부 요소(Nexon Open API, 로컬 저장소, OS 알림/타이머 API)는 모두 실패할 수 있다고 가정하고, 각 실패 모드마다 "마지막으로 알려진 정상 상태를 보여주고, 사용자에게 무엇이 실패했는지 명확히 알리고, 자동 재시도는 제한적으로만 한다"는 원칙을 따른다.

**Nexon Open API 호출 실패**:
- 네트워크 없음/타임아웃/5xx: 마지막 동기화 캐시를 그대로 표시하고, 화면에 "마지막 동기화: n분 전 · 새로고침" 표시. 자동 재시도는 하지 않고 사용자가 새로고침 버튼을 눌러야 재시도(백그라운드에서 무한 재시도로 배터리를 소모하지 않기 위함).
- 401/403(키 무효·삭제·재발급됨): 캐시는 유지하되 "API 키가 유효하지 않습니다 — 설정에서 다시 입력해주세요" 배너 표시. 무효 키로 반복 호출하지 않는다.
- 429(rate limit, 에러 코드 `OPENAPI00007`, [[ADR-007]] 참고): 지수 백오프로 다음 자동 동기화 시점을 늦추고, 사용자가 새로고침을 연타해도 최소 간격 내에는 실제 호출을 보내지 않는다(클라이언트 측 쓰로틀링). 여러 캐릭터를 동기화할 때도 병렬 호출 대신 초당 5건(개발 단계 기준) 이내로 순차 호출한다.
- ocid 조회 실패(캐릭터명/월드 불일치, 캐릭터 없음): 해당 캐릭터 등록 단계에서 즉시 에러 표시, 저장하지 않는다.
- 응답 스키마 불일치(신규 필드 추가, 필드 누락, 타입 변경): 파싱은 필드 단위로 방어적으로 처리한다 — 한 항목 파싱 실패가 전체 동기화를 실패시키지 않고, 해당 항목만 "표시 불가"로 건너뛴다. `src/data/` 매핑 테이블에 없는 보스명은 "알 수 없는 콘텐츠"로 원문 그대로 표시(앱이 죽지 않도록).
- **응답 자체가 JSON이 아닌 경우**(WAF/CDN 차단 페이지, 502 HTML 등): 필드 단위 파싱 이전 단계의 실패이므로, JSON 파싱 자체를 try/catch로 감싸 "네트워크 없음/타임아웃/5xx"와 동일하게 마지막 캐시 유지로 처리한다(응답 스키마 불일치와는 별도 분기).
- **HTTP 타임아웃 값**: 10초로 확정한다(2026-07-09). 공식 문서에 권장값이 없어 모바일 API 호출 관례상 채택.
- **호출량 한도**: 개발 단계 한도(1,000건/일)는 신경 쓰지 않기로 확정한다(2026-07-09) — 이 앱은 사용자가 이미 다른 서비스로 승인받은 서비스 단계 API 키(초당 500건/일 2,000만 건)를 그대로 사용하므로, 개발 단계 호출량 상한 계산이 더 이상 필요하지 않다.

**로컬 저장소 실패**: 쓰기 실패(디스크 부족 등) 시 사용자가 방금 한 조작(체크, 아이템 획득 기록 등)이 반영 안 됐음을 즉시 알린다 — 조용히 무시하지 않는다.
- **온보딩 중 API 키 저장 실패**: Keychain/Keystore 쓰기가 실패하면(권한 문제, 저장 공간 등) 온보딩을 완료 상태로 넘기지 않고 "키 저장에 실패했습니다, 다시 시도해주세요" 표시 후 재시도를 유도한다 — 이 단계는 앱의 유일한 진입 게이트라 일반 저장 실패보다 명시적으로 다뤄야 한다.
- **로컬 DB 스키마 마이그레이션(미정)**: 앱 업데이트로 로컬 저장소 스키마가 바뀌는 경우의 마이그레이션 전략이 아직 없다. 실제 스키마가 확정되는 시점에 별도 결정 필요.

**알림/타이머 네이티브 실패**: OS가 알림 권한을 거부하면 기능 진입 시점에 권한 요청 및 거부 시 안내. Android Foreground Service가 배터리 최적화로 종료되거나 iOS Live Activity 시작이 실패하면([[ADR-005]]), 상시 알림 없이도 최소한 완료 알림/사운드는 로컬 알림으로 폴백한다.
- **권한이 나중에(런타임 중) 취소되는 경우**: 최초 요청 시점 거부뿐 아니라, 사용자가 OS 설정에서 언제든 알림 권한을 끌 수 있다. 알림 예약/재확인 시점에 권한 상태를 다시 확인하고, 취소돼 있으면 예약을 건너뛰고 앱 내 배너로 "알림 권한이 꺼져 있습니다" 안내한다.
- **로컬 알림 64개 한도는 사냥 타이머 폴백 알림과 일간/주간 알림이 OS 레벨에서 같은 풀을 공유한다**(iOS/Android 모두 앱 단위 전역 한도). [[ADR-004]]의 64개 한도 우선순위 계산 시 사냥 타이머용 예약분도 포함해야 한다.

**크래시/에러 리포팅 — opt-in (확정, 2026-07-09)**: [[ADR-003]]의 "서버에 사용자 데이터를 저장하지 않는다"는 원칙과, 실제 프로덕션에서 발생하는 크래시·동기화 실패를 개발자가 전혀 알 수 없다는 현실적 문제 사이의 트레이드오프다. 설정 화면에 "익명 오류 보고" 토글을 두고, 기본값 OFF로 시작해 사용자가 명시적으로 켠 경우에만 Sentry(또는 유사 서비스)로 스택트레이스·에러 유형만 전송한다. 캐릭터명·API 키 등 개인 식별 가능 정보는 전송 페이로드에서 제외한다. 이 예외는 [[ADR-003]]에 "서버 없음" 원칙의 일부로 이미 인정된 Nexon API 직접 호출과 마찬가지로, 우리가 만든 백엔드가 아니라 제3자 SaaS로의 옵트인 전송이라는 점에서 원칙을 벗어나지 않는다.

**데이터 정합성 — 참조 무결성**: `src/data/`에서 보스가 제거되는 경우(예: 카이, [[ADR-006]])에도 그 보스를 참조하는 과거 로컬 기록(수익/드랍 히스토리)은 삭제하지 않는다. UI는 참조 테이블에 없는 보스명을 만나면 에러를 던지지 않고 "(더 이상 지원하지 않는 콘텐츠)" 같은 표시로 과거 기록을 그대로 보여준다.

**멱등성 — 중복 동기화**: 같은 주 안에 앱을 여러 번 열어 같은 보스의 `complete_flag: true`를 여러 번 관측해도, 보스 수익 기록은 캐릭터+보스+난이도+주차 조합당 1건만 생성한다(중복 생성 방지, 이미 있으면 갱신).

**이유**: 이 앱이 의존하는 두 외부 시스템(Nexon Open API, OS 네이티브 API) 모두 이 세션에서 처음 다뤄보거나(스케줄러 API 2주 차) 플랫폼 제약이 있는(iOS Live Activity 16.1+, 로컬 알림 64개 한도) 요소라, 실패를 예외 상황이 아니라 정상적으로 발생하는 상태로 다뤄야 한다.
**트레이드오프**: 모든 실패 모드에 대한 UI 상태(로딩/에러/빈 상태/부분 실패)를 각 화면마다 설계해야 해서 초기 구현 범위가 늘어난다. 그러나 이걸 생략하면 "동기화가 안 되는데 이유를 몰라 게임을 다시 켜보는" 식의 나쁜 사용자 경험으로 이어진다.

### ADR-009: 직업(전직) 기반 테마 시스템 도입
**결정**: 캐릭터 직업 고유 컬러를 Primary 색상으로 쓰는 다중 테마 시스템을 도입한다. `docs/UI_GUIDE.md`의 색상 파생 규칙(Hover/Active/Subtle/Border 계산, 라이트/다크 배경별 텍스트 대비 기준)은 모든 테마에 공통 적용하고, 테마마다 색상 값을 교체한다.
**정정(2026-07-10)**: ~~테마마다 바뀌는 건 Primary 색상 값 하나뿐~~ — 사용자가 실제 예시 테마("레테": primary `#452951`(진보라)·secondary `#A7446A`(로즈)·third `#FBFB8A`(연노랑))를 주면서 확인됨: 직업 테마는 **3색 세트**(Primary/Secondary/Third)로 온다. 3색 각각의 역할은 색상마다 다르게 매핑한다(와이어프레임 적용 사례 참고): Primary는 채움 배경 + 텍스트/아이콘 잉크 겸용(밝기가 충분히 낮으면 별도 잉크 보정 불필요), Secondary는 hover/눌림 상태 + 보더, Third는 옅은 강조 배경(subtle) + 채움 위 텍스트색. 이 매핑은 테마별 밝기 구성에 따라 달라질 수 있어 고정 공식이 아니라 색상마다 판단이 필요하다.
**이유**: 메이플스토리 직업 고유 컬러를 활용하면 사용자가 자신의 캐릭터/직업에 애착을 느끼는 개인화 경험을 제공할 수 있다.
**구현 방향(뼈대, 2026-07-09)**:
- 컬러 값은 `src/data/job-themes.json`에 직업별 참조 테이블로 저장한다. 다른 게임 레퍼런스 데이터와 동일하게 [[ADR-006]] CRITICAL 규칙이 적용되어, 실제 컬러 값은 AI가 임의로 채우지 않고 사용자(도메인 전문가) 확정 후 반영한다 — 현재 미작성
- 선택된 테마는 다른 전역 클라이언트 상태와 동일하게 Zustand로 관리하고 `storage/`에 영속화한다
- Tailwind는 정적 유틸리티 클래스 기반이라 런타임에 값이 바뀌는 다중 테마와 궁합이 안 맞는다 — Primary 관련 값은 CSS 커스텀 프로퍼티(예: `--color-primary`)로 노출하고, 선택된 테마 값을 `:root`에 주입하는 방식을 제안(구현 시 확정)
**트레이드오프**: Tailwind arbitrary value(`bg-[#...]`)를 그대로 못 쓰고 `bg-[var(--color-primary)]` 형태의 간접 참조가 필요해진다. `UI_GUIDE.md`에 정의한 정적 색상값(기본 테마)도 테마 적용 시점에는 CSS 변수 참조로 바뀌어야 한다.
**미확정 — 추후 확정 필요**:
- 직업별 실제 컬러 값(hex) — "레테" 예시는 검증용 테스트 팔레트로 와이어프레임에만 적용했고, `src/data/job-themes.json`에는 아직 반영하지 않았다(실제 직업 매핑인지 사용자 확인 전까지 [[ADR-006]] CRITICAL 규칙 적용)
- `job-themes.json` 스키마를 `{ primary, secondary, third }` 3필드로 확정할지, 색상 역할 매핑(어떤 색이 채움/보더/텍스트인지)을 테마마다 수동 지정할지 자동 규칙(예: 가장 어두운 색 = 채움)으로 계산할지
- 테마 단위: 직업 대분류(전사/마법사/궁수/도적/해적/초보자 등)인지 5차 전직 세부 단위인지
- 테마 전환 UX: 설정에서 수동 선택인지, 등록된 캐릭터의 직업에 따라 자동 적용인지
- **정정(2026-07-09)**: ~~테마 미선택 시 기본값 유지 여부~~ — 확정: 기본값은 계속 오렌지(#FF7033, `UI_GUIDE.md` "기본 테마")를 쓰되, 배경 팔레트는 다크에서 **라이트로 전환**했다(게임 앱다운 캐주얼한 톤을 위해, PRD "디자인" 섹션 참고). 다크는 폐기하지 않고 보조 팔레트로 유지
- 다크 모드를 사용자가 설정에서 직접 켤 수 있게 할지 여부는 여전히 미정

### ADR-010: 물욕템 수익 반영 및 랜덤 컨테이너 아이템 결과 기록
**결정**:
1. 물욕 아이템 드랍 기록에 아이템별 고정 시세를 반영해, "주간 보스 수익 계산기" 화면에서 결정 판매 수익과 물욕템 환산 가치를 항목별로 구분 표시하고 합계도 함께 보여준다. 새 화면을 만들지 않고 기존 화면을 "이번 주 / 월간 추이" 탭으로 확장한다(2026-07-09 확정).
2. "OO옥의 보스 반지 상자"·"혼돈의 칠흑 장신구 상자" 같은 컨테이너형(랜덤) 아이템은 개봉 결과(반지 레벨·반지 종류, 또는 장신구 종류)가 확률적으로 정해지므로, 획득 기록 시 사용자가 실제로 나온 결과를 후보 목록에서 직접 선택해 기록한다. 확률로 자동 추정하지 않는다.
**이유**: 사용자가 실제로 획득한 결과를 정확히 남기는 게 이 앱의 핵심 가치다([[ADR-008]] 참조 무결성 원칙과 동일한 맥락) — 확률 기반 추정은 실제 기록과 다를 수 있어 신뢰할 수 없다. 화면을 새로 만들지 않는 이유는 사용자가 두 수익원(보스 결정 판매·물욕템)을 "이번 주 수익"이라는 하나의 맥락에서 보고 싶어하기 때문(2026-07-09 확정).
**데이터**:
- `src/data/boss-ring-boxes.json` 신규(2026-07-09) — 녹옥/홍옥/흑옥/백옥/생명의 보스 반지 상자 5종의 레벨별 확률표·반지별 확률표(사용자 제공, 게임산업법 시행령 기준 반올림 표기). 각 박스가 어느 보스+난이도에서 드랍되는지는 `item-drop-table.json`의 `consumable` 카테고리(주석 "교환 불가")에 이미 반영되어 있어 별도 매핑을 만들지 않는다.
- `src/data/accessory-boxes.json` 신규(2026-07-09) — "혼돈의 칠흑 장신구 상자"·"메이린의 칠흑 장신구 상자"(이름만 다르고 후보 목록 동일, 사용자 확인)의 후보 아이템 7종(사용자 제공 아이템 설명 이미지). 반지 상자와 달리 게임 내 툴팁에 개별 획득 확률(%)이 공개돼 있지 않아 후보 목록만 확정, 확률 필드는 `null`
- 확률표는 "실제 획득 결과 기록 시 선택할 후보 목록" 용도로만 쓴다. 확률 수치 자체를 UI에 노출할지는 별도 결정 사항(미정).
- 가격 필드(`priceMeso`) 네이밍은 `boss-crystal-prices.json`과 동일한 컨벤션을 따르며, 값이 확정되기 전까지는 `null`로 둔다([[ADR-006]] CRITICAL 규칙).
**미확정 — 추후 확정 필요**:
- 물욕템(반지·장신구 포함) 개별 시세 — 아직 없음. 가격 없는 항목은 수익 계산에서 0원이 아니라 "가격 미확정"으로 별도 표시하고 합계에서 제외한다
- `item-drop-table.json`의 다른 항목들에 `priceMeso` 필드를 실제로 언제/어떻게 채울지 — 스키마 컨벤션만 정했고 기존 파일(2,100줄+)에 일괄로 `null` 필드를 미리 채워 넣지는 않았음(값 없는 필드를 대량으로 미리 넣는 건 정보가 없어 보류, 실제 시세 반영 시 함께 추가)
- 개봉해서 나온 개별 반지의 거래 가능 여부 — 박스 자체는 "교환 불가"로 명시돼 있으나 반지 자체의 거래 가능 여부는 미확인

### ADR-011: 물욕 아이템 드랍 화면의 보스 목록 출처, 난이도 무관 아이템 노출, 이미지 에셋
**결정**:
1. "물욕 아이템 드랍 현황"의 **보스 목록**은 독립적인 목록이 아니라, **주간 스케줄러(feature 2)와 동일한 Nexon 동기화 캐시**(`boss_contents` 중 `cycle: bossWeekly`, `registration_flag: true`)에서 가져온다. 즉 게임 내 스케줄러에 실제로 등록된 주간 보스만 뜬다(확정, 2026-07-09).
2. 보스를 선택해 들어간 상세 화면에서는 그 보스의 물욕템을 **난이도 구분 없이 전부 합쳐서** 보여준다 — `item-drop-table.json`에서 `boss` 이름이 같은 모든 엔트리(노멀/하드/익스트림/카오스 등)의 아이템을 이름 기준으로 합치고 중복 제거한다. 보스 목록·아이템 그리드 어디에도 난이도 표기를 하지 않는다(확정, 2026-07-09).
3. 물욕템 버튼은 텍스트가 아니라 **이미지**로 표시한다. 이미지 파일은 사용자가 직접 제공하며, `src/assets/items/`에 **아이템명(공백→언더스코어)를 파일명으로** 저장한다(예: `리스트레인트 링` → `리스트레인트_링.png`). `item-drop-table.json`에 이미지 경로 필드를 추가하지 않고, 아이템명으로부터 파일 경로를 계산하는 헬퍼 함수(`lib/item-icons`)로 조회한다. 이미지가 없는 항목은 플레이스홀더 아이콘으로 폴백한다(확정, 2026-07-09).
4. **이미지 에셋 2분류 확정(2026-07-10)**: 아이템 이미지 외에 **보스 초상화**도 별도 에셋으로 관리한다. "보스를 나타내는 컴포넌트"(주간 스케줄러의 보스 항목, 보스 수익 계산기의 보스 행, 물욕 아이템의 보스 목록·상세 헤더 등 — feature 2·4·5에 걸쳐 재사용)에서 텍스트 대신 이 초상화를 표시한다. 아이템과 동일한 컨벤션을 그대로 따른다: `src/assets/bosses/`에 **보스명(공백→언더스코어)을 파일명으로** 저장(예: `검은 마법사` → `검은_마법사.png`), 난이도별이 아니라 **보스 1종당 초상화 1장**(ADR-011 결정 2의 "난이도 무관" 원칙과 동일한 이유 — 초상화도 난이도로 쪼개지 않는다), 조회는 `lib/boss-icons` 헬퍼, 이미지 없으면 플레이스홀더 폴백. 반복되는 "이름 → 에셋 경로, 없으면 폴백" 로직은 `lib/item-icons`와 `lib/boss-icons`에 각각 독립적으로 둔다(로직이 한 줄짜리라 공용 추상화를 만들 정도는 아님).
5. **반지류는 한글명이 아니라 영문(GMS) 파일명 사용(확정, 2026-07-11)**: 특수 스킬 반지(`src/assets/items/rings/`)는 한글 아이템명을 파일명 변환 규칙만으로 못 옮긴다(예: "리스트레인트 링" → `Ring_of_Restraint.png`, 기계적 변환이 아니라 실제 GMS 명칭). 그래서 이 계열만 예외적으로 `boss-ring-boxes.json`의 각 아이템 항목에 `iconFile` 필드를 직접 추가해 한글명↔영문 파일명을 명시 매핑한다(나무위키 "특수 스킬 반지" 문서의 GMS-MSEA 표기 기준으로 사용자가 확인). "컨티뉴어스 링"만 나무위키에 GMS 표기가 없어 다른 반지들의 명명 규칙(형용사/명사 + Ring)에 맞춰 `Continuous_Ring.png`로 직접 지음 — 공식 확인된 이름이 아니라는 점을 데이터에도 명시해둠. 웨폰퍼프·레벨퍼프는 스탯(S/I/L/D)별로 별도 아이콘이 없어 스탯 무관 공용 파일(`Weapon_Jump_Ring.png`/`Level_Jump_Ring.png`)을 공유한다.
5. **`items/` 하위 `rings/` 서브폴더 신설(확정, 2026-07-10)**: 특수 스킬 반지(`boss-ring-boxes.json`에 나오는 리스트레인트 링 등, 30여 종)는 종류가 많아 `src/assets/items/rings/`로 따로 묶는다. `lib/item-icons`는 아이템명이 **"링"으로 끝나면** `items/rings/{이름}.png`, 아니면 `items/{이름}.png`를 조회한다(접미사 휴리스틱 — `boss-ring-boxes.json`의 반지 30종은 전부 "링"으로 끝나고, 같은 박스에서 나오는 비반지 아이템 "생명의 연마석"만 유일한 예외인데 이건 "링"으로 안 끝나 정확히 걸러짐. 실측 데이터로 검증된 규칙이지 추정 아님). 디렉토리는 미리 만들어둠(`.gitkeep`), 실제 이미지는 사용자가 추가.
6. **정정(2026-07-11) — 일반 아이템·보스 초상화도 파일명 그대로 쓰고 매핑 테이블로 조인한다**: 결정 3·4의 "한글명(공백→언더스코어)을 파일명으로" 컨벤션은 폐기한다. 사용자가 실제로 넣어둔 파일은 영문 내부 코드명 스타일이었고("변환"이 아니라 "이미 있는 파일명을 그대로 쓰고 우리 데이터와 매칭만 하라"는 사용자 지시, 2026-07-11), 반지 때와 같은 이유로 이름→파일명이 기계적 변환이 아니라 **명시적 매핑 테이블**이 필요하다.
   - **보스 초상화**: `weekly-bosses.json`의 각 보스 항목에 `portraitSlug` 필드 추가(예: `루시드` → `lucid`). 실제 파일명은 `{난이도접두사}_{portraitSlug}.png`이며 난이도 접두사는 이지→`easy`, 노멀→`normal`, 하드→`hard`, 카오스→`chaos`, 익스트림→`extreme`로 고정. 나무위키 "GMS 명칭" 확인 + 보스별 실제 파일 개수가 `difficulties` 배열 길이와 전부 일치하는 구조적 교차검증으로 확인함. `portraitSlug`가 없는 보스는 아직 이미지가 없다는 뜻(자쿰·매그너스·파풀라투스·반반·피에르·블러디 퀸·벨룸·벨로나).
   - **일반 아이템**: 신규 `src/data/item-icons.json`에 "한글 아이템명 → 기존 파일명" 매핑을 별도 테이블로 관리한다(`item-drop-table.json`은 건드리지 않음 — 같은 아이템이 여러 보스에 걸쳐 반복 등장해서 그 파일에 필드로 넣으면 중복이 심함). 확신도 높은 27개 항목만 우선 반영했고, 후보가 여러 개라 확정 못 한 것들은 사용자에게 별도로 물어봄(아래 미확정 참고).
   - `lib/item-icons`·`lib/boss-icons`는 이제 "이름으로부터 파일 경로를 계산"이 아니라 "매핑 테이블에서 조회, 없으면 플레이스홀더"로 동작이 바뀐다.
**이유**:
- 사용자 실측: "유저들이 자신이 처치하는 보스 난이도를 정확하게 등록해놓지 않았을 수도 있다" — 난이도 기준으로 아이템을 필터링하면 실제로 얻은 아이템이 목록에 없어서 기록을 못 하는 상황이 생길 수 있다. 난이도 대신 보스 단위로 전부 보여주는 게 더 안전하다.
- 이미지를 JSON 필드로 관리하지 않고 파일명 컨벤션으로 조회하는 이유: `item-drop-table.json`은 이미 사용자가 확정 반영을 완료한 2,100줄+ 파일이라([[ADR-006]]), 값이 아직 없는 필드를 대량으로 미리 넣는 건 [[ADR-010]]의 `priceMeso` 때와 동일한 이유로 보류한다 — 파일명 컨벤션 방식은 스키마 변경 없이 이미지 유무와 무관하게 항상 유효한 조회 경로를 제공한다.
**트레이드오프**: 아이템명·보스명이 바뀌면(오탈자 수정 등) 이미지 파일명도 같이 바꿔야 한다(고정 ID가 아니라 이름 기반 조회라서 생기는 결합). 아이템 수가 많아지면(300+) 파일명 기반 조회가 매핑 테이블보다 관리가 번거로워질 수 있음 — 그 시점엔 재검토. 보스명은 `weekly-bosses.json`이 이미 API 원문↔우리 표기 정규화(공백 제거 비교, `apiAlias`)를 하고 있으므로, 초상화 조회는 정규화된 **우리 쪽 보스명**(원문 API 문자열이 아니라)을 키로 써야 한다. "링" 접미사 휴리스틱은 현재 데이터로는 정확하지만, 향후 반지가 아닌데 이름이 "링"으로 끝나는 아이템이 추가되면 오분류될 수 있다 — 그 시점엔 명시적 분류로 전환.
**정정 — ARCHITECTURE.md 데이터 흐름 오류 수정**: 기존 문서에 "[보스 수익 계산기 / 물욕 아이템 드랍 — 기존과 동일, Nexon API와 무관]"이라고 적혀 있었는데, 이는 부정확했다. 보스 수익 계산기는 애초에 PRD 핵심 기능 4에 "Nexon API 동기화 데이터를 그대로 사용"이라고 명시돼 있었고([[ADR-007]]), 이번 ADR-011로 물욕 아이템 드랍의 보스 목록도 같은 캐시를 쓰게 되어 두 기능 모두 Nexon API 동기화 데이터에 의존한다. "API와 무관"한 부분은 파티원 수 입력·아이템 획득 기록·수익 계산 자체(로컬 전용)로 한정된다.
**미확정 — 추후 확정 필요**:
- 월간 보스(검은마법사, `cycle: bossMonthly`)가 이 화면의 보스 목록에서 제외되는데, 의도된 것인지 확인 필요 — 사용자 지시는 "주간 보스"로 한정했으나 월간 보스도 물욕템 추적 대상일 수 있음
- 이미지 파일 포맷(png/webp 등)·권장 해상도는 미정(아이템 이미지·보스 초상화 공통)

### ADR-012: 일간/주간 스케줄러 캐릭터 추적(선택) 정책 및 동기화 범위 제한
**결정**:
1. 일간 스케줄러와 주간 스케줄러는 각각 **독립적으로** "추적할 캐릭터"를 사용자가 직접 고른다 — 계정에 등록된 캐릭터 전체가 아니라 사용자가 선택한 부분집합만 화면에 표시한다. 두 화면의 추적 목록은 서로 공유하지 않는다(일간에서 추적하는 캐릭터와 주간에서 추적하는 캐릭터가 다를 수 있다, 확정 2026-07-11).
2. 아직 아무 캐릭터도 고르지 않았으면(최초 실행 등) 빈 상태("캐릭터를 선택해주세요")로 표시한다. 화면 안의 "캐릭터 관리" 버튼으로 언제든 후보 목록(계정의 전체 캐릭터)에서 체크박스로 고를 수 있고, 체크 상태는 "저장"을 눌러야 확정되며 취소 시 반영되지 않는다.
3. 화면에 표시되는 콘텐츠(일간 콘텐츠, 주간 퀘스트, 주간 보스)는 그중에서도 게임 클라이언트 스케줄러에 실제로 **등록된**(`registration_flag: "true"`) 항목만 보여준다 — 미등록 항목은 화면에서 완전히 제외한다(PRD 핵심 기능 1·2에 이미 명시된 "표시되는 항목은 등록된 것만 해당" 원칙을 화면 구현에도 명확히 반영, 확정 2026-07-11).
4. **성능 — 스케줄 동기화 자체를 추적 대상으로 제한(확정, 2026-07-11)**: Nexon 스케줄러 Open API 호출(`GET /maplestory/v1/scheduler/character-state`)은 계정의 전체 캐릭터가 아니라 **추적 목록에 있는 캐릭터에 대해서만** 순차 호출한다. "캐릭터 관리" 피커가 보여줄 후보 목록(캐릭터명만 필요)은 `GET /maplestory/v1/character/list`(API 호출 1번, 가벼움)로 별도 조회하고 스케줄 동기화와는 분리한다.

**이유**:
- 실사용 계정에 캐릭터가 30명 이상 있는 경우가 실측으로 확인됨 — 대부분 방치된 부캐릭터라 사용자가 실제로 관심 있는 캐릭터는 소수다. 계정 전체를 대상으로 순차 스케줄 동기화를 하면([[ADR-008]] 순차 호출 정책과 맞물려) 캐릭터 수에 비례해 로딩 시간이 크게 늘어난다.
- 추적 대상만 동기화하면 불필요한 API 호출을 줄여 로딩 속도를 개선하고, 화면도 사용자가 실제로 보고 싶은 캐릭터만 간결하게 보여줄 수 있다.

**저장소**: `storage/`에 일간/주간 각각 독립된 키(`trackedCharacters:daily`/`trackedCharacters:weekly`)로 추적 캐릭터 `ocid` 배열을 저장한다. `null`(한 번도 설정 안 함)과 `[]`(사용자가 명시적으로 전부 해제함)은 값의 의미가 다르므로 구분해서 다룬다.

**트레이드오프**: 캐릭터 관리 피커의 후보 목록은 스케줄 동기화 없이 이름만 보여주므로, 그 화면만 봐서는 어떤 캐릭터가 실제로 진행 중인 콘텐츠가 있는지 미리 알 수 없다(추적 대상으로 고른 뒤에야 알 수 있음). 새로 캐릭터를 추적 대상에 추가하면 그 즉시 스케줄 재동기화가 트리거된다.

### ADR-013: 화면 구조를 "일간/주간 스케줄러"에서 "컨텐츠/보스 스케줄러"로 개편
**결정**: 리셋 주기(일간/주간) 기준으로 나뉘어 있던 최상위 화면 2개를 콘텐츠 종류(일반 콘텐츠/보스) 기준으로 재편한다.
1. **컨텐츠 스케줄러**: 일간 탭(기존 `daily_contents`) + 주간 탭(기존 `weekly_contents`)만 표시한다. **월간 탭은 두지 않는다** — Nexon 스케줄러 API에 월간 주기 일반 콘텐츠 자체가 없다.
2. **보스 스케줄러**: 주간 탭(`cycle: bossWeekly`) + 월간 탭(`cycle: bossMonthly`, 현재 검은마법사 1종)을 표시한다. **일간 탭은 두지 않는다** — [[ADR-007]]에서 확정한 대로 `bossDaily`(힐라 하드·핑크빈 카오스 등 일간으로 격하된 보스)는 계속 완전히 무시한다. 이번 개편은 이 정책을 재검토하지 않는다(확정, 2026-07-11).
3. 화면 내부의 일간/주간/월간 전환은 별도 라우트가 아니라 화면 내 탭(로컬 컴포넌트 상태)으로 구현한다 — 두 화면 모두 딥링크가 필요한 시나리오가 없다.

**보스 cycle 분리 표시 (버그 성격의 정정 포함)**: 기존 주간 스케줄러는 `bossContents`(cycle: weekly/monthly 전체)를 cycle 구분 없이 "주간 보스" 섹션 하나에 표시했다 — `weekly-bosses.json`에 이미 `monthly`(검은마법사) 참조 데이터가 있었음에도 화면에서 주간/월간을 나누지 않고 있었다. 개편 후 `boss-scheduler`는 cycle로 명확히 분리해 주간 탭/월간 탭 각각에 표시한다. `weeklyBossClearCount`/`weeklyBossClearLimitCount`(12마리 제한, [[ADR-007]])는 주간 탭에서만 의미 있는 값이다 — 월간 보스는 이 카운트에 포함되지 않는다.

**캐릭터 추적 목록 재구성 ([[ADR-012]] 원칙 유지, 키만 변경)**: [[ADR-012]]가 도입한 "화면별 독립 추적 목록" 원칙은 그대로 유지하되, 화면이 컨텐츠/보스 2개로 바뀌므로 `storage/`의 키도 `trackedCharacters:content`/`trackedCharacters:boss`로 바뀐다.
- **1회 마이그레이션(확정, 2026-07-11)**: `trackedCharacters:content` 키가 Preferences에 아직 한 번도 쓰인 적 없는 상태(키 자체가 없음, 즉 `Preferences.get`이 `null` 반환)로 조회되면 다음 규칙으로 마이그레이션한다.
  - `legacyDaily = get('trackedCharacters:daily')`, `legacyWeekly = get('trackedCharacters:weekly')` (둘 다 `string[] | null`)
  - `legacyDaily`와 `legacyWeekly`가 **둘 다 `null`**이면 마이그레이션할 데이터가 없는 것이므로 아무 키도 쓰지 않는다(`content`/`boss` 모두 계속 미설정 상태로 남음 — 다음 조회 때 다시 이 조건을 평가하지만 매번 결과가 같으므로 문제 없음)
  - 그 외의 경우 `content = dedupe([...(legacyDaily ?? []), ...(legacyWeekly ?? [])])`(ocid 기준 중복 제거), `boss = legacyWeekly`(그대로 복사, `null`이면 `null`)로 계산해 `trackedCharacters:content`/`trackedCharacters:boss`에 쓴다
  - 기존 주간 화면이 주간 퀘스트와 주간 보스를 같은 추적 목록으로 함께 보여주고 있었으므로, 분리 이후에도 사용자가 다시 선택할 필요 없이 같은 캐릭터가 양쪽에 그대로 보이게 하기 위함(사용자 확정, 2026-07-11). `content`가 daily·weekly 합집합인 이유: 컨텐츠 스케줄러 화면 하나가 일간/주간 탭을 함께 보여주므로, 둘 중 하나에만 추적 등록돼 있던 캐릭터도 새 화면에서 계속 보여야 정보 손실이 없다.
- 마이그레이션은 딱 1회만 수행한다 — `trackedCharacters:content`가 이미 존재하면(마이그레이션을 이미 했거나 사용자가 새로 설정했으면) 다시 덮어쓰지 않는다. 마이그레이션 이후 두 목록은 완전히 독립적으로 갈라진다(한쪽만 변경해도 다른 쪽에 영향 없음, [[ADR-012]] 원칙 그대로).
- 이전 키(`trackedCharacters:daily`/`trackedCharacters:weekly`)는 마이그레이션 후 정리(삭제)한다.

**이유**: 사용자가 이 앱을 켤 때 실제로 궁금해하는 축은 "오늘/이번 주에 뭘 해야 하나"가 아니라 "일반 콘텐츠 진행 상황"과 "보스 처치 상황"이 서로 다른 관심사라는 판단에 따른 제품 방향 확정(2026-07-11).

**트레이드오프**: 화면이 2개(daily/weekly)로 딱 나뉘어 있던 이전 구조 대비, 각 화면 내부에 탭 전환 UI가 추가되어 화면 하나의 복잡도가 늘어난다. 컨텐츠 스케줄러는 기존에 분리돼 있던 두 도메인(일간 콘텐츠, 주간 콘텐츠)을 한 화면에서 조합해야 한다.

### ADR-014: 보스 수익 계산기 — 완료 감지 시 파티원 수 자동 기록 및 캐릭터별 레이아웃
**결정**:
1. 스케줄 동기화로 새로 처치가 확인된 보스는 사용자가 화면에 들어와 파티원 수를 입력하기 전에도 **즉시 수익 기록을 자동 생성**한다. 기본 파티원 수는 같은 캐릭터(ocid)+보스+난이도 조합의 **가장 최근 기록값**을 그대로 이어 쓰고, 그 조합에 대한 과거 기록이 전혀 없으면 **1**을 기본값으로 쓴다.
2. 사용자가 파티원 수를 수정하면 해당 주(또는 달)의 기록만 갱신된다. 그 값은 이후 같은 캐릭터+보스+난이도 조합에서 새로 완료가 감지될 때 결정 1의 "가장 최근 기록값"이 되어 자동으로 이어진다 — 별도의 "기본값" 저장소나 테이블을 새로 두지 않고, 기존 `boss_profit_records` 테이블에서 `period_key` 조건 없이 `(ocid, boss, difficulty)`로 가장 최근 `recorded_at` 레코드 1건을 조회하는 쿼리로 구현한다(스키마 변경 없음, [[ADR-003]] 진행 상황 참고).
3. 화면 레이아웃을 상단 "이번 주" 총 수익(주간 보스, cycle: weekly만 합산) 1개 숫자 + 그 아래 캐릭터별 드롭다운(아코디언) 목록으로 재구성한다. 각 드롭다운 헤더에 그 캐릭터의 "이번 주" 총 수익을 표기하고, 펼치면 보스별 내역(파티원 수 입력 포함)을 보여준다. 월간 보스(검은마법사) 수익은 상단 합계에서 제외하고, 각 캐릭터 드롭다운 안에 "이번 달" 구분으로 별도 표시한다(기존 주간/월간 섹션 구분은 유지하되, 표시 단위를 화면 전체에서 캐릭터 단위로 옮긴다).

**이유**:
- 사용자가 매번 화면에 들어와 파티원 수를 입력해야만 수익 합계에 반영되는 기존 흐름은, 실제로는 "고정 파티로 거의 같은 인원과 도는" 사용 패턴과 맞지 않는다 — 매주 같은 숫자를 반복 입력해야 하는 불필요한 마찰이다(사용자 확인, 2026-07-11).
- 파티원 수를 캐릭터+보스+난이도 단위로 기억하는 이유: 보스마다 파티 구성이 다를 수 있다(예: 특정 보스는 6인 고정 파티, 다른 보스는 혼자 처치). 캐릭터 단위로 단일 값을 쓰면 보스마다 실제 인원이 다른 경우를 표현할 수 없다.
- 새 테이블 없이 기존 레코드에서 최근값을 조회하는 이유: 이미 캐릭터+보스+난이도+주차 단위로 기록이 쌓이고 있으므로([[ADR-003]]), 여기서 `period_key` 조건만 빼고 가장 최근 것을 고르면 "마지막으로 쓴 값"을 그대로 얻을 수 있다. 별도 "기본값 테이블"을 두면 사용자가 특정 주만 다르게 입력했을 때 "이게 새 기본값인지, 그 주만의 예외인지"를 구분하는 추가 로직이 필요해진다 — 항상 최신 기록을 기본값으로 쓰는 단순한 규칙이 이 문제를 자연히 피한다.
- 화면을 캐릭터별 드롭다운으로 재구성하는 이유: 추적 캐릭터가 여러 명이면([[ADR-012]]에서 실사용 계정에 캐릭터가 30명 이상인 경우도 확인됨) 기존의 "주간/월간 평면 목록"은 어떤 보스가 어느 캐릭터 것인지 한눈에 파악하기 어렵다. 상단에 전체 합계를 먼저 보여주고 캐릭터 단위로 접어두면, 여러 캐릭터를 동시에 추적할 때도 화면이 덜 길어진다.

**트레이드오프**:
- 사용자가 확인하기 전에 값이 자동으로 기록되므로, 실제 파티원 수와 기본값(특히 최초 감지 시의 1)이 다르면 사용자가 직접 확인하지 않는 한 부정확한 수익이 그대로 표시될 수 있다. "완료는 됐는데 수익이 0으로 잡히는" 기존 방식보다, 틀린 숫자를 사용자가 못 보고 지나칠 위험이 늘어난다는 뜻이다 — 자동 기본값이 적용된 행을 시각적으로 구분할지는 아래 "미확정" 참고.
- "가장 최근 기록값"을 기본값으로 쓰기 때문에, 파티원 수를 일시적으로만 바꿨다가 되돌리는 걸 깜빡하면 그 값이 계속 이어진다 — 사용자가 실제 파티 구성이 바뀔 때마다 정확히 반영해야 정확도가 유지된다.
- 캐릭터별 드롭다운 구조는 추적 캐릭터가 많을수록 전부 펼쳤을 때 화면이 길어질 수 있다 — 기본 접힘 상태로 시작하는 것으로 완화한다.

**미확정 — 추후 확정 필요**:
- 자동 기본값(특히 "처음 감지라 1로 기록됨")이 적용된 행을 사용자에게 시각적으로 강조할지(예: "확인 필요" 배지 등) — 위 트레이드오프 참고, UI 구현 시 결정
- 드롭다운 초기 펼침/접힘 상태(전부 접힘 vs 캐릭터 수가 적으면 자동 펼침 등) — UI 구현 시 결정
