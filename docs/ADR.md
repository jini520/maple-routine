## 철학
로컬 우선(local-first) MVP. 백엔드/로그인 없이 기기 안에서 완결되는 최소 구현을 우선 선택하고, 알림·백그라운드 사운드처럼 웹만으로 불가능한 기능만 네이티브 브릿지에 의존한다. Nexon Open API 연동([[ADR-007]])도 사용자 개인이 발급받은 API 키를 기기에 저장해 직접 호출하는 방식이라 이 원칙을 벗어나지 않는다. 게임 밸런스에 따라 자주 바뀌는 수치 데이터는 로직과 분리해 유지보수 비용을 낮춘다.

### ADR-001: Capacitor 기반 하이브리드 앱 채택
**결정**: React + Capacitor로 웹 코드를 Android/iOS WebView 앱으로 패키징한다.
**이유**: 웹 코드베이스를 거의 그대로 재사용하면서 `@capacitor/local-notifications`, Background Runner 등 공식/커뮤니티 플러그인으로 일간·주간 알림과 사냥 타이머 백그라운드 사운드를 구현할 수 있다. Cordova 대비 유지보수가 활발하고, 커스텀 네이티브 WebView 대비 두 플랫폼 코드를 따로 관리할 필요가 없다.
**트레이드오프**: 완전 커스텀 네이티브 UX/성능은 포기한다. 앱스토어 심사, 네이티브 빌드 파이프라인(Xcode/Gradle) 관리 비용이 순수 웹 배포보다 추가된다.

### ADR-002: Vite 기반 React SPA (Next.js 미사용)
**결정**: Next.js App Router 대신 Vite + React 기반 SPA로 개발한다.
**이유**: WebView 안에서 실행되는 앱이라 SSR/SEO가 불필요하다. Capacitor는 정적 빌드 산출물을 그대로 패키징하는 방식과 궁합이 좋고, 빌드/배포 파이프라인이 단순해진다.
**트레이드오프**: Next.js의 파일 기반 라우팅, API 라우트, 이미지 최적화 등은 사용하지 않는다. 라우팅은 React Router 같은 별도 라이브러리로 직접 구성해야 한다.

### ADR-003: 백엔드 없이 기기 로컬 저장소만 사용 (MVP)
**결정**: 별도 서버/DB 없이 기기 로컬 저장소(Capacitor SQLite 또는 Preferences)에 모든 데이터를 저장한다. 사용자가 openapi.nexon.com에서 개인적으로 발급받은 Nexon API 키도 이 로컬 저장소에 저장하고, 앱이 이 키로 Nexon Open API를 직접 호출한다([[ADR-007]]).
**이유**: 로그인·서버 인프라 없이도 개인 기록 관리라는 핵심 가치를 제공할 수 있어 MVP 범위를 최소화할 수 있다.
**트레이드오프**: 기기 변경/앱 삭제 시 데이터가 유실될 수 있고 멀티 디바이스 동기화가 불가능하다. 추후 백업(export/import) 또는 클라우드 동기화로 확장 가능하도록 저장소 접근을 `storage/` 레이어로 추상화해둔다.
**확정 — SQLite vs Preferences 하이브리드 채택 (2026-07-11)**: 단순 key-value 데이터(Nexon API 키, 선택된 계정, 일간/주간 화면의 추적 캐릭터 목록, 스케줄 동기화 캐시)는 `@capacitor/preferences`로 충분해 그대로 쓴다. 반면 앞으로 만들 보스 수익 기록(캐릭터+보스+난이도+주차 unique key로 upsert, [[ADR-008]] 멱등성 요구사항)과 물욕템 드랍 히스토리(날짜 범위로 "이번 주/월간 추이" 조회, PRD 핵심 기능 4)는 실제 쿼리(WHERE/날짜 범위/upsert)가 필요해 Preferences로는 비효율적이다 — 이 두 기능을 구현하는 시점에 `@capacitor-community/sqlite`를 도입하는 **하이브리드**로 확정한다. 지금 당장 기존 Preferences 기반 저장소를 SQLite로 전면 마이그레이션하지 않는다. SQLite 도입 시 웹 환경 테스트를 위한 `jeep-sqlite` 폴리필도 함께 검토해야 한다.
**검토했다가 기각한 대안**: Nexon Open ID(OAuth 로그인) 연동 시 authorization code → access token 교환에 서버가 필요해 이 ADR을 일시 수정하는 방향으로 검토했었다. 하지만 Nexon Open ID는 "메이플스토리 파트너스" 승인이 필요한 제한된 기능이고(사용자 미신청 상태), 실측 결과 스케줄러 Open API가 개인 API 키 + ocid만으로 호출 가능함이 확인되어([[ADR-007]]) 이 대안은 기각하고 원래 결정을 유지한다 (2026-07-09).
**진행 상황 — SQLite 도입 시작 (2026-07-11)**: 위 하이브리드 결정의 "이 두 기능을 구현하는 시점"이 도래해, 보스 수익 계산기(핵심 기능 4) 1차 구현의 첫 step으로 `@capacitor-community/sqlite` + 웹 테스트용 `jeep-sqlite` 폴리필을 도입한다. 이 시점에는 보스 수익 기록(`boss_profit_records`, 캐릭터+보스+난이도+주차 unique key upsert) 테이블만 만든다 — 물욕템 드랍 히스토리 테이블은 물욕 아이템 드랍(핵심 기능 5)이 별도로 구현될 때 추가한다.

### ADR-004: 서버 푸시 없이 로컬 알림만 사용
**결정**: 일간/주간 스케줄 알림은 Capacitor Local Notifications로 기기에서 직접 예약한다. FCM/APNs 등 서버발 푸시는 사용하지 않는다.
**이유**: 백엔드가 없는 로컬 우선 아키텍처와 일치하며, 리셋 시각·사용자 지정 알림 시각을 기기에서 미리 예약하는 것만으로 요구사항을 충족한다.
**트레이드오프**: 기기 재부팅, 강제 종료, OS 배터리 최적화 정책에 따라 알림이 지연되거나 누락될 수 있다. 특히 iOS는 예약 가능한 로컬 알림 개수에 제한(64개)이 있어 스케줄링 전략(예: 오래된 예약 정리, 앱 실행 시 재예약)을 별도로 설계해야 한다.
**미완료 판단 기준 — 확정 (2026-07-09)**: [[ADR-007]] 도입 이후 일간/주간 미완료 여부 판단은 알림 발송 예정 시각에 **백그라운드에서 Nexon API를 실시간 재호출**해 정확한 최신 상태를 확인한 뒤 결정한다(단순 로컬 캐시 기준이 아님, 정확도를 우선). 이는 순수 "예약 후 무조건 발송"이 아니라 "예약된 시각에 백그라운드 작업을 트리거 → API 재확인 → 조건부로 알림 표시" 패턴이 필요함을 뜻한다 — Android는 WorkManager로 네트워크 호출 후 조건부 알림 발생, iOS는 BGAppRefreshTask/BGProcessingTask를 활용하되 OS가 정확한 시각 실행을 보장하지 않는다는 플랫폼 제약이 있다. 재호출 시점에 API 실패(오프라인, 429 등)하면 마지막 캐시 기준으로 폴백한다([[ADR-008]]).
**트레이드오프(추가)**: 배터리·네트워크 사용량이 "예약 후 무조건 발송" 방식보다 늘어난다. iOS의 백그라운드 실행 시간 제약상 정확히 예약 시각에 재호출이 안 될 수 있어 완벽한 실시간성은 보장 못 한다(베스트 에포트).
**64개 한도 초과 시 우선순위 정책**: 캐릭터 수 × (일간 1개 + 주간 1개) 알림이 64개를 넘어설 수 있으므로, 초과 시 (1) 리셋 시각이 가장 임박한 항목 우선, (2) 동일 시각이면 마지막으로 앱을 연 캐릭터 우선 순으로 예약하고 나머지는 예약하지 않는다. 예약 못 한 항목이 있으면 설정 화면에 "일부 알림이 예약되지 않았습니다" 안내를 표시한다([[ADR-008]]).

### ADR-005: 사냥 타이머는 상시 표시 알림 + 주기적 사운드로 구현
**결정**: 타이머 시작 시 네이티브 타이머 앱처럼 "경과 시간이 표시되는 상시 알림"을 띄우고, 그 알림이 떠 있는 동안 지정 주기(1분/2분 또는 사용자 지정 N분)마다 알림음을 재생한다. 플랫폼별 구현은 다음과 같이 분리한다.
- **Android**: Foreground Service를 실행하고, `Notification.Builder.setUsesChronometer(true)`로 경과 시간을 OS가 자동 갱신하는 알림을 표시. 동일 서비스 내에서 주기 타이머를 돌려 알림음 재생.
- **iOS**: `Live Activity`(ActivityKit, iOS 16.1+)로 잠금화면/Dynamic Island에 경과 시간을 표시. 사운드 재생은 Live Activity와 별도로 로컬 알림 스케줄 또는 백그라운드 오디오 세션으로 트리거.
- 두 플랫폼 모두 Capacitor 공식 플러그인이 이 기능을 지원하지 않으므로, **Swift/Kotlin으로 커스텀 네이티브 플러그인을 직접 작성**해 `native/hunting-timer` 모듈로 노출한다.
**이유**: "네이티브 타이머 앱처럼 알림창에 항시 떠 있고 경과 시간도 표시"라는 요구사항은 단순 백그라운드 사운드 재생을 넘어, OS가 보장하는 상시 알림 UI(Android Chronometer 알림 / iOS Live Activity)가 있어야 신뢰성 있게 구현된다.
**트레이드오프**: 공식 플러그인만으로 해결되지 않아 커스텀 네이티브 개발·유지보수 비용이 발생한다. iOS Live Activity는 **iOS 16.1 이상**에서만 동작하므로 구버전 iOS 사용자는 동일 UX를 제공받지 못한다(대체로 로컬 알림 + 사운드만 제공하는 폴백 필요). 정식 구현 전 기술 스파이크(PoC)로 두 플랫폼 모두 실현 가능성과 배터리 영향을 검증해야 한다.

### ADR-006: 게임 레퍼런스 데이터는 코드와 분리된 데이터 소스로 관리
**결정**: 주간 보스 목록(`weekly-bosses.json`), 결정 가격표(`boss-crystal-prices.json`), 보스별 아이템 드랍 테이블(`item-drop-table.json`)은 `src/data/`의 JSON으로 분리하고, 실제 수치는 사용자(도메인 전문가)가 확정해 채운다.
**이유**: 게임 패치로 자주 바뀌는 정보를 로직과 분리해두면 다음 패치 때 코드 변경 없이 데이터만 업데이트할 수 있다. AI가 정확하지 않을 수 있는 실제 게임 수치를 임의로 하드코딩하는 위험도 피할 수 있다.
**트레이드오프**: 초기 출시 시 정확한 데이터가 채워지기 전까지는 계산기/드랍 기록 기능이 placeholder 데이터로만 동작한다.
**진행 상황**: 사용자가 데이터를 확정해 세 파일 모두 반영 완료. 가격은 사용자가 지정한 시세표 값을 채택(보스별 상세 정보 쪽 금액은 폐기), 갱신일은 주간/이벤트 주간 보스 `2026-06-25`, 월간 보스(검은마법사) `2026-07-01`로 일괄 표기. 힐라(하드)·핑크빈(카오스)는 일간 보스로 격하되어 세 파일에서 모두 제거. 벨로나(전 난이도)는 미출시 보스라 `status: "unreleased"`로 표시하고 가격/아이템은 `null`로 보류. 파티원 수별 가격은 `floor(priceMeso / 파티원 수)` 공식으로 확정. 메이린(이벤트 주간 보스)은 "강력한 힘의 결정"을 드랍하지 않고 대신 황금 메소 주머니(1개=1000만 메소)를 드랍하므로, 가격은 주머니 총 개수 × 1000만으로 `boss-crystal-prices.json`에 반영(`2026-07-09`)하고 `item-drop-table.json`의 황금 메소 주머니 항목은 제거함. 카이는 시즌이 종료된 레거시 보스라 `weekly-bosses.json`의 이벤트 주간 보스 목록에서 제외(`2026-07-09`).
**파티 입장 인원 제한 (2026-07-09 추가)**: 보스 입장 가능 인원은 기본 1~6인이나 예외가 있다 — 스우 익스트림 2인, 최초의 대적자·찬란한 흉성·림보·발드릭스·유피테르 3인, 메이린 1인. `boss-crystal-prices.json`의 `partySizeScaling.defaultMaxPartySize`(6)와 각 entry의 `maxPartySize`(예외가 있는 보스만)로 반영. 수익 계산기의 파티원 수 입력 검증에 사용.
**참고**: [[ADR-007]] 도입 이후 `weekly-bosses.json`의 역할은 "앱 내 보스 선택 UI용 목록"에서 "Nexon API 응답의 보스명/난이도 표기(영문)를 우리 데이터(한글)와 매핑하기 위한 참조 테이블"로 바뀐다. `boss-crystal-prices.json`·`item-drop-table.json`은 API와 무관하게 기존 용도(수익 계산, 물욕템 기록) 그대로 유지.

### ADR-007: 일간/주간 콘텐츠·보스 진행 상태는 Nexon Open API로 동기화
**결정**: 일간 스케줄러, 주간 스케줄러(퀘스트+보스)의 진행 상태는 사용자가 앱 안에서 수동으로 체크하지 않는다. 대신 Nexon Open API의 "스케줄러 Open API"(2026-06-25 출시)를 통해, 게임 클라이언트의 "스케줄러" 기능에 실제 등록된 항목과 그 완료 여부를 그대로 동기화해 읽기 전용으로 표시한다. 앱 자체의 보스 선택 UI(예: 기존에 검토했던 "최대 12마리 선택" 화면)는 두지 않으며, 게임 내 스케줄러 등록 상태가 유일한 기준이다.
**인증 방식**: 사용자가 [openapi.nexon.com](https://openapi.nexon.com)에서 개인적으로 애플리케이션을 등록해 발급받은 API 키를 앱 설정 화면에 직접 입력한다. NEXON Open ID(OAuth 로그인)는 사용하지 않는다 — 별도의 "메이플스토리 파트너스" 승인이 필요한 제한된 기능이고(2026-07-09 확인, `openapi.nexon.com/support/notice/2726294/`), 사용자가 이 프로그램에 신청하지 않은 상태다. 실측 결과 개인 API 키만으로 다음 흐름이 정상 동작함을 확인함(2026-07-09, 사용자 실측):
1. API 키 등록 직후 `GET /maplestory/v1/character/list`로 계정 소속 캐릭터 목록을 자동 조회(캐릭터명+월드 수동 입력 폼 없음, 2026-07-09 확인)
2. 각 캐릭터의 ocid → 스케줄러 Open API 호출로 `daily_contents`/`weekly_contents`/`boss_contents` 획득
**이유**: 사용자가 게임과 앱에 같은 내용을 이중으로 입력해야 하는 문제, 그리고 수동 입력이 실제 게임 상태와 어긋날 위험을 원천적으로 없앤다. OAuth 없이 개인 API 키만으로 동작이 확인되어 [[ADR-003]](백엔드 없음)을 그대로 유지할 수 있다.
**제약**:
- 응답에는 게임 클라이언트의 "스케줄러" 기능에 실제로 등록된 항목만 포함된다. 사용자가 게임에서 먼저 등록하지 않은 콘텐츠/보스는 이 앱에서도 보이지 않는다 — 온보딩 시 이 전제 조건을 안내해야 한다.
- 최근 14일치 기록만 조회 가능하다. 월간 수익 추이 등 장기 히스토리는 계속 로컬(`storage/`)에 누적 저장한다.
- 2026-06-25에 막 출시된 기능이라(확인 시점 기준 2주 차) 커뮤니티 문서가 거의 없고 스펙이 바뀔 수 있다.
- 개인 API 키는 사용자 본인의 Nexon 개발자 계정에 귀속되며(공용 비밀키 아님), 유출 시 영향 범위가 그 사용자 본인의 API 호출 한도에 한정된다 — 그래도 평문 노출은 피해야 하므로 `storage/`의 보안 영역(Keychain/Keystore 또는 암호화된 Preferences)에 저장한다.

**이용약관 준수 사항 (2026-07-09, `openapi.nexon.com/ko/support/terms/` 확인)**:
- **API 키 비공유**: 이용자는 발급받은 API 키를 타인에게 제공·공개·공유할 수 없다(약관 제5조①). 우리 앱은 각 사용자가 본인 키를 본인 기기에만 입력하는 구조라 이 조항과 자연히 부합한다 — 절대 앱에 공용 키를 내장하면 안 됨.
- **출처 표기 의무**: API 서비스 이용 결과물에 출처를 명시해야 한다(약관 제6조④). **정정(2026-07-13)**: ~~"NEXON Open API를 이용한 것임"을 앱 footer에~~ → 실제 요구 문구는 영문 **"Data based on NEXON Open API"**이며, 앱 전역 footer 대신 **설정 화면 하단**에 상시 노출하는 것으로 확정(전역 footer는 만들지 않음 — 아래 "확정 — 온보딩 및 이용약관 표기" 섹션 참고).
- **영리 목적 이용 제한**: 회사 승낙 없이 API 데이터를 영리 목적으로 이용할 수 없다(약관 제6조⑥). 현재 MVP는 무료 개인용 도구라 문제 없으나, 추후 유료화/광고 등을 고려한다면 별도 승낙 절차가 필요하다.
- **데이터 재배포 제약**: 결과 데이터를 약관이 허용한 범위를 넘어 무단 복제·저장·가공·배포할 수 없다(약관 제5조⑤). 우리 앱의 로컬 저장은 사용자 본인의 개인 기록 목적이라 통상적 이용 범위로 보이나, 법률 자문이 필요한 영역은 아니라 확정적으로 단언하지는 않음.

**전제 재검토 (2026-07-09) — 검증 안 된 위험**:
- ~~멀티 Nexon 계정 문제~~ — 정정 및 재확인 완료(2026-07-09): 앞서 "사용자는 단일 계정만 써서 리스크 없음"으로 정리했던 건 **실측 데이터로 반증됨**. 실제로 `character/list` 응답을 보면 `account_list`에 서로 다른 `account_id` 2개가 존재하고 각각 별도의 캐릭터 목록을 가진다. 사용자 설명: "하나의 넥슨 계정에 여러 메이플 ID가 존재할 수 있음" — 즉 걱정했던 "서로 다른 Nexon 로그인 계정(별도 API 키 필요)" 문제가 아니라, **하나의 Nexon 계정·API 키로 여러 "메이플 ID"(게임 계정)의 캐릭터가 한꺼번에 조회되는 게 정상 동작**이다. 이건 이제 v1 범위 밖 리스크가 아니라 **반드시 설계해야 하는 핵심 흐름**이 됐다 — 아래 "계정(메이플 ID) 선택" 항목 참고.
- **게임 내 스케줄러 등록이라는 전제의 실효성**: "스케줄러" 기능 자체가 2026-06-25 출시라 대부분의 유저가 아직 써본 적 없을 가능성이 높다. 이 앱을 처음 켜는 유저 상당수가 "게임에서 아무것도 등록 안 해서 앱이 텅 비어 보이는" 상태를 겪을 수 있다 — 이건 버그가 아니라 예상된 첫 화면 상태이므로, 빈 상태(empty state) UI와 온보딩 안내가 필수다.
- ~~리셋 타이밍 정합성 미검증~~ — 확인 완료(2026-07-09): Nexon 서버는 한국시간(KST) 기준 목요일 00:00에 초기화되며, `lib/reset-clock`이 계산하는 일간 00:00 KST / 주간 목요일 00:00 KST 리셋 시각과 정확히 일치한다.
- **API 성숙도 리스크**: 2주 차 API라는 것은 단순히 "문서가 부족하다"를 넘어, 스펙이 실제로 바뀔 수 있다는 뜻이다. 이 기능을 "베타" 취급하고, 동기화가 계속 실패할 경우 "게임 내에서 직접 확인해주세요" 같은 폴백 안내 문구를 준비해야 한다 (에러 핸들링은 [[ADR-008]] 참고).

**실제 응답 스키마 예시** (2026-07-09, 사용자 제공, 캐릭터 "낟낟" 조회 결과 일부):
```json
{
  "date": "2026-07-09T00:00+09:00",
  "character_name": "낟낟",
  "world_name": "엘리시움",
  "character_level": 293,
  "character_class": "렌",
  "daily_contents": [
    { "content_name": "몬스터파크", "type": "contents", "registration_flag": "true", "now_count": 7, "max_count": 14, "quest_state": null }
  ],
  "weekly_contents": [
    { "content_name": "에픽 던전 : 악몽선경", "type": "contents", "registration_flag": "true", "now_count": 5, "max_count": 0, "quest_state": null },
    { "content_name": "[메이플 유니온] 주간 드래곤 퇴치", "type": "quest", "registration_flag": "false", "now_count": 0, "max_count": 0, "quest_state": "0" }
  ],
  "boss_contents": [
    { "content_name": "검은 마법사", "difficulty": "extreme", "cycle": "bossMonthly", "list_order_no": 53, "registration_flag": "true", "complete_flag": "true" }
  ],
  "weekly_boss_clear_count": 0,
  "weekly_boss_clear_limit_count": 0
}
```

**필드 주의사항**:
- `registration_flag`/`complete_flag`는 boolean이 아니라 문자열 `"true"`/`"false"`로 온다. 파싱 시 문자열 비교 필요.
- `difficulty`는 영문 소문자(`hard`, `extreme` 등)로 온다. `src/data/`는 한글 표기(`하드`, `익스트림`)를 쓰므로 매핑 테이블이 필요하다.
- `content_name`의 보스명이 `src/data/weekly-bosses.json` 등의 표기와 띄어쓰기가 다를 수 있다(예: API `"검은 마법사"` vs 우리 데이터 `"검은마법사"`, 반대로 API `"블러디퀸"` vs 우리 데이터 `"블러디 퀸"`). **공백 방향이 일정하지 않으므로, 조인 전 양쪽 문자열에서 공백을 전부 제거하고 비교해야 한다** — "API 쪽에 공백이 더 있다"처럼 한쪽으로 가정하면 틀린다(2026-07-09, 실제 응답으로 재확인).
- `cycle`은 확정: `"bossDaily"` / `"bossWeekly"` / `"bossMonthly"` 세 값 중 하나다(2026-07-09, 사용자 실측). 이 앱은 **`bossWeekly`와 `bossMonthly`만 사용**하고 `bossDaily` 항목은 무시한다 — 힐라(하드)·핑크빈(카오스)처럼 원래 주간 보스였다가 일간으로 격하된 보스들이 실제로 `bossDaily`로 온다는 게 이 응답으로도 확인됨([[ADR-006]] 데이터 정리와 일치).
- **이름 별칭 필요 — "시즌 보스 메이린"**: API는 메이린을 `content_name: "시즌 보스 메이린"`으로 반환한다(공백 제거만으로는 못 잡는 접두어 차이). `src/data/weekly-bosses.json`의 `메이린` 엔트리와 매핑하려면 별도 별칭 테이블 항목이 필요하다.

**확정 (2026-07-09)**:
- 일간/주간 화면은 완전 읽기 전용 — API 값만 표시, 앱 내 수동 체크 없음
- 알림 발송 판단은 마지막 캐시가 아니라 **알림 시각에 실시간 API 재호출**로 확인 (세부 사항은 [[ADR-004]] 참고)
- `weekly_boss_clear_count`/`weekly_boss_clear_limit_count`의 의미: 메이플스토리에는 "캐릭터당 주간 보스는 최대 12마리(난이도 조합 단위)까지만 처치 가능"이라는 게임 자체 제한이 있고, 이 두 필드가 정확히 그 카운트(현재 처치 수/상한 12)다. 단, "시즌보스"(현재는 카이·메이린뿐, 카이는 [[ADR-006]]에 따라 시즌 종료로 목록에서 이미 제외)는 이 12마리 제한에서 예외라 카운트에 잡히지 않는다. 샘플 응답에서 값이 `0`이었던 건 그 응답에 포함된 완료 보스(검은 마법사)가 월간 보스라 애초에 주간 카운트 대상이 아니었기 때문 — 필드 자체는 미스터리가 아니었고 샘플 데이터 선택의 문제였다(2026-07-09 확인).

**확정 — 캐릭터 목록 조회 및 응답 스키마 (2026-07-09, 사용자 실측)**: `GET /maplestory/v1/character/list`로 캐릭터 목록을 자동 조회한다. 캐릭터명+월드를 수동 입력하는 등록 폼은 두지 않는다.
```json
{
  "account_list": [
    {
      "account_id": "da9b2f2...",
      "character_list": [
        { "ocid": "50119a0...", "character_name": "내옆에최성일", "world_name": "베라", "character_class": "아크메이지(썬,콜)", "character_level": 211 }
      ]
    },
    {
      "account_id": "69e3525...",
      "character_list": [
        { "ocid": "23be5de...", "character_name": "낟낟", "world_name": "엘리시움", "character_class": "렌", "character_level": 293 }
      ]
    }
  ]
}
```
**확정 — 계정(메이플 ID) 선택 UI 필요**: 응답이 `account_list` 배열이라는 것 자체가 "하나의 Nexon 계정 안에 여러 메이플 ID가 있을 수 있다"는 게임 시스템을 그대로 반영한다. 따라서:
1. API 키를 처음 입력하면 `character/list`를 호출해 `account_list`를 받고, 항목이 2개 이상이면 **어느 메이플 ID를 쓸지 선택하는 화면**을 보여준다(예: 각 계정의 캐릭터 수·대표 월드명으로 구분 표시 — `account_id` 자체는 해시라 사람이 읽기 어려움).
2. 선택된 계정의 `character_list`만 앱에 캐릭터로 등록한다.
3. 설정 화면에서 **나중에 계정을 변경**할 수 있어야 한다(확정, 사용자 지시 2026-07-09).
4. ocid 문자열 길이가 계정/캐릭터마다 다르게 관찰됨(예: 32자 vs 65자) — 고정 길이로 가정하지 말고 불투명한 식별자 문자열로만 다뤄야 함.

**확정 — 스케줄러 API 엔드포인트 (2026-07-09, 사용자 제공)**: `GET /maplestory/v1/scheduler/character-state` (ocid 파라미터로 호출, 응답 스키마는 위 샘플 참고)

**추가 확인(2026-07-14, `openapi.nexon.com` 공식 문서)**: 이 엔드포인트는 `date`(query, string, `YYYY-MM-DD`) 파라미터도 지원한다 — "조회 기준일, 미입력 시 오늘 날짜. 캐릭터가 해당 기준일에 접속하지 않은 경우, 응답 결과가 없을 수 있음"이 문서 원문이다. 지금까지 이 앱의 어떤 호출도 이 파라미터를 쓰지 않았다(`nexon/schedule/client.ts`는 `ocid`만 쿼리스트링에 넣는다) — [[ADR-023]]의 과거 기간 조회·백필 기능이 이 파라미터를 최초로 활용한다.

**확정 — API 호출 방식·제한 (2026-07-09, 공식 가이드 `openapi.nexon.com/ko/guide/`·FAQ 확인)**:
- 실제 API 호출 도메인은 문서 사이트(`openapi.nexon.com`)와 다른 `https://open.api.nexon.com/`이다. 예: `https://open.api.nexon.com/maplestory/v1/...`
- API 키는 HTTP 요청 헤더 `x-nxopen-api-key`에 담아 전송한다.
- 호출 제한은 애플리케이션 단계에 따라 다르다: **개발 단계** 초당 5건 / 일 1,000건, **서비스 단계** 초당 500건 / 일 20,000,000건. 서비스 단계로 전환하려면 API Key를 새로 발급받아야 한다(신규 발급, 기존 키 재사용 아님).
- 호출량 초과 시 에러 코드 `OPENAPI00007`, HTTP 상태 `429`("Too Many Requests")를 반환한다 — [[ADR-008]]의 429 처리 정책과 일치.
- **개발 단계(1,000건/일) 기준으로도 개인 사용에는 충분할 것으로 추정**: 캐릭터 수 개 × (일간/주간 동기화 + 알림 시각 실시간 재확인)을 감안해도 하루 수십 건 수준이라 여유가 크다. 다만 5건/초 제한이 있으므로, 여러 캐릭터를 동시에 병렬 호출하지 않고 약간의 간격을 두고 순차 호출하도록 구현해야 한다.

**확정 — 온보딩 및 이용약관 표기 (2026-07-09)**:
- 온보딩 흐름: API 키 발급 안내는 앱 내 단계별 위저드가 아니라, **openapi.nexon.com 링크 제공 + 간단한 샘플 이미지 첨부 + 설명 문구**로 안내한다(가벼운 안내, 전용 튜토리얼 아님).
- NEXON Open API 이용 출처 표기(약관 제6조④, 위 "이용약관 준수 사항")는 **정정(2026-07-13)**: ~~앱 footer~~ → **설정 화면 하단**에 영문 문구 그대로 "Data based on NEXON Open API"를 넣는다. 같은 위치에 앱 버전(`package.json`의 `version`)과 카피라이트 문구("© {연도} 메이플 루틴")도 함께 둔다.
- 계정(메이플 ID) 선택 UI: 각 `account_id`는 그 계정 내 **가장 레벨이 높은 캐릭터의 닉네임 + 직업 + 레벨**로 표기한다(예: "낟낟 · 렌 Lv.293"). `account_id` 해시 자체는 화면에 노출하지 않는다.

### ADR-008: 에러 핸들링 및 복원력 정책
**결정**: 이 앱이 의존하는 외부 요소(Nexon Open API, 로컬 저장소, OS 알림/타이머 API)는 모두 실패할 수 있다고 가정하고, 각 실패 모드마다 "마지막으로 알려진 정상 상태를 보여주고, 사용자에게 무엇이 실패했는지 명확히 알리고, 자동 재시도는 제한적으로만 한다"는 원칙을 따른다.

**Nexon Open API 호출 실패**:
- 네트워크 없음/타임아웃/5xx: 마지막 동기화 캐시를 그대로 표시하고, 화면에 "마지막 동기화: n분 전 · 새로고침" 표시. 자동 재시도는 하지 않고 사용자가 새로고침 버튼을 눌러야 재시도(백그라운드에서 무한 재시도로 배터리를 소모하지 않기 위함).
- 401/403(키 무효·삭제·재발급됨): 캐시는 유지하되 "API 키가 유효하지 않습니다 — 설정에서 다시 입력해주세요" 배너 표시. 무효 키로 반복 호출하지 않는다.
- 429(rate limit, 에러 코드 `OPENAPI00007`, [[ADR-007]] 참고): 지수 백오프로 다음 자동 동기화 시점을 늦추고, 사용자가 새로고침을 연타해도 최소 간격 내에는 실제 호출을 보내지 않는다(클라이언트 측 쓰로틀링). 여러 캐릭터를 동기화할 때도 병렬 호출 대신 초당 5건(개발 단계 기준) 이내로 순차 호출한다.
- ocid 조회 실패(캐릭터명/월드 불일치, 캐릭터 없음): 해당 캐릭터 등록 단계에서 즉시 에러 표시, 저장하지 않는다.
- 응답 스키마 불일치(신규 필드 추가, 필드 누락, 타입 변경): 파싱은 필드 단위로 방어적으로 처리한다 — 한 항목 파싱 실패가 전체 동기화를 실패시키지 않고, 해당 항목만 "표시 불가"로 건너뛴다. `src/data/` 매핑 테이블에 없는 보스명은 "알 수 없는 콘텐츠"로 원문 그대로 표시(앱이 죽지 않도록).
- **응답 자체가 JSON이 아닌 경우**(WAF/CDN 차단 페이지, 502 HTML 등): 필드 단위 파싱 이전 단계의 실패이므로, JSON 파싱 자체를 try/catch로 감싸 "네트워크 없음/타임아웃/5xx"와 동일하게 마지막 캐시 유지로 처리한다(응답 스키마 불일치와는 별도 분기).
- **HTTP 타임아웃 값**: 10초로 확정한다(2026-07-09). 공식 문서에 권장값이 없어 모바일 API 호출 관례상 채택.
- **호출량 한도**: 개발 단계 한도(1,000건/일)는 신경 쓰지 않기로 확정한다(2026-07-09) — 이 앱은 사용자가 이미 다른 서비스로 승인받은 서비스 단계 API 키(초당 500건/일 2,000만 건)를 그대로 사용하므로, 개발 단계 호출량 상한 계산이 더 이상 필요하지 않다.

**로컬 저장소 실패**: 쓰기 실패(디스크 부족 등) 시 사용자가 방금 한 조작(체크, 아이템 획득 기록 등)이 반영 안 됐음을 즉시 알린다 — 조용히 무시하지 않는다.
- **온보딩 중 API 키 저장 실패**: Keychain/Keystore 쓰기가 실패하면(권한 문제, 저장 공간 등) 온보딩을 완료 상태로 넘기지 않고 "키 저장에 실패했습니다, 다시 시도해주세요" 표시 후 재시도를 유도한다 — 이 단계는 앱의 유일한 진입 게이트라 일반 저장 실패보다 명시적으로 다뤄야 한다.
- **로컬 DB 스키마 마이그레이션(미정)**: 앱 업데이트로 로컬 저장소 스키마가 바뀌는 경우의 마이그레이션 전략이 아직 없다. 실제 스키마가 확정되는 시점에 별도 결정 필요.

**알림/타이머 네이티브 실패**: OS가 알림 권한을 거부하면 기능 진입 시점에 권한 요청 및 거부 시 안내. Android Foreground Service가 배터리 최적화로 종료되거나 iOS Live Activity 시작이 실패하면([[ADR-005]]), 상시 알림 없이도 최소한 완료 알림/사운드는 로컬 알림으로 폴백한다.
- **권한이 나중에(런타임 중) 취소되는 경우**: 최초 요청 시점 거부뿐 아니라, 사용자가 OS 설정에서 언제든 알림 권한을 끌 수 있다. 알림 예약/재확인 시점에 권한 상태를 다시 확인하고, 취소돼 있으면 예약을 건너뛰고 앱 내 배너로 "알림 권한이 꺼져 있습니다" 안내한다.
- **로컬 알림 64개 한도는 사냥 타이머 폴백 알림과 일간/주간 알림이 OS 레벨에서 같은 풀을 공유한다**(iOS/Android 모두 앱 단위 전역 한도). [[ADR-004]]의 64개 한도 우선순위 계산 시 사냥 타이머용 예약분도 포함해야 한다.

**크래시/에러 리포팅 — opt-in (확정, 2026-07-09)**: [[ADR-003]]의 "서버에 사용자 데이터를 저장하지 않는다"는 원칙과, 실제 프로덕션에서 발생하는 크래시·동기화 실패를 개발자가 전혀 알 수 없다는 현실적 문제 사이의 트레이드오프다. 설정 화면에 "익명 오류 보고" 토글을 두고, 기본값 OFF로 시작해 사용자가 명시적으로 켠 경우에만 Sentry(또는 유사 서비스)로 스택트레이스·에러 유형만 전송한다. 캐릭터명·API 키 등 개인 식별 가능 정보는 전송 페이로드에서 제외한다. 이 예외는 [[ADR-003]]에 "서버 없음" 원칙의 일부로 이미 인정된 Nexon API 직접 호출과 마찬가지로, 우리가 만든 백엔드가 아니라 제3자 SaaS로의 옵트인 전송이라는 점에서 원칙을 벗어나지 않는다.

**데이터 정합성 — 참조 무결성**: `src/data/`에서 보스가 제거되는 경우(예: 카이, [[ADR-006]])에도 그 보스를 참조하는 과거 로컬 기록(수익/드랍 히스토리)은 삭제하지 않는다. UI는 참조 테이블에 없는 보스명을 만나면 에러를 던지지 않고 "(더 이상 지원하지 않는 콘텐츠)" 같은 표시로 과거 기록을 그대로 보여준다.

**멱등성 — 중복 동기화**: 같은 주 안에 앱을 여러 번 열어 같은 보스의 `complete_flag: true`를 여러 번 관측해도, 보스 수익 기록은 캐릭터+보스+난이도+주차 조합당 1건만 생성한다(중복 생성 방지, 이미 있으면 갱신).

**이유**: 이 앱이 의존하는 두 외부 시스템(Nexon Open API, OS 네이티브 API) 모두 이 세션에서 처음 다뤄보거나(스케줄러 API 2주 차) 플랫폼 제약이 있는(iOS Live Activity 16.1+, 로컬 알림 64개 한도) 요소라, 실패를 예외 상황이 아니라 정상적으로 발생하는 상태로 다뤄야 한다.
**트레이드오프**: 모든 실패 모드에 대한 UI 상태(로딩/에러/빈 상태/부분 실패)를 각 화면마다 설계해야 해서 초기 구현 범위가 늘어난다. 그러나 이걸 생략하면 "동기화가 안 되는데 이유를 몰라 게임을 다시 켜보는" 식의 나쁜 사용자 경험으로 이어진다.

### ADR-009: 직업(전직) 기반 테마 시스템 도입
**결정**: 캐릭터 직업 고유 컬러를 Primary 색상으로 쓰는 다중 테마 시스템을 도입한다. `docs/UI_GUIDE.md`의 색상 파생 규칙(Hover/Active/Subtle/Border 계산, 라이트/다크 배경별 텍스트 대비 기준)은 모든 테마에 공통 적용하고, 테마마다 색상 값을 교체한다.
**정정(2026-07-10)**: ~~테마마다 바뀌는 건 Primary 색상 값 하나뿐~~ — 사용자가 실제 예시 테마("레테": primary `#452951`(진보라)·secondary `#A7446A`(로즈)·third `#FBFB8A`(연노랑))를 주면서 확인됨: 직업 테마는 **3색 세트**(Primary/Secondary/Third)로 온다. 3색 각각의 역할은 색상마다 다르게 매핑한다(와이어프레임 적용 사례 참고): Primary는 채움 배경 + 텍스트/아이콘 잉크 겸용(밝기가 충분히 낮으면 별도 잉크 보정 불필요), Secondary는 hover/눌림 상태 + 보더, Third는 옅은 강조 배경(subtle) + 채움 위 텍스트색. 이 매핑은 테마별 밝기 구성에 따라 달라질 수 있어 고정 공식이 아니라 색상마다 판단이 필요하다.
**이유**: 메이플스토리 직업 고유 컬러를 활용하면 사용자가 자신의 캐릭터/직업에 애착을 느끼는 개인화 경험을 제공할 수 있다.
**구현 방향(뼈대, 2026-07-09)**:
- 컬러 값은 `src/data/job-themes.json`에 직업별 참조 테이블로 저장한다. 다른 게임 레퍼런스 데이터와 동일하게 [[ADR-006]] CRITICAL 규칙이 적용되어, 실제 컬러 값은 AI가 임의로 채우지 않고 사용자(도메인 전문가) 확정 후 반영한다 — 현재 미작성
- 선택된 테마는 다른 전역 클라이언트 상태와 동일하게 Zustand로 관리하고 `storage/`에 영속화한다
- Tailwind는 정적 유틸리티 클래스 기반이라 런타임에 값이 바뀌는 다중 테마와 궁합이 안 맞는다 — Primary 관련 값은 CSS 커스텀 프로퍼티(예: `--color-primary`)로 노출하고, 선택된 테마 값을 `:root`에 주입하는 방식을 제안(구현 시 확정)
**트레이드오프**: Tailwind arbitrary value(`bg-[#...]`)를 그대로 못 쓰고 `bg-[var(--color-primary)]` 형태의 간접 참조가 필요해진다. `UI_GUIDE.md`에 정의한 정적 색상값(기본 테마)도 테마 적용 시점에는 CSS 변수 참조로 바뀌어야 한다.
**미확정 — 추후 확정 필요**:
- 직업별 실제 컬러 값(hex) — "레테" 예시는 검증용 테스트 팔레트로 와이어프레임에만 적용했고, `src/data/job-themes.json`에는 아직 반영하지 않았다(실제 직업 매핑인지 사용자 확인 전까지 [[ADR-006]] CRITICAL 규칙 적용)
- `job-themes.json` 스키마를 `{ primary, secondary, third }` 3필드로 확정할지, 색상 역할 매핑(어떤 색이 채움/보더/텍스트인지)을 테마마다 수동 지정할지 자동 규칙(예: 가장 어두운 색 = 채움)으로 계산할지
- 테마 단위: 직업 대분류(전사/마법사/궁수/도적/해적/초보자 등)인지 5차 전직 세부 단위인지

- **정정(2026-07-09)**: ~~테마 미선택 시 기본값 유지 여부~~ — 확정: 기본값은 계속 오렌지(#FF7033, `UI_GUIDE.md` "기본 테마")를 쓰되, 배경 팔레트는 다크에서 **라이트로 전환**했다(게임 앱다운 캐주얼한 톤을 위해, PRD "디자인" 섹션 참고). 다크는 폐기하지 않고 보조 팔레트로 유지

**정정 — 테마 이름 한글화 + 4번째 테마 "혼테일" 추가 + 시스템 다크 모드 연동(2026-07-14)**: 사용자가 `mushmom` 표기를 다른 테마와 동일하게 한글 "머쉬맘"으로 정정 요청 — `ThemeName` 타입·`job-themes.json` 키·`ThemeSelector.tsx`·`storage/theme.ts`·`features/theme/store.ts`·테스트 전체에서 `'mushmom'`을 `'머쉬맘'`으로 개명한다(값 자체는 불변). 동시에 4번째 테마 "혼테일"(다크)을 17토큰 값과 함께 추가했다(값은 `docs/UI_GUIDE.md` "테마 시스템" 표 참고, [[ADR-006]] 확인 완료). "혼테일"을 **기본 다크 테마**로 지정하면서, "시스템 컬러로 자동으로 라이트/다크 테마가 바뀌는" 동작을 요청 — 기존에는 저장된 테마가 없으면 무조건 "머쉬맘"이었는데, 이제 `restoreFromStorage()`에서 `window.matchMedia('(prefers-color-scheme: dark)')`로 OS 설정을 확인해 라이트면 "머쉬맘", 다크면 "혼테일"을 기본값으로 쓴다. 앱 실행(정확히는 `restoreFromStorage` 호출) 시점에 1회만 판정하고 실시간으로 OS 설정 변경을 감지해 앱 실행 중 전환하지는 않는다(스코프 축소 — 필요해지면 추후 `matchMedia` change 리스너를 추가). 사용자가 설정 화면에서 테마를 한 번이라도 명시적으로 선택하면 그 값이 영속화되어 이후에는 시스템 설정과 무관하게 그 값을 쓴다(기존 수동 선택 동작과 동일). `src/index.css`의 `@theme` 기본 블록은 계속 "머쉬맘" 값이고, "혼테일"은 "레테"·"렌"과 동일하게 `:root[data-theme='혼테일']` 오버라이드 블록을 갖는다. jsdom 테스트 환경은 `window.matchMedia`를 기본 제공하지 않아, `resolveSystemTheme()`은 `window.matchMedia`가 없으면 라이트("머쉬맘")로 안전하게 폴백한다.

### ADR-010: 물욕템 수익 반영 및 랜덤 컨테이너 아이템 결과 기록
**결정**:
1. 물욕 아이템 드랍 기록에 아이템별 고정 시세를 반영해, "주간 보스 수익 계산기" 화면에서 결정 판매 수익과 물욕템 환산 가치를 항목별로 구분 표시하고 합계도 함께 보여준다. 새 화면을 만들지 않고 기존 화면을 "이번 주 / 월간 추이" 탭으로 확장한다(2026-07-09 확정).
2. "OO옥의 보스 반지 상자"·"혼돈의 칠흑 장신구 상자" 같은 컨테이너형(랜덤) 아이템은 개봉 결과(반지 레벨·반지 종류, 또는 장신구 종류)가 확률적으로 정해지므로, 획득 기록 시 사용자가 실제로 나온 결과를 후보 목록에서 직접 선택해 기록한다. 확률로 자동 추정하지 않는다.
**이유**: 사용자가 실제로 획득한 결과를 정확히 남기는 게 이 앱의 핵심 가치다([[ADR-008]] 참조 무결성 원칙과 동일한 맥락) — 확률 기반 추정은 실제 기록과 다를 수 있어 신뢰할 수 없다. 화면을 새로 만들지 않는 이유는 사용자가 두 수익원(보스 결정 판매·물욕템)을 "이번 주 수익"이라는 하나의 맥락에서 보고 싶어하기 때문(2026-07-09 확정).
**데이터**:
- `src/data/boss-ring-boxes.json` 신규(2026-07-09) — 녹옥/홍옥/흑옥/백옥/생명의 보스 반지 상자 5종의 레벨별 확률표·반지별 확률표(사용자 제공, 게임산업법 시행령 기준 반올림 표기). 각 박스가 어느 보스+난이도에서 드랍되는지는 `item-drop-table.json`의 `consumable` 카테고리(주석 "교환 불가")에 이미 반영되어 있어 별도 매핑을 만들지 않는다.
- `src/data/accessory-boxes.json` 신규(2026-07-09) — "혼돈의 칠흑 장신구 상자"·"메이린의 칠흑 장신구 상자"(이름만 다르고 후보 목록 동일, 사용자 확인)의 후보 아이템 7종(사용자 제공 아이템 설명 이미지). 반지 상자와 달리 게임 내 툴팁에 개별 획득 확률(%)이 공개돼 있지 않아 후보 목록만 확정, 확률 필드는 `null`
- 확률표는 "실제 획득 결과 기록 시 선택할 후보 목록" 용도로만 쓴다. 확률 수치 자체를 UI에 노출할지는 별도 결정 사항(미정).
- 가격 필드(`priceMeso`) 네이밍은 `boss-crystal-prices.json`과 동일한 컨벤션을 따르며, 값이 확정되기 전까지는 `null`로 둔다([[ADR-006]] CRITICAL 규칙).
**미확정 — 추후 확정 필요**:
- 물욕템(반지·장신구 포함) 개별 시세 — 아직 없음. 가격 없는 항목은 수익 계산에서 0원이 아니라 "가격 미확정"으로 별도 표시하고 합계에서 제외한다
- `item-drop-table.json`의 다른 항목들에 `priceMeso` 필드를 실제로 언제/어떻게 채울지 — 스키마 컨벤션만 정했고 기존 파일(2,100줄+)에 일괄로 `null` 필드를 미리 채워 넣지는 않았음(값 없는 필드를 대량으로 미리 넣는 건 정보가 없어 보류, 실제 시세 반영 시 함께 추가)
- 개봉해서 나온 개별 반지의 거래 가능 여부 — 박스 자체는 "교환 불가"로 명시돼 있으나 반지 자체의 거래 가능 여부는 미확인

### ADR-011: 물욕 아이템 드랍 화면의 보스 목록 출처, 난이도 무관 아이템 노출, 이미지 에셋
**결정**:
1. "물욕 아이템 드랍 현황"의 **보스 목록**은 독립적인 목록이 아니라, **주간 스케줄러(feature 2)와 동일한 Nexon 동기화 캐시**(`boss_contents` 중 `cycle: bossWeekly`, `registration_flag: true`)에서 가져온다. 즉 게임 내 스케줄러에 실제로 등록된 주간 보스만 뜬다(확정, 2026-07-09).
2. 보스를 선택해 들어간 상세 화면에서는 그 보스의 물욕템을 **난이도 구분 없이 전부 합쳐서** 보여준다 — `item-drop-table.json`에서 `boss` 이름이 같은 모든 엔트리(노멀/하드/익스트림/카오스 등)의 아이템을 이름 기준으로 합치고 중복 제거한다. 보스 목록·아이템 그리드 어디에도 난이도 표기를 하지 않는다(확정, 2026-07-09).
3. 물욕템 버튼은 텍스트가 아니라 **이미지**로 표시한다. 이미지 파일은 사용자가 직접 제공하며, `src/assets/items/`에 **아이템명(공백→언더스코어)를 파일명으로** 저장한다(예: `리스트레인트 링` → `리스트레인트_링.png`). `item-drop-table.json`에 이미지 경로 필드를 추가하지 않고, 아이템명으로부터 파일 경로를 계산하는 헬퍼 함수(`lib/item-icons`)로 조회한다. 이미지가 없는 항목은 플레이스홀더 아이콘으로 폴백한다(확정, 2026-07-09).
4. **이미지 에셋 2분류 확정(2026-07-10)**: 아이템 이미지 외에 **보스 초상화**도 별도 에셋으로 관리한다. "보스를 나타내는 컴포넌트"(주간 스케줄러의 보스 항목, 보스 수익 계산기의 보스 행, 물욕 아이템의 보스 목록·상세 헤더 등 — feature 2·4·5에 걸쳐 재사용)에서 텍스트 대신 이 초상화를 표시한다. 아이템과 동일한 컨벤션을 그대로 따른다: `src/assets/bosses/`에 **보스명(공백→언더스코어)을 파일명으로** 저장(예: `검은 마법사` → `검은_마법사.png`), 난이도별이 아니라 **보스 1종당 초상화 1장**(ADR-011 결정 2의 "난이도 무관" 원칙과 동일한 이유 — 초상화도 난이도로 쪼개지 않는다), 조회는 `lib/boss-icons` 헬퍼, 이미지 없으면 플레이스홀더 폴백. 반복되는 "이름 → 에셋 경로, 없으면 폴백" 로직은 `lib/item-icons`와 `lib/boss-icons`에 각각 독립적으로 둔다(로직이 한 줄짜리라 공용 추상화를 만들 정도는 아님).
5. **반지류는 한글명이 아니라 영문(GMS) 파일명 사용(확정, 2026-07-11)**: 특수 스킬 반지(`src/assets/items/rings/`)는 한글 아이템명을 파일명 변환 규칙만으로 못 옮긴다(예: "리스트레인트 링" → `Ring_of_Restraint.png`, 기계적 변환이 아니라 실제 GMS 명칭). 그래서 이 계열만 예외적으로 `boss-ring-boxes.json`의 각 아이템 항목에 `iconFile` 필드를 직접 추가해 한글명↔영문 파일명을 명시 매핑한다(나무위키 "특수 스킬 반지" 문서의 GMS-MSEA 표기 기준으로 사용자가 확인). "컨티뉴어스 링"만 나무위키에 GMS 표기가 없어 다른 반지들의 명명 규칙(형용사/명사 + Ring)에 맞춰 `Continuous_Ring.png`로 직접 지음 — 공식 확인된 이름이 아니라는 점을 데이터에도 명시해둠. 웨폰퍼프·레벨퍼프는 스탯(S/I/L/D)별로 별도 아이콘이 없어 스탯 무관 공용 파일(`Weapon_Jump_Ring.png`/`Level_Jump_Ring.png`)을 공유한다.
5. **`items/` 하위 `rings/` 서브폴더 신설(확정, 2026-07-10)**: 특수 스킬 반지(`boss-ring-boxes.json`에 나오는 리스트레인트 링 등, 30여 종)는 종류가 많아 `src/assets/items/rings/`로 따로 묶는다. `lib/item-icons`는 아이템명이 **"링"으로 끝나면** `items/rings/{이름}.png`, 아니면 `items/{이름}.png`를 조회한다(접미사 휴리스틱 — `boss-ring-boxes.json`의 반지 30종은 전부 "링"으로 끝나고, 같은 박스에서 나오는 비반지 아이템 "생명의 연마석"만 유일한 예외인데 이건 "링"으로 안 끝나 정확히 걸러짐. 실측 데이터로 검증된 규칙이지 추정 아님). 디렉토리는 미리 만들어둠(`.gitkeep`), 실제 이미지는 사용자가 추가.
6. **정정(2026-07-11) — 일반 아이템·보스 초상화도 파일명 그대로 쓰고 매핑 테이블로 조인한다**: 결정 3·4의 "한글명(공백→언더스코어)을 파일명으로" 컨벤션은 폐기한다. 사용자가 실제로 넣어둔 파일은 영문 내부 코드명 스타일이었고("변환"이 아니라 "이미 있는 파일명을 그대로 쓰고 우리 데이터와 매칭만 하라"는 사용자 지시, 2026-07-11), 반지 때와 같은 이유로 이름→파일명이 기계적 변환이 아니라 **명시적 매핑 테이블**이 필요하다.
   - **보스 초상화**: `weekly-bosses.json`의 각 보스 항목에 `portraitSlug` 필드 추가(예: `루시드` → `lucid`). 실제 파일명은 `{난이도접두사}_{portraitSlug}.png`이며 난이도 접두사는 이지→`easy`, 노멀→`normal`, 하드→`hard`, 카오스→`chaos`, 익스트림→`extreme`로 고정. 나무위키 "GMS 명칭" 확인 + 보스별 실제 파일 개수가 `difficulties` 배열 길이와 전부 일치하는 구조적 교차검증으로 확인함. `portraitSlug`가 없는 보스는 아직 이미지가 없다는 뜻(자쿰·매그너스·파풀라투스·반반·피에르·블러디 퀸·벨룸·벨로나).
   - **일반 아이템**: 신규 `src/data/item-icons.json`에 "한글 아이템명 → 기존 파일명" 매핑을 별도 테이블로 관리한다(`item-drop-table.json`은 건드리지 않음 — 같은 아이템이 여러 보스에 걸쳐 반복 등장해서 그 파일에 필드로 넣으면 중복이 심함). 확신도 높은 27개 항목만 우선 반영했고, 후보가 여러 개라 확정 못 한 것들은 사용자에게 별도로 물어봄(아래 미확정 참고).
   - `lib/item-icons`·`lib/boss-icons`는 이제 "이름으로부터 파일 경로를 계산"이 아니라 "매핑 테이블에서 조회, 없으면 플레이스홀더"로 동작이 바뀐다.
**이유**:
- 사용자 실측: "유저들이 자신이 처치하는 보스 난이도를 정확하게 등록해놓지 않았을 수도 있다" — 난이도 기준으로 아이템을 필터링하면 실제로 얻은 아이템이 목록에 없어서 기록을 못 하는 상황이 생길 수 있다. 난이도 대신 보스 단위로 전부 보여주는 게 더 안전하다.
- 이미지를 JSON 필드로 관리하지 않고 파일명 컨벤션으로 조회하는 이유: `item-drop-table.json`은 이미 사용자가 확정 반영을 완료한 2,100줄+ 파일이라([[ADR-006]]), 값이 아직 없는 필드를 대량으로 미리 넣는 건 [[ADR-010]]의 `priceMeso` 때와 동일한 이유로 보류한다 — 파일명 컨벤션 방식은 스키마 변경 없이 이미지 유무와 무관하게 항상 유효한 조회 경로를 제공한다.
**트레이드오프**: 아이템명·보스명이 바뀌면(오탈자 수정 등) 이미지 파일명도 같이 바꿔야 한다(고정 ID가 아니라 이름 기반 조회라서 생기는 결합). 아이템 수가 많아지면(300+) 파일명 기반 조회가 매핑 테이블보다 관리가 번거로워질 수 있음 — 그 시점엔 재검토. 보스명은 `weekly-bosses.json`이 이미 API 원문↔우리 표기 정규화(공백 제거 비교, `apiAlias`)를 하고 있으므로, 초상화 조회는 정규화된 **우리 쪽 보스명**(원문 API 문자열이 아니라)을 키로 써야 한다. "링" 접미사 휴리스틱은 현재 데이터로는 정확하지만, 향후 반지가 아닌데 이름이 "링"으로 끝나는 아이템이 추가되면 오분류될 수 있다 — 그 시점엔 명시적 분류로 전환.
**정정 — ARCHITECTURE.md 데이터 흐름 오류 수정**: 기존 문서에 "[보스 수익 계산기 / 물욕 아이템 드랍 — 기존과 동일, Nexon API와 무관]"이라고 적혀 있었는데, 이는 부정확했다. 보스 수익 계산기는 애초에 PRD 핵심 기능 4에 "Nexon API 동기화 데이터를 그대로 사용"이라고 명시돼 있었고([[ADR-007]]), 이번 ADR-011로 물욕 아이템 드랍의 보스 목록도 같은 캐시를 쓰게 되어 두 기능 모두 Nexon API 동기화 데이터에 의존한다. "API와 무관"한 부분은 파티원 수 입력·아이템 획득 기록·수익 계산 자체(로컬 전용)로 한정된다.

### ADR-012: 일간/주간 스케줄러 캐릭터 추적(선택) 정책 및 동기화 범위 제한
**결정**:
1. 일간 스케줄러와 주간 스케줄러는 각각 **독립적으로** "추적할 캐릭터"를 사용자가 직접 고른다 — 계정에 등록된 캐릭터 전체가 아니라 사용자가 선택한 부분집합만 화면에 표시한다. 두 화면의 추적 목록은 서로 공유하지 않는다(일간에서 추적하는 캐릭터와 주간에서 추적하는 캐릭터가 다를 수 있다, 확정 2026-07-11).
2. 아직 아무 캐릭터도 고르지 않았으면(최초 실행 등) 빈 상태("캐릭터를 선택해주세요")로 표시한다. 화면 안의 "캐릭터 관리" 버튼으로 언제든 후보 목록(계정의 전체 캐릭터)에서 체크박스로 고를 수 있고, 체크 상태는 "저장"을 눌러야 확정되며 취소 시 반영되지 않는다.
3. 화면에 표시되는 콘텐츠(일간 콘텐츠, 주간 퀘스트, 주간 보스)는 그중에서도 게임 클라이언트 스케줄러에 실제로 **등록된**(`registration_flag: "true"`) 항목만 보여준다 — 미등록 항목은 화면에서 완전히 제외한다(PRD 핵심 기능 1·2에 이미 명시된 "표시되는 항목은 등록된 것만 해당" 원칙을 화면 구현에도 명확히 반영, 확정 2026-07-11).
4. **성능 — 스케줄 동기화 자체를 추적 대상으로 제한(확정, 2026-07-11)**: Nexon 스케줄러 Open API 호출(`GET /maplestory/v1/scheduler/character-state`)은 계정의 전체 캐릭터가 아니라 **추적 목록에 있는 캐릭터에 대해서만** 순차 호출한다. "캐릭터 관리" 피커가 보여줄 후보 목록(캐릭터명만 필요)은 `GET /maplestory/v1/character/list`(API 호출 1번, 가벼움)로 별도 조회하고 스케줄 동기화와는 분리한다.

**이유**:
- 실사용 계정에 캐릭터가 30명 이상 있는 경우가 실측으로 확인됨 — 대부분 방치된 부캐릭터라 사용자가 실제로 관심 있는 캐릭터는 소수다. 계정 전체를 대상으로 순차 스케줄 동기화를 하면([[ADR-008]] 순차 호출 정책과 맞물려) 캐릭터 수에 비례해 로딩 시간이 크게 늘어난다.
- 추적 대상만 동기화하면 불필요한 API 호출을 줄여 로딩 속도를 개선하고, 화면도 사용자가 실제로 보고 싶은 캐릭터만 간결하게 보여줄 수 있다.

**저장소**: `storage/`에 일간/주간 각각 독립된 키(`trackedCharacters:daily`/`trackedCharacters:weekly`)로 추적 캐릭터 `ocid` 배열을 저장한다. `null`(한 번도 설정 안 함)과 `[]`(사용자가 명시적으로 전부 해제함)은 값의 의미가 다르므로 구분해서 다룬다.

**트레이드오프**: 캐릭터 관리 피커의 후보 목록은 스케줄 동기화 없이 이름만 보여주므로, 그 화면만 봐서는 어떤 캐릭터가 실제로 진행 중인 콘텐츠가 있는지 미리 알 수 없다(추적 대상으로 고른 뒤에야 알 수 있음). 새로 캐릭터를 추적 대상에 추가하면 그 즉시 스케줄 재동기화가 트리거된다.

### ADR-013: 화면 구조를 "일간/주간 스케줄러"에서 "컨텐츠/보스 스케줄러"로 개편
**결정**: 리셋 주기(일간/주간) 기준으로 나뉘어 있던 최상위 화면 2개를 콘텐츠 종류(일반 콘텐츠/보스) 기준으로 재편한다.
1. **컨텐츠 스케줄러**: 일간 탭(기존 `daily_contents`) + 주간 탭(기존 `weekly_contents`)만 표시한다. **월간 탭은 두지 않는다** — Nexon 스케줄러 API에 월간 주기 일반 콘텐츠 자체가 없다.
2. **보스 스케줄러**: 주간 탭(`cycle: bossWeekly`) + 월간 탭(`cycle: bossMonthly`, 현재 검은마법사 1종)을 표시한다. **일간 탭은 두지 않는다** — [[ADR-007]]에서 확정한 대로 `bossDaily`(힐라 하드·핑크빈 카오스 등 일간으로 격하된 보스)는 계속 완전히 무시한다. 이번 개편은 이 정책을 재검토하지 않는다(확정, 2026-07-11).
3. 화면 내부의 일간/주간/월간 전환은 별도 라우트가 아니라 화면 내 탭(로컬 컴포넌트 상태)으로 구현한다 — 두 화면 모두 딥링크가 필요한 시나리오가 없다.

**보스 cycle 분리 표시 (버그 성격의 정정 포함)**: 기존 주간 스케줄러는 `bossContents`(cycle: weekly/monthly 전체)를 cycle 구분 없이 "주간 보스" 섹션 하나에 표시했다 — `weekly-bosses.json`에 이미 `monthly`(검은마법사) 참조 데이터가 있었음에도 화면에서 주간/월간을 나누지 않고 있었다. 개편 후 `boss-scheduler`는 cycle로 명확히 분리해 주간 탭/월간 탭 각각에 표시한다. `weeklyBossClearCount`/`weeklyBossClearLimitCount`(12마리 제한, [[ADR-007]])는 주간 탭에서만 의미 있는 값이다 — 월간 보스는 이 카운트에 포함되지 않는다.

**캐릭터 추적 목록 재구성 ([[ADR-012]] 원칙 유지, 키만 변경)**: [[ADR-012]]가 도입한 "화면별 독립 추적 목록" 원칙은 그대로 유지하되, 화면이 컨텐츠/보스 2개로 바뀌므로 `storage/`의 키도 `trackedCharacters:content`/`trackedCharacters:boss`로 바뀐다.
- **1회 마이그레이션(확정, 2026-07-11)**: `trackedCharacters:content` 키가 Preferences에 아직 한 번도 쓰인 적 없는 상태(키 자체가 없음, 즉 `Preferences.get`이 `null` 반환)로 조회되면 다음 규칙으로 마이그레이션한다.
  - `legacyDaily = get('trackedCharacters:daily')`, `legacyWeekly = get('trackedCharacters:weekly')` (둘 다 `string[] | null`)
  - `legacyDaily`와 `legacyWeekly`가 **둘 다 `null`**이면 마이그레이션할 데이터가 없는 것이므로 아무 키도 쓰지 않는다(`content`/`boss` 모두 계속 미설정 상태로 남음 — 다음 조회 때 다시 이 조건을 평가하지만 매번 결과가 같으므로 문제 없음)
  - 그 외의 경우 `content = dedupe([...(legacyDaily ?? []), ...(legacyWeekly ?? [])])`(ocid 기준 중복 제거), `boss = legacyWeekly`(그대로 복사, `null`이면 `null`)로 계산해 `trackedCharacters:content`/`trackedCharacters:boss`에 쓴다
  - 기존 주간 화면이 주간 퀘스트와 주간 보스를 같은 추적 목록으로 함께 보여주고 있었으므로, 분리 이후에도 사용자가 다시 선택할 필요 없이 같은 캐릭터가 양쪽에 그대로 보이게 하기 위함(사용자 확정, 2026-07-11). `content`가 daily·weekly 합집합인 이유: 컨텐츠 스케줄러 화면 하나가 일간/주간 탭을 함께 보여주므로, 둘 중 하나에만 추적 등록돼 있던 캐릭터도 새 화면에서 계속 보여야 정보 손실이 없다.
- 마이그레이션은 딱 1회만 수행한다 — `trackedCharacters:content`가 이미 존재하면(마이그레이션을 이미 했거나 사용자가 새로 설정했으면) 다시 덮어쓰지 않는다. 마이그레이션 이후 두 목록은 완전히 독립적으로 갈라진다(한쪽만 변경해도 다른 쪽에 영향 없음, [[ADR-012]] 원칙 그대로).
- 이전 키(`trackedCharacters:daily`/`trackedCharacters:weekly`)는 마이그레이션 후 정리(삭제)한다.

**이유**: 사용자가 이 앱을 켤 때 실제로 궁금해하는 축은 "오늘/이번 주에 뭘 해야 하나"가 아니라 "일반 콘텐츠 진행 상황"과 "보스 처치 상황"이 서로 다른 관심사라는 판단에 따른 제품 방향 확정(2026-07-11).

**트레이드오프**: 화면이 2개(daily/weekly)로 딱 나뉘어 있던 이전 구조 대비, 각 화면 내부에 탭 전환 UI가 추가되어 화면 하나의 복잡도가 늘어난다. 컨텐츠 스케줄러는 기존에 분리돼 있던 두 도메인(일간 콘텐츠, 주간 콘텐츠)을 한 화면에서 조합해야 한다.

### ADR-014: 보스 수익 계산기 — 완료 감지 시 파티원 수 자동 기록 및 캐릭터별 레이아웃
**결정**:
1. 스케줄 동기화로 새로 처치가 확인된 보스는 사용자가 화면에 들어와 파티원 수를 입력하기 전에도 **즉시 수익 기록을 자동 생성**한다. 기본 파티원 수는 같은 캐릭터(ocid)+보스+난이도 조합의 **가장 최근 기록값**을 그대로 이어 쓰고, 그 조합에 대한 과거 기록이 전혀 없으면 **1**을 기본값으로 쓴다.
2. 사용자가 파티원 수를 수정하면 해당 주(또는 달)의 기록만 갱신된다. 그 값은 이후 같은 캐릭터+보스+난이도 조합에서 새로 완료가 감지될 때 결정 1의 "가장 최근 기록값"이 되어 자동으로 이어진다 — 별도의 "기본값" 저장소나 테이블을 새로 두지 않고, 기존 `boss_profit_records` 테이블에서 `period_key` 조건 없이 `(ocid, boss, difficulty)`로 가장 최근 `recorded_at` 레코드 1건을 조회하는 쿼리로 구현한다(스키마 변경 없음, [[ADR-003]] 진행 상황 참고).
3. 화면 레이아웃을 상단 "이번 주" 총 수익(주간 보스, cycle: weekly만 합산) 1개 숫자 + 그 아래 캐릭터별 드롭다운(아코디언) 목록으로 재구성한다. 각 드롭다운 헤더에 그 캐릭터의 "이번 주" 총 수익을 표기하고, 펼치면 보스별 내역(파티원 수 입력 포함)을 보여준다. 월간 보스(검은마법사) 수익은 상단 합계에서 제외하고, 각 캐릭터 드롭다운 안에 "이번 달" 구분으로 별도 표시한다(기존 주간/월간 섹션 구분은 유지하되, 표시 단위를 화면 전체에서 캐릭터 단위로 옮긴다).

**이유**:
- 사용자가 매번 화면에 들어와 파티원 수를 입력해야만 수익 합계에 반영되는 기존 흐름은, 실제로는 "고정 파티로 거의 같은 인원과 도는" 사용 패턴과 맞지 않는다 — 매주 같은 숫자를 반복 입력해야 하는 불필요한 마찰이다(사용자 확인, 2026-07-11).
- 파티원 수를 캐릭터+보스+난이도 단위로 기억하는 이유: 보스마다 파티 구성이 다를 수 있다(예: 특정 보스는 6인 고정 파티, 다른 보스는 혼자 처치). 캐릭터 단위로 단일 값을 쓰면 보스마다 실제 인원이 다른 경우를 표현할 수 없다.
- 새 테이블 없이 기존 레코드에서 최근값을 조회하는 이유: 이미 캐릭터+보스+난이도+주차 단위로 기록이 쌓이고 있으므로([[ADR-003]]), 여기서 `period_key` 조건만 빼고 가장 최근 것을 고르면 "마지막으로 쓴 값"을 그대로 얻을 수 있다. 별도 "기본값 테이블"을 두면 사용자가 특정 주만 다르게 입력했을 때 "이게 새 기본값인지, 그 주만의 예외인지"를 구분하는 추가 로직이 필요해진다 — 항상 최신 기록을 기본값으로 쓰는 단순한 규칙이 이 문제를 자연히 피한다.
- 화면을 캐릭터별 드롭다운으로 재구성하는 이유: 추적 캐릭터가 여러 명이면([[ADR-012]]에서 실사용 계정에 캐릭터가 30명 이상인 경우도 확인됨) 기존의 "주간/월간 평면 목록"은 어떤 보스가 어느 캐릭터 것인지 한눈에 파악하기 어렵다. 상단에 전체 합계를 먼저 보여주고 캐릭터 단위로 접어두면, 여러 캐릭터를 동시에 추적할 때도 화면이 덜 길어진다.

**트레이드오프**:
- 사용자가 확인하기 전에 값이 자동으로 기록되므로, 실제 파티원 수와 기본값(특히 최초 감지 시의 1)이 다르면 사용자가 직접 확인하지 않는 한 부정확한 수익이 그대로 표시될 수 있다. "완료는 됐는데 수익이 0으로 잡히는" 기존 방식보다, 틀린 숫자를 사용자가 못 보고 지나칠 위험이 늘어난다는 뜻이다 — 자동 기본값이 적용된 행을 시각적으로 구분할지는 아래 "미확정" 참고.
- "가장 최근 기록값"을 기본값으로 쓰기 때문에, 파티원 수를 일시적으로만 바꿨다가 되돌리는 걸 깜빡하면 그 값이 계속 이어진다 — 사용자가 실제 파티 구성이 바뀔 때마다 정확히 반영해야 정확도가 유지된다.
- 캐릭터별 드롭다운 구조는 추적 캐릭터가 많을수록 전부 펼쳤을 때 화면이 길어질 수 있다 — 기본 접힘 상태로 시작하는 것으로 완화한다.

**미확정 — 추후 확정 필요**:
- 자동 기본값(특히 "처음 감지라 1로 기록됨")이 적용된 행을 사용자에게 시각적으로 강조할지(예: "확인 필요" 배지 등) — 위 트레이드오프 참고, UI 구현 시 결정
- 드롭다운 초기 펼침/접힘 상태(전부 접힘 vs 캐릭터 수가 적으면 자동 펼침 등) — UI 구현 시 결정

### ADR-015: 캐릭터 관리 피커 — 즐겨찾기 우선 정렬, 이미지 카드 UI, `character/basic` 병렬 조회
**결정**:
1. `CharacterTrackingPicker`("캐릭터 관리" 피커, [[ADR-012]])의 후보 목록은 **즐겨찾기(선택)한 캐릭터가 먼저, 그다음 즐겨찾기하지 않은 캐릭터**가 오도록 정렬한다. 두 그룹 각각의 내부는 **레벨 내림차순**이다(확정, 2026-07-12 — 와이어프레임 검토로 오름/내림차순 여부 확인 완료). 레벨이 같으면 기존 대표 캐릭터 선정 로직(`features/onboarding/representative-character.ts`에서 export한 `compareByName` — 한글>알파벳>숫자 그룹 후 로케일 비교)과 동일한 기준으로 2차 정렬한다. 즐겨찾기 상태를 토글하면 그 즉시 이 규칙대로 카드가 재배치된다.
2. 후보 목록을 체크박스 목록이 아니라 **캐릭터 이미지가 포함된 3열 카드형 그리드**로 표시한다(컬럼 수 확정, 2026-07-12 — 와이어프레임 검토). 각 카드는 캐릭터 이미지(얼굴 크롭, 아래 결정 4) + 이름 + 레벨만 보여준다 — **직업은 표시하지 않는다**(확정, 2026-07-12). 카드 전체가 토글 버튼이다(체크박스 엘리먼트 제거, `aria-pressed`로 선택 상태 노출). 선택 여부는 카드 **오른쪽 위 모서리의 별(★) 아이콘**만으로 표시한다(체크마크 배지 초안 → 배경 원 없는 별 아이콘으로 2차 정정, 2026-07-12) — 배경 동그라미·테두리 없이 아이콘 색만 바뀐다: 미선택은 `text-text-muted` 아웃라인(채움 없음), 선택(즐겨찾기)은 `fill-primary text-primary` 채움. 아바타가 아니라 **카드(버튼) 자체의 모서리**를 기준으로 위치를 잡아(`absolute` 부모를 아바타에서 카드로 이동) 상단 여백과 우측 여백이 항상 동일하게 맞도록 한다(카드마다 아바타 주변 여백이 달라 아바타 기준으로 잡으면 카드 모서리 대비 여백이 안 맞았던 문제 수정, 2026-07-12). 카드 자체의 선택 상태 스타일은 `AccountSelectionList`의 하이라이트 버튼 패턴을 재사용한다(선택: `border-primary bg-primary/15`, 미선택: `border-border hover:bg-primary/15`).
3. 캐릭터 이미지는 `GET /maplestory/v1/character/basic`(ocid 파라미터)의 `character_image` 필드를 사용한다. 피커가 열릴 때 후보 전체 캐릭터의 ocid로 이 API를 **병렬 호출**한다(순차 호출이 아님). [[ADR-007]]에서 이미 확정한 대로 이 앱은 서비스 단계 API 키(초당 500건)를 사용하므로 병렬 호출이 호출 한도 정책과 충돌하지 않는다. 기존 `syncSchedules`가 캐릭터별로 **순차** 호출하는 것은 이 결정과 무관하게 그대로 유지한다 — 그건 호출 한도 때문이 아니라 401/429 같은 전역 실패를 조기에 감지해 불필요한 후속 호출을 막기 위함([[ADR-008]])이라 이유 자체가 다르다.
4. `character_image`는 전신 이미지(기본 300×300)라 얼굴만 보이도록 프론트엔드에서 CSS로 확대·정렬해 자른다(고정 정사각형 프레임 + `overflow-hidden` + 절대 위치로 확대한 `<img>`). 크롭 좌표(원본 기준 x:115, y:120, 64×64px 정사각형)는 실제 캐릭터 이미지 1장을 알파 채널로 분석해 얻은 값이며, 실제 브라우저 렌더링으로 시각 검증까지 마쳤다(확정, 2026-07-12). **주의**: Tailwind preflight의 `img { max-width: 100% }`가 확대된 `<img>`의 너비를 다시 눌러버려 크롭이 찌그러지는 버그가 실제로 있었다 — `max-w-none` 유틸리티로 해결(구현 중 발견, 2026-07-12). 이 좌표는 이 캐릭터(고양이 귀·긴 헤어) 한 장 기준의 근사치라 다른 헤어스타일/포즈에서는 어긋날 수 있다(아래 트레이드오프 참고).
5. `character/basic` 응답의 `access_flag`가 문자열 `"false"`인 캐릭터는 카드 목록에서 **완전히 제외**한다. `registration_flag`/`complete_flag`와 동일하게 boolean이 아니라 문자열로 온다고 가정한다([[ADR-007]] 필드 주의사항과 동일 패턴). 이미 추적 목록(`trackedCharacters:*`)에 저장돼 있던 캐릭터라도 이 시점부터 후보 목록에는 안 뜬다 — 다만 이미 저장된 ocid 자체를 추적 목록에서 지우지는 않는다([[ADR-008]] 참조 무결성과 동일한 원칙).

**이유**:
- 즐겨찾기 우선 정렬: 레벨순만으로는 "자주 관리하는 캐릭터"와 "레벨이 우연히 높은 방치 캐릭터"를 구분 못 한다 — 사용자가 직접 별표 한 캐릭터를 맨 앞에 고정해 매번 스크롤하지 않도록 함(2026-07-12, 와이어프레임 피드백 반영).
- 이미지 카드: 이름만 나열된 체크박스 목록은 캐릭터가 많을수록 어떤 캐릭터인지 즉시 알아보기 어렵다 — 캐릭터 룩 이미지를 붙이면 스캔 속도가 빨라진다. 직업을 표시하지 않는 이유: 카드가 3열로 좁아 텍스트 공간이 부족하고, 얼굴 이미지 + 이름 + 레벨만으로도 캐릭터를 특정하기에 충분하다는 사용자 판단(2026-07-12).
- 별 배지: 체크마크는 "지금 트래킹 대상으로 저장될 것"이라는 사실 전달에는 맞지만, "내가 챙겨보는 캐릭터"라는 의미를 직관적으로 전달하지 못한다 — 즐겨찾기 UI 관용구(★)가 이 의미와 우선 정렬 동작 둘 다에 더 잘 맞는다(2026-07-12).
- `access_flag` 필터링: 이 필드는 정보 제공에 동의하지 않은 캐릭터(넥슨 정보공개 설정에서 비공개로 전환한 캐릭터 등)를 뜻하는 것으로 보인다 — 동의하지 않은 캐릭터는 이미지·정보가 정상적으로 오지 않을 수 있어 관리 화면에서 선택 대상으로 노출하지 않는 편이 안전하다.
- 병렬 호출: 캐릭터 수가 많은 계정에서 순차 호출 시 이미지 로딩까지 체감 지연이 크게 늘어난다 — 이미 서비스 단계 키로 확정된 호출 한도 여유([[ADR-007]])를 활용해 응답 시간을 단축한다.

**트레이드오프**:
- 피커를 열 때마다 후보 전체 캐릭터 수만큼 `character/basic` 호출이 발생한다(예: 30명이면 30건) — [[ADR-012]]가 "피커 후보 목록은 이름만 필요해 가벼운 `character/list` 1회 호출로 충분"이라고 정리했던 것과 달리, 이제 후보 목록 자체의 호출량이 늘어난다. 서비스 단계 한도(초당 500건)로는 문제 없다고 보지만, 그 전제(사용자가 실제로 서비스 단계 키를 쓰고 있다는 것)가 깨지면(예: 개발 단계 키를 쓰는 사용자) 초당 5건 제한에 걸릴 수 있다.
- 개별 `character/basic` 호출이 실패해도(네트워크 일시 오류 등) 그 캐릭터를 후보 목록에서 완전히 빼지 않고, 이름/레벨(`character/list` 응답)만으로 플레이스홀더 이미지("?" — `BossPortrait`의 빈 상태와 동일한 스타일)와 함께 보여준다 — 단, 401/403/429처럼 전역 실패로 볼 수 있는 에러는 `syncSchedules`와 동일하게 전체를 에러로 던진다(개별 캐릭터 폴백과 구분, `getCharacterPickerRoster`가 던진 에러를 화면단에서 어떻게 표시할지는 1차 구현에서 조용히 무시 — 아래 미확정 참고).
- 얼굴 크롭은 API가 제공하는 정밀 크롭이 아니라 프론트엔드 CSS 근사치라, 캐릭터 헤어스타일·포즈에 따라 완벽하게 얼굴만 나오지 않을 수 있다(예: 큰 헤어 장식이 프레임 밖으로 잘릴 수 있음). 검증에 쓴 캐릭터(고양이 귀·긴 은발) 기준으로는 결과가 자연스러웠다.

**미확정 — 추후 확정 필요**:
- Nexon `character/look` 이미지 URL이 크롭/리사이즈용 쿼리 파라미터(예: width/height/x/y 등)를 공식 지원하는지는 여전히 미확인이다 — 다만 CSS 근사 크롭이 실제 이미지로 시각 검증까지 끝나 충분히 쓸 만하다고 판단해, 이 확인은 더 이상 구현을 막는 항목이 아니다. 다양한 헤어스타일/포즈에서 크롭이 어긋나는 사례가 쌓이면 재검토.
- `character/basic` 호출에 `date` 쿼리 파라미터가 필요한지(다른 Nexon 캐릭터 API처럼 특정 날짜 스냅샷을 요구하는지) — 1차 구현은 생략했고(테스트는 모킹된 응답 기준), 실제 API로 호출해본 적은 없어 정말 최신 데이터가 오는지는 확인 필요.
- 전역 실패(401/403/429) 발생 시 피커에 에러 상태를 보여줄지, 조용히 빈 목록으로 둘지 — 1차 구현은 `ContentScreen`/`BossScreen` 모두 `.catch(() => {})`로 조용히 무시하고 있어(기존 `getRegisteredCharacters` 폴백 패턴을 그대로 답습), 에러 상태 UI는 아직 없다.
- 이 개선의 적용 범위는 `CharacterTrackingPicker`(캐릭터 관리 피커)로 한정했다 — 온보딩의 계정 선택 화면(`AccountSelectionList`), 단일 선택 드롭다운(`CharacterSelectDropdown`), 보스 수익 계산기의 캐릭터별 드롭다운([[ADR-014]])은 이번 변경 대상이 아니었다(확정, 2026-07-12). 확장 여부는 필요해지면 별도로 결정.

### ADR-016: 온보딩 시 전체 캐릭터 데이터 예열 + 캐시 우선 표시(Stale-While-Revalidate)
**배경**: [[ADR-015]] 구현 후 실사용 피드백 — 화면을 열 때마다(컨텐츠/보스 스케줄러, 캐릭터 관리 피커) 매번 Nexon API를 처음부터 다시 불러오다 보니 로딩 시간이 매번 체감된다. 화면에 "표시할 데이터가 하나도 없는 시간"이 아예 없도록 만드는 것이 목표.

**결정**:
1. **온보딩 완료 직전에 전체 캐릭터의 데이터를 한 번에 예열(prefetch)한다.** 계정이 확정되면(단일 계정 자동 확정 또는 다중 계정 중 선택) `character/list`로 이미 알고 있는 **계정 내 전체 캐릭터**(30명이든 60명이든 전부, 추적 대상 여부 무관)에 대해 캐릭터별로 독립적인 파이프라인을 병렬로 돌린다:
   - `character/basic` 조회 → 성공하면 즉시 로컬 캐시(아래 결정 3)에 기록
   - 그 응답의 `access_flag`가 `true`인 캐릭터에 한해서만 이어서 `scheduler/character-state`도 조회 → 성공하면 즉시 스케줄 캐시([[ADR-008]]에서 이미 쓰던 `storage/scheduler-cache`)에 기록
   - 캐릭터 하나의 실패(네트워크 오류 등)는 그 캐릭터만 캐시 없이 남기고 넘어간다 — 다른 캐릭터의 진행을 막지 않는다([[ADR-008]] 원칙과 동일)
   - **`Promise.all`로 한꺼번에 모아서 처리하지 않는다** — N개 캐릭터의 파이프라인이 각자 끝나는 대로 즉시 캐시에 반영하고 진행률을 갱신한다(`Promise.all`은 오직 "전체가 끝났는지"를 기다리는 용도로만 쓰고, 개별 결과 처리는 각 파이프라인이 스스로 한다). 예: 45번째 캐릭터가 완료되면 다른 캐릭터들의 완료 여부와 무관하게 그 즉시 캐시·진행률에 반영된다.
2. **온보딩 화면에 진행률 바를 도입한다.** 계정 확정 → 예열 파이프라인 실행 → 완료까지를 `OnboardingStatus`에 `'prefetching'` 상태로 추가하고, 이 동안 0~100% 진행률과 안내 문구를 보여준다. 진행률은 "완료된 단계 수 / 전체 예상 단계 수"로 계산한다 — 전체 단계 수는 처음엔 `2 × N`(모든 캐릭터가 정보+일정 둘 다 필요하다고 가정)으로 시작해서, 각 캐릭터의 `character/basic` 응답으로 `access_flag: false`임을 알게 될 때마다 1씩 줄인다(그 캐릭터는 일정 조회를 안 하므로). 완료된 단계는 성공/실패 관계없이(실패해도 "시도는 끝났다"는 의미로) 1씩 늘린다.
   - **문구는 아래처럼 정보/일정 조회를 하나로 합쳐서 보여준다** — 캐릭터별 파이프라인이 서로 엇갈려 진행되기 때문에(어떤 캐릭터는 이미 일정까지 끝났는데 다른 캐릭터는 아직 정보 조회 중인 상태가 실제로 함께 존재함) "전부 정보 조회 끝난 뒤 전부 일정 조회 시작" 같은 깔끔한 2단계 전환 시점이 기술적으로 없다. 그래서 "정보 조회 중 → 일정 조회 중"을 엄격한 순차 2단계로 보여주는 대신, 목록 조회(빠르고 항상 먼저 끝남) 다음에는 정보+일정을 하나의 메시지로 합쳐 보여준다:
     - "캐릭터 목록을 불러오는 중이에요"
     - "캐릭터 정보를 준비하고 있어요 (18/45)" (18=완료된 단계 수, 45=현재 추정 전체 단계 수)
   - **미확정**: 이 "합쳐서 표시" 방향이 맞는지, 아니면 사용자가 원래 제안한 대로 "목록 → 정보 → 일정" 엄격한 3단계 문구를 원하는지 확인 필요(엄격한 3단계로 하려면 "정보 조회가 전부 끝나야 일정 조회 문구로 전환"하는 별도 게이팅 로직이 필요해 약간의 지연이나 부정확한 표시가 생길 수 있음).
3. **`character/basic` 응답을 로컬에 캐시하는 새 저장소를 추가한다** — `storage/character-basic-cache.ts`(`storage/scheduler-cache.ts`와 동일한 구조): ocid별 key-value로 `@capacitor/preferences`에 저장(이름 `characterBasicCache:{ocid}`). 값은 `{ name, level, imageUrl, accessFlag, cachedAt }`. `@capacitor/preferences`는 기기 로컬에 영속 저장되어 앱을 껐다 켜도 남아있고, 새 API 응답이 오면 그대로 덮어써 최신화한다(이미 앱 전역에서 API 키·추적 목록·스케줄 캐시에 쓰고 있는 것과 동일한 저장소, SQLite 아님 — [[ADR-003]] "단순 key-value는 Preferences" 원칙 그대로 따름).
4. **캐시 우선 표시(Stale-While-Revalidate)를 컨텐츠/보스 스케줄러와 캐릭터 관리 피커에 적용한다.** 화면에 진입하면 캐시에 값이 있는 캐릭터는 **즉시** 그 값으로 렌더링하고(로딩 문구로 화면을 비우지 않음), 그 뒤에도 지금과 동일하게 매번 API를 새로 호출한다 — 다만 이번엔 재호출이다. 새 응답이 도착하면(성공이든 실패든) 그 즉시 화면 데이터를 새 값으로 갱신한다. 값이 실제로 같으면 리렌더링 자체는 일어나도 화면상 눈에 띄는 변화는 없다(React가 알아서 처리하므로 "변경 감지" 로직을 별도로 만들 필요 없음).
   - `content-scheduler`/`boss-scheduler` 스토어: `refresh()`가 `syncSchedules` 호출 *전에* 각 ocid의 `getCachedSchedulerState`를 먼저 읽어 `characters`를 채우고(있으면), 그 다음 API 응답으로 덮어쓴다. 화면(JSX)도 `status === 'loading'`이어도 `characters.length > 0`이면 목록을 계속 보여주도록 조건을 바꾼다(지금은 로딩 중엔 목록 자체를 안 그림 — 이 부분이 실제 버그 성격의 문제였음).
   - `CharacterTrackingPicker`가 열릴 때: `getCharacterPickerRoster()` 호출 *전에* 캐시된 `character-basic-cache` 값으로 먼저 그리드를 그리고, API 응답이 개별적으로 도착하는 대로(3에서 설명한 것과 동일하게 `Promise.all`로 뭉치지 않고) 각 캐릭터를 patch한다.
5. **TTL(유효기간)을 두지 않는다.** 화면에 들어올 때마다 항상 새로 재검증(refetch)하기 때문에 "캐시가 오래됐는지" 판단할 필요가 없다 — "가장 최근에 성공한 값"이 곧 캐시이고, 새 응답이 오면 그냥 덮어쓴다.

**이유**:
- 온보딩 시점 일괄 예열: 사용자가 이미 "설정 중"이라는 걸 인지하고 있어 기다림이 자연스러운 유일한 시점이다 — 이후 어떤 캐릭터를 추적 대상으로 고르든([[ADR-012]]) 이미 정보/일정이 캐시돼 있어 그 뒤로는 화면 진입이 항상 빠르다.
- `Promise.all`로 뭉치지 않는 이유: 45명 중 40명이 빨리 끝나도 가장 느린 1명 때문에 진행률·캐시 반영이 전부 늦어지는 걸 피하기 위함 — 실사용자 피드백(2026-07-12)으로 명시적으로 요청됨.
- 캐시 우선 표시: 이미 `storage/scheduler-cache`가 존재하고 매 성공마다 쓰고 있었는데 지금까지는 "실패했을 때 폴백"으로만 읽혔다(정독 시 확인) — 이 캐시를 "항상 먼저 보여주는 용도"로 확장하는 건 새 저장소를 만드는 게 아니라 읽는 시점만 앞당기는 것이라 구현 리스크가 낮다.
- TTL 없음: 어차피 매번 재검증하므로 "얼마나 오래됐는지"가 사용자 경험에 영향을 주지 않는다 — 굳이 만료 로직을 넣으면 복잡도만 늘어난다.

**정정 — 캐릭터명/레벨 캐싱 정책 ([[ADR-007]]·ARCHITECTURE.md "데이터 흐름" 2026-07-11 정정 부분)**: ~~캐릭터명·직업·레벨이 바뀔 수 있어 아예 캐싱하지 않고 매번 새로 조회한다~~ — "캐싱 안 함"의 진짜 의도는 "오래된 값을 못 믿고 계속 쓰는 것"을 막는 것이었지, "즉시 표시용으로 마지막 값을 잠깐 보여주는 것"까지 막는 게 아니었다. 이 ADR부터는 **항상 재검증(refetch)한다는 원칙은 그대로 유지**하되, 재검증 응답이 오기 전까지 화면이 비어있지 않도록 마지막 캐시를 잠깐 보여주는 것으로 정정한다. "매번 새로 조회"라는 핵심 정책 자체는 바뀌지 않는다.

**정정 — 스케줄 API 호출을 추적 대상으로만 제한 ([[ADR-012]] 결정 4)**: ~~스케줄 동기화는 추적 목록에 있는 캐릭터에 대해서만 호출한다~~ — 이 원칙은 "화면 방문할 때마다 반복되는 동기화"에는 그대로 유지된다(변경 없음). 다만 **온보딩 1회 예열 단계에서만** 예외적으로 전체 캐릭터(추적 대상 여부 무관, `access_flag: true`인 캐릭터 전체)의 스케줄까지 미리 받아 캐시를 채운다 — 반복 비용이 아니라 최초 1회 투자이기 때문에 [[ADR-012]]가 막으려던 "매번 캐릭터 수에 비례해 느려지는 문제"와 충돌하지 않는다.

**트레이드오프**:
- 온보딩 완료까지 걸리는 시간 자체는 늘어난다(캐릭터가 많을수록) — 사용자가 명시적으로 이 트레이드오프를 받아들임(2026-07-12, "30개가 됐든 60개가 됐든 병렬로 조회해").
- 예열 단계에서 실패한 캐릭터는 캐시가 비어있는 채로 온보딩이 끝난다 — 이 캐릭터들은 이후 화면 진입 시 캐시 우선 표시의 이점 없이 첫 조회처럼 동작한다(정상적인 폴백, 별도 재시도 로직 없음).
- 캐시 우선 표시로 인해 이론상 아주 짧게 "오래된 값 → 새 값"으로 화면이 바뀌는 순간이 생길 수 있다(예: 캐릭터가 방금 개명한 경우) — 실제로는 새 요청이 거의 항상 빠르게 끝나 체감되지 않을 가능성이 높다.

**확정(2026-07-12) — 위 두 미확정 항목 결론**:
- 진행률 문구는 "목록 조회 중 → 정보+일정 통합(x/y)" 2단계로 확정한다. 캐릭터별 파이프라인이 서로 엇갈려 진행되는 실제 동작과 맞지 않는 인위적인 3단계 게이팅은 만들지 않는다.
- 예열 파이프라인 도중의 401/429도 다른 개별 실패와 동일하게 그 캐릭터만 캐시 없이 넘어가고 온보딩 전체를 막지 않는다. 온보딩 자체의 키 유효성 검증은 이미 `character/list` 호출 시점에 끝나 있으므로 이 단계에서 별도의 전역 에러 처리를 두지 않는다.

### ADR-017: 보스 수익 계산기 캐시 확장 + 캐릭터 순서/선택 일관성 버그 수정 + 주간 화면 정리 (설계, 구현 전)

**배경**: [[ADR-016]]은 컨텐츠/보스 스케줄러·캐릭터 관리 피커에만 캐시 우선 표시(Stale-While-Revalidate)를 적용했다. `src/app/` 아래 화면 중 **보스 수익 계산기(`boss-profit`)만 아직 캐시가 없다.** 여기에 더해 실사용 중 다음 문제·개선 요청이 나와 함께 정리한다: (1) 컨텐츠/보스 스케줄러 드롭다운의 캐릭터 순서가 캐릭터 관리 화면과 다름, (2) 보스 스케줄러 진입 시 처음 보이는 캐릭터가 API 응답 도착 후 다른 캐릭터로 바뀌는 버그, (3) 보스 수익 계산기의 수익금 중복 표시, (4) 월간 보스(검은마법사)를 "주간" 화면에 함께 보여주는 것에 대한 재검토.

**결정**:

1. **보스 수익 계산기에도 스케줄 캐시 우선 표시 적용**: `features/boss-profit/store.ts`의 `refresh()`에 [[ADR-016]]과 동일한 패턴을 적용한다 — `syncSchedules(ocids)` 호출 전에 각 ocid의 `getCachedSchedulerState`를 먼저 읽어 캐시에 남아있는 완료 보스로 화면을 먼저 채우고, API 응답이 오면 덮어쓴다. SQLite `boss_profit_records`(파티원 수·수익 금액, [[ADR-014]] 자동 기록)는 캐시 단계에서도 **조회**는 한다 — 이미 저장된 기록이 있으면 그 값을 즉시 병합해 partySize/payoutMeso를 보여준다(단순 읽기라 안전하고, 이걸 생략하면 재검증 전까지 매번 "0 메소"·"파티원 수를 입력해주세요"가 잠깐 보였다 사라지는 깜빡임이 생긴다 — 2026-07-12 구현 중 확인된 문제). 다만 **기록(upsert)**은 캐시 단계에서 하지 않는다 — 자동 기록·최근 파티원 수 조회는 지금처럼 실제 API 재검증 이후에만 수행한다(낡은 캐시를 근거로 잘못된 파티원 수를 기록해버리는 걸 방지). 화면(`BossProfitScreen.tsx`)의 렌더링 조건도 `status === 'loaded'`가 아니라 `rows.length > 0` 기준으로 바꿔야 한다 — 그렇지 않으면 스토어가 캐시를 채워도 화면이 `status: 'loading'` 동안 무조건 "불러오는 중"만 보여줘 캐시 우선 표시가 무의미해진다(컨텐츠/보스 스케줄러 화면의 기존 패턴과 동일하게 맞춘다).

2. **`CharacterSelectDropdown`(컨텐츠/보스 스케줄러) 캐릭터 순서를 캐릭터 관리 피커와 통일**: 지금은 `characters` 스토어 배열 순서를 그대로 쓰는데, 이 순서가 캐시 단계(`trackedOcids` 저장 순서)와 동기화 단계(`syncSchedules`가 필터링하는 계정 전체 캐릭터 순서, 아래 결정 3의 원인 분석 참고)에서 서로 달라 일관성이 없다. `CharacterTrackingPicker`([[ADR-015]])와 동일한 규칙(레벨 내림차순, 동레벨이면 `compareByName`)으로 정렬한다. 이 드롭다운에 뜨는 캐릭터는 이미 추적 대상으로 선택된(피커에서 "즐겨찾기"로 표시되는) 캐릭터들뿐이므로, 피커의 "즐겨찾기 그룹 우선" 규칙은 자동으로 만족되고 실질적으로는 "레벨 내림차순 정렬"만 적용하면 두 화면의 순서가 일치한다. 정렬 키(`level`)는 `storage/character-basic-cache.ts`([[ADR-016]])에서 조회해 `ContentScreen`/`BossScreen`이 쓰는 캐릭터 뷰에 병합한다 — 아직 캐시가 없는 캐릭터(레벨 정보 없음)는 정렬 목록 맨 뒤로 보낸다.
   - 보스 수익 계산기의 캐릭터별 아코디언 순서는 이번 결정 범위에 포함하지 않는다(사용자가 명시적으로 요청한 범위는 "컨텐츠+보스 드롭다운"까지) — 필요하면 별도로 확인 후 진행.

3. **화면별 "마지막 선택 캐릭터" 캐시 도입 — 초기 캐릭터 뒤바뀜 버그 수정 (컨텐츠+보스 스케줄러 선제 적용)**:
   - **원인 분석**: `features/boss-scheduler/store.ts`의 `refresh()`는 캐시 단계에서 `cachedCharacters`를 `ocids.map(...)`으로 만들어 순서가 `trackedOcids`(저장된 배열 — 캐릭터 관리에서 체크한 순서)를 따른다. 반면 동기화 단계의 `syncSchedules(ocids)`(`features/schedule-sync/schedule-sync.ts:165`)는 `characters.filter(character => ocids.includes(character.ocid))`로 **계정 전체 캐릭터 목록**(`character/list` 응답 순서 — Nexon 쪽 기준이라 우리가 통제 불가)에서 추적 대상만 걸러낸 결과라 순서가 다르다. `selectedOcid`가 아직 `null`(사용자가 명시적으로 고른 적 없는 최초 진입)이면 `effectiveSelectedOcid`가 `characters[0]?.ocid`로 폴백하는데(`BossScreen.tsx:82-85`), 이 두 배열의 첫 번째 캐릭터가 다르면 캐시로 먼저 보여준 캐릭터와 동기화 완료 후 보이는 캐릭터가 화면에서 바뀌는 것처럼 보인다. **결정 2의 정렬 통일만으로는 완전히 해결되지 않는다** — 정렬 키(레벨)가 아직 캐시에 없는 캐릭터가 섞여 있으면 두 단계의 순서가 여전히 잠깐 어긋날 수 있어, 아래처럼 직접적인 고정 장치를 별도로 둔다.
   - **`ContentScreen.tsx`도 동일 코드 패턴 — 선제 적용으로 범위 확대**: 사용자가 실제로 보고한 증상은 보스 스케줄러 한정이지만, `features/content-scheduler/store.ts`의 `refresh()`와 `ContentScreen.tsx:42-45`도 캐시 단계 `ocids.map` 순서 vs 동기화 단계 `syncSchedules`의 계정 전체 순서, 동일한 `characters[0]` 폴백을 그대로 쓰고 있어 **같은 버그가 발생할 가능성이 있다** — 아직 증상이 보고되지 않은 건 우연히 순서가 맞아떨어졌을 뿐일 수 있다. 증상이 실제로 나타난 뒤 따로 고치는 대신, 이번 ADR에서 컨텐츠 스케줄러에도 동일한 수정을 함께 적용해 미리 막는다.
   - **결정**: 사용자가 `CharacterSelectDropdown`에서 마지막으로 선택한 ocid를 화면별로 `storage/`에 Preferences로 저장한다(신규 키 `lastSelectedCharacter:content`/`lastSelectedCharacter:boss` — `trackedCharacters:content`/`trackedCharacters:boss`와 동일한 화면 단위 분리 원칙, [[ADR-013]]). 각 화면 진입 시 이 값이 있고 현재 `characters` 목록에도 존재하면(추적 목록에서 빠졌을 수 있어 검증 필요) `characters[0]` 폴백보다 우선해서 `effectiveSelectedOcid`로 쓴다. 값이 없거나(최초 진입) 더 이상 유효하지 않으면 기존처럼 정렬된 목록의 첫 캐릭터로 폴백한다. 사용자가 드롭다운에서 캐릭터를 바꿀 때마다 그 화면의 캐시를 즉시 갱신한다.

4. **보스 수익 계산기 — 수익금 중복 표시 제거**: 캐릭터별 아코디언은 접힌 상태에서도 헤더에 `{캐릭터명} · {이번 주 합계} 메소`를 보여준다(`BossProfitScreen.tsx:165-167`). 펼치면 그 아래 "이번 주" 섹션에 `이번 주 합계 {총액} 메소`라는 문구가 다시 나온다(`BossProfitSection`, `BossProfitScreen.tsx:106-108`) — 결정 5(월간 제거)까지 반영되면 이 화면엔 "이번 주" 섹션 하나만 남아 헤더와 완전히 같은 숫자를 바로 아래에서 그대로 반복하게 된다. **섹션 타이틀의 "이번 주 합계 N 메소" 문구를 제거**하고, 펼친 영역에는 보스별 개별 행만 보여준다. 합계 표시는 아코디언 헤더에만 유지한다.

5. **보스 수익 계산기에서 월간 보스(검은마법사) 제거 — 화면명("주간 보스 수익 계산기")과 실제 표시 범위 일치**: 지금까지는 [[ADR-014]]에 따라 캐릭터별 아코디언 안에 "이번 달" 구분으로 월간 보스(검은마법사) 수익을 함께 보여주고 있었다(`PERIOD_ORDER`의 `'이번 달'`, `docs/PRD.md` "확정(2026-07-11) — 대상 보스" 참고). 화면 자체가 "주간 보스 수익 계산기"인데 월간 보스가 섞여 나오는 게 어색하다는 판단에 따라, 이 화면에서는 **월간 보스 관련 표시를 당장 제거**한다(세부 구현 — `rows` 생성 시 `cycle: 'monthly'`를 걸러낼지, 화면에서 `periodLabel === '이번 달'` 섹션만 안 그릴지는 착수 시 결정). 월간 보스 수익 기록 자체(SQLite `boss_profit_records`, [[ADR-014]] 자동 기록)는 계속 그대로 쌓인다 — 데이터 유실은 없고, 이 화면에서의 노출만 잠시 멈추는 것이다.

~~**미확정 — 월간 보스를 이후 어떻게 다시 보여줄지**: 별도 탭으로 분리하는 안을 검토 중이나, 월간 보스가 현재 검은마법사 1종뿐이라 탭 하나에 보스 하나만 있는 구성이 적절한지 확신이 없어 보류 — 별도 결정 필요.~~ → **확정(2026-07-14, [[ADR-023]])**: 주간/월간 탭으로 분리하기로 확정. "탭 하나에 보스 하나뿐"이라는 우려는, 월간 탭을 보스 나열이 아니라 "주차별 합계 + 월간 보스" 구성으로 만들어 해소했다 — 탭 콘텐츠가 보스 한 종에 국한되지 않는다.

6. **(2026-07-12 추가, 같은 날 재수정) "캐릭터 관리" 피커 — 여는 순간 잠깐 비어 보이는 문제 완화**: 실사용 확인 결과, 피커를 열 때마다 `getCharacterPickerRoster`가 매번 `GET /maplestory/v1/character/list`를 새로 호출하는 동안(이 목록 자체는 [[ADR-007]] 이후 개명·전직·레벨업 정확성을 위해 캐싱하지 않기로 확정됨, 2026-07-11 정정) 후보 목록이 잠깐 빈 상태로 보이는 문제가 있었다. `character-basic-cache`가 아무리 따뜻해도 이 네트워크 응답을 기다려야만 후보를 그릴 수 있었기 때문이다.
   - **최초 결정(추적 캐릭터 한정, 폐기)**: 처음에는 이미 추적 중인 캐릭터(`trackedCharacters:content`/`trackedCharacters:boss`)만 stub으로 먼저 보여주는 방식으로 구현했으나, 실사용 확인 결과 "추적하지 않는 나머지 캐릭터는 여전히 비었다가 나타난다"는 문제가 그대로 남아 있어 불충분했다(사용자 확인, 2026-07-12).
   - **재수정 결정**: `character-basic-cache`에 **지금까지 캐싱된 적 있는 모든 ocid**(추적 여부 무관 — 계정 소속 캐릭터 전원이 대상. 온보딩 예열([[ADR-016]])이 계정 전체 캐릭터를 최초 1회 채워두고, 이후 피커를 열 때마다의 `character/basic` 스트리밍 갱신이 계속 갱신한다)를 stub 대상으로 삼는다. 이를 위해 `storage/character-basic-cache.ts`에 별도 인덱스(`characterBasicCache:index`, Preferences에 ocid 배열로 저장)를 추가해 `setCachedCharacterBasic` 호출 시마다 갱신하고, `getAllCachedCharacterBasicOcids()`로 조회한다(개별 ocid별로 흩어진 캐시 키만으로는 "지금까지 캐싱된 ocid가 무엇인지" 자체를 알 방법이 없었기 때문). `getCharacterPickerRoster`는 이제 `trackedOcids`를 인자로 받지 않고, 이 인덱스로 `character/list` 응답 전에 접근 가능한(`accessFlag: true`) 모든 캐릭터를 stub 후보 목록으로 만들어 첫 `onUpdate`로 보여준다. `character/list` 응답이 도착하면 기존 흐름(계정 전체 캐릭터 기준 캐시 우선 표시 → `character/basic` 스트리밍 갱신)이 그대로 이어져 stub을 최신 값으로 교체한다.
   - **트레이드오프**: 이 인덱스에 아직 한 번도 안 잡힌 캐릭터(예: 온보딩 예열 도중 개별 실패해 캐시가 안 남은 경우)는 여전히 `character/list` 응답을 기다려야 그려진다. 또한 stub이 보여주는 이름/레벨/이미지는 캐시 시점 기준 근사치라, 그 사이 개명·전직·레벨업·캐릭터 삭제가 있었다면 잠깐 옛 정보가 보였다가 `character/list` 응답 도착 시 바로잡힌다(찰나의 근사치, ADR-007의 "character/list는 캐싱하지 않는다" 원칙 자체를 뒤집는 것은 아님 — 후보 목록의 진실 원천은 여전히 `character/list`이고, 이 인덱스는 그 응답을 기다리는 동안의 임시 근사치를 위한 것일 뿐이다).

**트레이드오프**:
- 결정 1(보스 수익 계산기 캐시)은 화면 진입 시 "이미 처치한 보스" 목록을 더 빨리 보여주지만, 캐시로 먼저 보이는 완료 보스에 아직 파티원 수/수익 기록이 없으면(처음 보는 조합) 그 항목은 재검증 이후에야 자동 기록되어 값이 채워진다 — 세부 로딩 UI는 구현 시 결정.
- 결정 3(마지막 선택 캐릭터)은 화면별로 독립된 키(`lastSelectedCharacter:content`/`lastSelectedCharacter:boss`)를 쓰므로, 사용자가 컨텐츠 스케줄러와 보스 스케줄러에서 서로 다른 캐릭터를 마지막으로 봤다면 두 화면은 각자 자기 화면의 마지막 선택값만 기억한다(의도된 동작).
- 결정 5로 월간 보스 수익이 이 화면에서 당분간 안 보이므로, 그동안 검은마법사를 잡은 사용자는 이 화면에서 수익을 확인할 방법이 일시적으로 없다(기록 자체는 유지되어 향후 노출 방법이 정해지면 즉시 다시 보인다).

### ADR-018: 보스 스케줄러 UI 개편 — 탭 pill 스타일 통일, 보스 리스트 개별 카드화, 보스 초상화 크롭 설정 도입 (설계, 구현 전)

**배경**: 보스 스케줄러 상단 컨트롤(캐릭터 드롭다운 + 주간/월간 탭)과 보스 목록 UI를 와이어프레임으로 검토했다([[ADR-013]] 화면 개편 이후 처음 있는 시각 디자인 리뷰). 두 가지 문제를 확인했다: (1) 활성 탭과 비활성 탭의 시각적 구분이 레테 테마(저채도 보라 팔레트)에서 약했고, 카운트 배지(`n/12`)가 탭과 시각적으로 분리돼 있어 어떤 탭에 속한 수치인지 즉시 읽히지 않았다. (2) 보스 목록이 하나의 큰 `<ul>` 카드 안에 체크 아이콘 + 24px 원형 초상화 + 텍스트로 나열되는 방식이라, 사용자가 직접 준비할 예정인 새 보스 일러스트(두상이 보이는 큰 그림)를 살릴 수 없었다.

**결정**:
1. **탭 스타일 — 활성 탭에 배지와 동일한 pill 재사용**: 주간/월간(또는 일간/주간) 탭 중 활성 탭은 `rounded-full bg-primary/15 text-primary` pill로, 비활성 탭은 기존처럼 `text-text-muted` 텍스트만으로 표시한다. 새 색상을 추가하지 않고 이미 배지(`n/12`)에 쓰이던 조합을 그대로 재사용한다. 카운트 배지가 있는 화면(보스 스케줄러 주간 탭)은 배지를 탭과 같은 행, 오른쪽 끝에 배치해 "주간 탭 = 이 수치"라는 관계를 공간으로 표현한다.
2. **드롭다운·탭을 별도 카드로 묶지 않는다**: 와이어프레임 초안은 캐릭터 드롭다운 + 탭 + 배지를 `rounded-[14px] bg-surface border` 카드 하나로 감싸는 안이었으나, 화면에 카드가 하나 더 늘어나는 것을 피하기 위해 기각한다(2026-07-13, 사용자 명시 반려). 각 요소는 지금처럼 배경 위에 직접 놓이고, 탭 스타일(결정 1)만 적용한다.
3. **이 컨벤션은 보스 스케줄러·컨텐츠 스케줄러 두 화면에 동일 적용**: 컨텐츠 스케줄러의 일간/주간 탭도 같은 UI 패턴(캐릭터 드롭다운 + 텍스트 탭 토글)을 쓰고 있으므로([[ADR-013]]), 탭 pill 스타일을 두 화면에 동일하게 적용한다. 컨텐츠 스케줄러는 카운트 배지가 없으므로 배지 관련 레이아웃은 해당 사항 없음.
4. **보스 목록 — 하나의 카드에서 개별 카드 나열로 전환**: 기존 `BossList`(`<ul className="rounded-[14px] bg-surface border border-border p-4 space-y-2">`로 보스 전체를 하나의 카드에 담던 방식)를 폐기하고, 보스 하나당 독립된 카드(`rounded-[14px] border border-border bg-surface`, 세로로 `space-y-2` 나열)로 바꾼다. 리스트를 감싸는 상위 카드는 두지 않는다(결정 2와 같은 이유 — 카드 중첩을 늘리지 않는다).
5. **완료 표시 — 왼쪽 체크 도형 → 오른쪽 완료 뱃지**: 기존 `StatusDot`(왼쪽 원형, 완료 시 `bg-secondary` 채움 + 체크 문자)을 제거하고, 완료된 보스는 카드 오른쪽 끝에 `완료` 텍스트 뱃지(`rounded-full bg-secondary text-bg`)를 표시한다. 미완료 보스는 오른쪽에 아무 것도 표시하지 않는다. 색상은 기존 `StatusDot`이 쓰던 `bg-secondary`를 그대로 재사용한다 — "완료" 상태를 나타내는 색은 이미 이 값으로 확정돼 있었으므로 새 색을 만들지 않는다.
6. **난이도 표시 — 텍스트 → 게임 UI 스타일 뱃지**: `{보스명} · {난이도}` 텍스트 조합을 폐기하고, 난이도(이지/노멀/하드/카오스/익스트림)를 실제 게임 내 난이도 뱃지와 동일한 시각 언어(글로시 캡슐형, 난이도별 고정 색)의 작은 뱃지로 표시한다. 색상은 사용자가 제공한 게임 UI 스크린샷에서 픽셀 색상을 추출해 근사했다(정확한 1px 단위 재현은 아님) — 값은 `docs/UI_GUIDE.md` "난이도 뱃지" 참고.
7. **보스 일러스트 — 두상이 보이는 큰 그림으로 교체, 같은 폴더·같은 파일명 컨벤션 유지**: 기존 `src/assets/bosses/{난이도접두사}_{portraitSlug}.png`(원형 초상화용, [[ADR-011]])를 사용자가 직접 새 일러스트(두상이 잘 보이는 큰 그림)로 **같은 경로·같은 파일명 그대로 교체**한다. 폴더 구조·조회 로직(`lib/boss-icons.getBossPortraitUrl`)은 변경하지 않는다 — 파일 내용만 바뀐다. **정정(2026-07-13, 실제 파일 반입 후)**: 아래 "정정 — 보스 초상화 에셋 컨벤션 변경" 참고, 이 항목의 "같은 파일명 그대로"는 실제로는 유지되지 않았다.
8. **카드 안 일러스트 배치 — 왼쪽에서 bleed, 페이드, 보스별 크롭 설정 도입**: 새 보스 카드는 일러스트를 카드 왼쪽에 배경처럼 깔고(카드 높이를 채우도록 확대), 오른쪽으로 갈수록 `mask-image` 그라디언트로 카드 배경색에 자연스럽게 녹아들게 한다(38%까지 선명, 76%부터 완전 투명). 블러 필터는 쓰지 않는다(2026-07-13, 사용자 확인 — 흐림 없이 선명하게 유지). 보스별로 일러스트의 인물 구도·여백이 달라 하나의 고정된 `background-size`/`background-position` 값으로는 모든 보스에서 두상이 잘 보이지 않는다 — 그래서 보스별로 값을 따로 지정할 수 있는 설정 테이블을 새로 둔다: `src/data/boss-portrait-crops.json`, `portraitSlug`를 키로 `{ size: string, position: string }`(CSS `background-size`/`background-position` 값 그대로) 매핑. **[[ADR-006]] 참고**: 이 파일은 보스 수치·목록 같은 게임 레퍼런스 데이터가 아니라 순수 UI 표시 파라미터이므로 ADR-006의 "게임 데이터는 AI가 임의로 채우지 않는다" 규칙 대상은 아니다. 다만 실제 값은 이미지 내용을 보고 판단해야 하므로 AI가 미리 채우지 않고, 사용자가 각 일러스트를 넣을 때마다 직접 값을 넣거나 조정한다(빈 객체로 시작). 매핑이 없는 `portraitSlug`는 기본값(`size: "cover"`, `position: "center"`)으로 폴백한다.
9. **크롭 설정은 새 보스 카드 전용 — 기존 `BossPortrait`/보스 수익 계산기/물욕 아이템 화면은 변경하지 않는다**: `BossPortrait`(원형, `object-cover`)와 그걸 쓰는 보스 수익 계산기(`h-10 w-10`)·물욕 아이템 드랍 화면은 이번 결정의 영향을 받지 않는다. `src/assets/bosses/`의 파일 내용이 바뀌므로 두 화면에 노출되는 이미지도 자동으로 새 일러스트로 바뀌지만(같은 파일 경로를 그대로 참조하므로 코드 변경 불필요), 원형 크롭은 여전히 브라우저 기본 `object-cover`(중앙 기준)로 처리되고 결정 8의 크롭 설정 테이블을 참조하지 않는다. 보스 수익 계산기 화면 디자인 자체를 이번 범위에서 다시 다듬지 않는다(사용자 명시, 2026-07-13) — 새 일러스트가 원형 중앙 크롭으로 보기 좋지 않은 보스가 나오면 별도로 재검토.

**이유**:
- 탭/카드 스타일은 이미 있는 배지·카드·라운딩 컨벤션을 재사용해 새 시각 언어를 최소화한다(`UI_GUIDE.md` "AI 슬롭 안티패턴" — 불필요한 신규 패턴 추가 금지 원칙과 일치).
- 카드 묶음을 기각한 이유는 화면당 카드 개수가 늘어나는 것 자체가 사용자가 원치 않는 방향이었기 때문(2026-07-13 명시 반려) — "카드가 적어야 좋다"는 원칙이 절대적으로 확정된 것은 아니고, 이번 화면에 한해 사용자가 선택한 방향이다.
- 크롭 설정을 분리 테이블로 둔 이유는 일러스트 각각의 인물 구도가 달라, 하드코딩된 단일 크롭 값으로는 두상이 잘리거나 배경만 보이는 보스가 다수 나올 수 있기 때문 — 실제로 카링(원화 제공)과 스우(원화 제공) 두 예시만으로도 서로 다른 `background-size`(카링 300%, 스우 190%)·`background-position`이 필요함을 확인했다.

**트레이드오프**:
- 보스 카드가 이제 일러스트 파일 존재 여부·크롭 설정 여부에 따라 시각적으로 들쭉날쭉할 수 있다(일러스트 없는 보스는 placeholder, 크롭 미설정 보스는 기본 `cover`/`center`로 어색하게 보일 수 있음) — 사용자가 순차적으로 일러스트·크롭 값을 채워나가는 동안의 과도기적 상태로 간주하고, 별도의 "미완성 보스 숨김" 같은 처리는 하지 않는다.
- 개별 카드 나열 방식은 기존 하나의 `<ul>` 카드보다 카드 사이 여백(`space-y-2`)만큼 화면을 더 차지한다 — 보스 개수가 많은 캐릭터는 스크롤이 길어질 수 있다(과거 [[ADR-014]] 아코디언 접힘 결정과 유사한 트레이드오프이나, 이번엔 접기 기능을 두지 않기로 함 — 필요해지면 별도 검토).
- 난이도 뱃지 색상은 게임 UI 스크린샷에서 추출한 근사값이라, 실제 게임 클라이언트와 픽셀 단위로 다를 수 있다.

**미확정 — 추후 확정 필요**:
- `src/data/boss-portrait-crops.json`에 매핑이 없는(아직 크롭을 안 맞춘) 보스가 실제로 어색하게 보이는 경우가 쌓이면, 기본 폴백 값(`cover`/`center`)을 재검토할 수 있다.
- 보스 수익 계산기·물욕 아이템 화면의 원형 크롭이 새 일러스트와 시각적으로 잘 안 맞는 사례가 나오면(예: 인물이 작게 나와 원형 안에서 거의 안 보임) 그 화면들의 크롭 방식도 재검토가 필요할 수 있다 — 이번 결정 범위 밖.

**정정 — 보스 초상화 에셋 컨벤션 변경 (2026-07-13, 실제 일러스트 반입 후)**: 결정 7이 가정했던 "같은 경로·같은 파일명 그대로 교체"는 실제로 지켜지지 않았다. 사용자가 `src/assets/bosses/`에 25개 신규 일러스트(`.webp`)를 한글 보스명 파일명으로 추가했고, 확인 결과 아래처럼 구조가 바뀌었다:
- **난이도별 분리 → 보스당 통합**: 기존에는 `{난이도접두사}_{portraitSlug}.png`로 같은 보스도 난이도마다 별도 파일이었으나(예: `easy_lucid.png`/`normal_lucid.png`/`hard_lucid.png`), 신규 일러스트는 난이도 구분 없이 보스 하나당 한 장이다. 파일명 컨벤션을 `{portraitSlug}.webp`(난이도 접두사 없음)로 변경한다. 이에 따라 `lib/boss-icons.getBossPortraitUrl`의 시그니처에서 `difficulty` 파라미터를 제거하고(`getBossPortraitUrl(portraitSlug: string | null): string | null`), 이 함수를 쓰는 `BossPortrait` 컴포넌트(원형, 보스 수익 계산기·물욕 아이템 화면)의 `difficulty` prop도 함께 제거한다 — 결정 9의 "원형 크롭 방식(`object-cover`)은 변경하지 않는다"는 그대로 유지되고, 조회 시그니처만 바뀐다(사용자 확인, 2026-07-13).
- **신규 일러스트가 추가되며 `portraitSlug`가 처음 생긴 보스 7종**: 기존에 `portraitSlug` 필드 자체가 없어(이미지가 없다는 뜻으로만 쓰임, `src/data/__tests__/boss-portraits.test.ts` 참고) 항상 플레이스홀더로 표시되던 자쿰·매그너스·파풀라투스·반반·피에르·블러디 퀸·벨룸에 이번에 실제 일러스트가 생겨 `weekly-bosses.json`에 `portraitSlug`를 신규 반영한다(**[[ADR-006]] CRITICAL 규칙에 따라 AI가 임의로 짓지 않고 사용자 확인 완료, 2026-07-13**): 자쿰→`zakum`, 매그너스→`magnus`, 파풀라투스→`papulatus`, 반반→`vonBon`, 피에르→`pierre`, 블러디 퀸→`crimsonQueen`, 벨룸→`vellum`. 벨로나는 미출시 보스라 이번에도 이미지 없이 `portraitSlug` 미부여 상태 유지.
- **기존 난이도별 png 40개 삭제**: 신규 통합 일러스트로 완전히 대체되므로 삭제한다(사용자 확인, 2026-07-13). `hard_kai.png`/`normal_kai.png`는 애초에 대체 대상이 아니지만(카이는 [[ADR-006]]으로 이미 목록에서 제외된 레거시 보스라 `portraitSlug` 참조가 없음), 어차피 미참조 상태였던 고아 파일이라 함께 정리한다.

### ADR-019: 파티 관리 — 캐릭터+보스+난이도 단위 상시 파티 인원 설정 도입 (설계, 구현 전)

**배경**: 보스 수익 계산기([[ADR-014]])는 파티원 수를 캐릭터+보스+난이도 **완료 기록**에 종속된 값으로만 다뤄왔다 — 아직 이번 주(달)에 처치하지 않은 보스는 파티/솔로 여부를 전혀 표현할 방법이 없었고, 파티원 수를 확인·수정하려면 보스 수익 계산기 화면에 들어가야만 했다. 사용자 요청(2026-07-13): 보스 스케줄러 화면에서 보스별로 파티 인원을 미리 설정해두고, 보스 카드에서 바로 솔로/파티 여부를 확인하고, 주간/월간 탭과 별개로 솔로/파티로도 필터링할 수 있게 해달라.

**결정**:
1. 신규 SQLite 테이블 `boss_party_settings`(캐릭터 `ocid` + `boss` + `difficulty`를 유니크 키로, `party_size` 저장)를 `storage/boss-profit`과 동일한 DB(`boss_profit`)에 추가한다. **완료(clear) 여부·주차와 무관한 상시 설정**이다 — 기존 `boss_profit_records`(주차별 완료 기록)와는 별도 테이블.
2. 설정 단위는 캐릭터+보스+**난이도**다(기존 `boss_profit_records`·`boss-crystal-prices.json`과 동일한 키 — 같은 보스도 난이도별로 파티 구성이 다를 수 있다는 [[ADR-014]]의 이유를 그대로 따름).
3. 아직 한 번도 설정하지 않은 캐릭터+보스+난이도는 **솔로(1인)로 취급**한다. 별도의 "미설정" 3번째 상태를 두지 않는다 — 보스 카드에는 파티 배지가 뜨지 않고, 필터의 "솔로"에 포함된다.
4. ~~보스 스케줄러의 보스 카드에서 직접 파티 인원을 설정한다(신규 화면을 만들지 않는다) — 카드를 탭하면 파티 인원 입력 모달이 뜬다.~~ → **정정(2026-07-13)**: 보스 카드 안 버튼(카드 전체 탭 → 카드 안 아이콘 버튼으로 1차 정정한 뒤에도) 방식은 폐기하고, 보스 스케줄러 화면 상단 "캐릭터 관리" 버튼 옆에 **"파티 관리" 버튼**을 신설한다. 탭하면 모달(`PartyManagementModal`)이 열리는데, 등록된 보스 전체를 목록으로 나열하는 대신 **보스 드롭다운 → 난이도 뱃지 선택 → 파티원 수 입력**의 3단 폼으로 한 번에 하나의 (보스, 난이도) 조합만 편집한다(**재정정, 2026-07-13** — 전체 목록 나열 방식에서 변경). **정정(2026-07-13)**: 보스 드롭다운의 옵션은 캐릭터가 게임 내 스케줄러에 등록해둔 보스가 아니라 **항상 전체 보스 목록**(`weekly-bosses.json`의 weekly+eventWeekly+monthly 전체)이다 — 아직 등록하지 않은 보스도 미리 파티 인원을 설정해둘 수 있어야 한다는 사용자 요청 반영. 다만 **정정(2026-07-13)**: 난이도 기본 선택만큼은 등록 여부를 다시 참고한다 — 선택한 보스가 캐릭터의 스케줄러에 등록돼있으면 그 등록된 난이도, 아니면 `boss-crystal-prices.json` 등록 순서상 첫 난이도를 기본 선택한다(보스 목록은 등록 여부와 무관하지만, 난이도 기본값은 실제 플레이 중인 난이도를 우선하는 것이 사용자에게 더 유용하다는 판단). 난이도는 보스 카드와 동일한 뱃지 디자인(`DifficultyBadge`)을 버튼으로 감싸 재사용하고, 그 보스가 지원하는 난이도 목록은 `boss-crystal-prices.json`에서 조회한다(가격 데이터가 아직 없는 보스는 설정 불가 안내로 대체). 선택 표시는 **정정(2026-07-13)**: 처음엔 `ring-2 ring-primary` 테두리를 썼으나, 테두리를 없애고 투명도 차이만으로 구분하도록 변경(선택: 불투명, 비선택: opacity-40) — 대비를 더 주기 위해 비선택 투명도를 기존보다 낮춤. 입력 범위(최대 파티원 수)도 새 게임 데이터가 아니라 기존 `boss-crystal-prices.json`의 `maxPartySize`/`defaultMaxPartySize`를 그대로 재사용한다([[ADR-006]]). **정정(2026-07-13)**: 파티원 수 입력은 자유 숫자 입력이 아니라 **-/+ 스테퍼**로 바꿨다 — 감소 버튼은 1에서, 증가 버튼은 해당 (보스,난이도)의 `maxPartySize`에서 비활성화되어 범위 밖 값을 아예 입력할 수 없다(사용자 요청 — 저장 시점 검증 대신 입력 자체를 범위 내로 제한). "저장" 버튼 클릭 시 현재 값을 그대로 저장하며, 1로 저장하면 솔로 취급과 동일해 배지가 사라진다. 보스 카드 자체에는 더 이상 설정 진입점이 없다 — 파티 배지(결정 5)만 표시한다.
5. 파티 인원이 2 이상으로 설정된 보스는 카드에 사람 여러 명 아이콘(`lucide-react` `Users`) + "n인" 배지를 표시한다(위치·스타일은 `docs/UI_GUIDE.md` "보스 카드" 참고).
6. 보스 스케줄러 화면에 주간/월간 탭 아래 "전체/솔로/파티" 서브 필터를 추가한다(기존 탭과 동일한 pill 스타일 재사용, [[ADR-018]]). 필터는 현재 활성 탭(주간 또는 월간) 안에서만 적용되고, 두 축(주간·월간 / 전체·솔로·파티)은 서로 독립적으로 상태를 유지한다.
7. **보스 수익 계산기와의 연동**: 스케줄 동기화로 새로 처치가 확인된 보스의 수익 기록을 자동 생성할 때, 파티원 수 기본값은 이제 `boss_party_settings`을 조회해 있으면 그 값, 없으면 1을 쓴다 — 기존 [[ADR-014]]의 "같은 캐릭터+보스+난이도의 가장 최근 완료 기록값 이어받기" 로직은 **완전히 폐기**하고 이 조회로 대체한다(사용자 확정, 2026-07-13). 보스 수익 계산기 화면에서 특정 주(달)만 다르게 입력하는 기존 주차별 override 기능은 그대로 유지한다 — `boss_party_settings`은 어디까지나 "새로 생성되는 기록의 기본값"일 뿐, 이미 기록된 과거 주차의 `party_size`는 그 기록 자체에 그대로 남아있고 override로 언제든 개별 수정 가능하다.
8. **과거 주차 기록·추이 차트와의 관계**: `boss_profit_records`는 주차(`period_key`)마다 그 시점의 `party_size`를 스냅샷으로 저장하므로, 이후 `boss_party_settings`을 바꾸거나 다시 override해도 이미 기록된 과거 주차의 값·수익은 소급 변경되지 않는다. 추후 구현될 월간 추이 차트(`docs/PRD.md` 핵심 기능 4, 1차 구현 범위 밖)는 각 주차 레코드에 저장된 실제 `party_size`를 그대로 읽으면 되므로, 파티원 수가 주마다 달라져도(예: 이번 달은 솔로, 다음 달부터 4인 고정 파티로 전환) 차트는 정확한 주차별 값을 표시한다 — `boss_party_settings`은 새 레코드 생성 시점의 기본값 조회에만 관여하고 이미 있는 레코드를 되돌아가 갱신하지 않기 때문이다.

**이유**:
- 완료 여부와 무관한 상시 설정으로 둔 이유: 사용자가 "이 보스는 원래 파티로 도는 보스"라는 사실을 실제로 처치하기 전부터 알고 있고 기록해두고 싶어한다 — 완료 기록에 종속되면 아직 안 잡은 보스는 파티/솔로를 표현할 방법이 없어 카드 배지·필터 요구사항을 충족 못 한다.
- 캐릭터+보스+난이도 단위로 둔 이유: 기존 [[ADR-014]]와 동일 — 같은 보스도 난이도마다 파티 구성이 다를 수 있다.
- 기존 ADR-014 "최근 기록값 이어받기" 로직을 폐기하는 이유: 두 로직이 공존하면 카드 배지(설정 없음=솔로)와 수익 계산 기본값(과거 기록에서 4인 이어받음)이 서로 다른 값을 가리켜 사용자가 혼란스럽다. 하나의 소스(`boss_party_settings`)로 단일화해 카드와 수익 계산이 항상 같은 값을 보게 한다(사용자 확정, 2026-07-13).
- 필터를 탭 아래 서브 pill 행으로 둔 이유: 기존 주간/월간 탭 pill 스타일을 그대로 재사용할 수 있고([[ADR-018]]), 두 축을 하나의 드롭다운으로 합치는 안보다 각 축의 상태 변경이 서로 완전히 독립적이라는 점이 더 명확하게 드러난다(사용자 선택, 2026-07-13 — 주간/월간 탭 UI(ADR-018)를 갈아엎지 않는 절충안).

### ADR-020: 일일퀘스트 카드 도입 — 보스 카드 패턴 재사용, `quest_state` 정규화, 지역 배경 매칭 (설계, 구현 전)

**배경**: 컨텐츠 스케줄러 일간 탭은 현재 `daily_contents`의 모든 항목을 동일한 리스트 카드(`{이름} · {now_count}/{max_count}` + 진행률 바)로 표시한다. 이 중 API가 `type: "quest"`로 내려주는 항목(아케인리버·그란디스 지역별로 반복되는 필드 퀘스트)은 `now_count`/`max_count`가 항상 0/0으로 와서 진행률 표기가 의미가 없고, 대신 `quest_state` 필드(0=시작 안 함, 1=진행중, 2=완료)로 실제 상태를 구분해야 한다(사용자 확인, 2026-07-14). `quest_state`는 지금까지 wire 타입(`NexonDailyContentWire`)에만 정의돼 있었고 `normalizeDailyContent`가 이 필드를 파싱하지 않고 버리고 있었다. 사용자는 보스 스케줄러의 보스 카드([[ADR-018]]) 패턴 — 이름 옆 상태 뱃지 + 카드 배경에 bleed로 깔리는 일러스트 — 을 일일퀘스트 항목에도 그대로 적용해달라고 요청했다(2026-07-14).

**실측 확인(2026-07-14, 사용자 제공 실제 API 응답)**: 초안 작성 시 추정했던 접두어·매칭 규칙이 실제 응답과 달라 아래 결정에 반영해 수정했다.
- 접두어는 `"[일일퀘스트] "`가 아니라 `"[일일 퀘스트] "`다(일일과 퀘스트 사이에 공백이 있음).
- 접두어를 제거한 퀘스트명은 지역명과 정확히 같지 않다 — 지역명 뒤에 조사·서술어가 붙은 전체 문장이다(예: `"[일일 퀘스트] 레헬른의 평온한 밤"` → 표시명 "레헬른의 평온한 밤", 지역명은 "레헬른"). 17개 전체 목록과 정확한 매핑은 결정 7 표 참고.

**결정**:
1. **표시 이름 접두어 제거**: `kind: 'quest'` 항목의 `content_name`은 `"[일일 퀘스트] "` 접두어를 포함해 온다. 표시할 땐 이 접두어만 제거하고 나머지 퀘스트명 전체(예: "레헬른의 평온한 밤")를 그대로 보여준다 — 지역 배경 매칭은 이 표시명 전체가 아니라 별도 규칙(결정 6)을 따른다.
2. **`quest_state` 정규화 도입**: `NexonDailyContentWire.type`을 `NexonWeeklyContentWire`와 동일하게 `'contents' | 'quest'`로 좁히고, `quest_state`(`string | null`)를 숫자(`0 | 1 | 2 | null`)로 파싱한다. 도메인 타입 `DailyContent`(`src/types/scheduler.ts`)에 `WeeklyContent`와 대칭으로 `kind: 'contents' | 'quest'`, `questState: 0 | 1 | 2 | null` 필드를 추가한다 — 지금까지 조용히 버려지던 필드를 실제로 쓰기 시작하는 변경.
3. **적용 범위는 `kind === 'quest'` 항목만**: `kind === 'contents'`인 기존 항목(몬스터파크 등)은 지금의 "이름 · now/max + 진행률 바" 표시를 그대로 유지한다. 이번 변경은 일간 탭 전체를 다시 디자인하는 게 아니라 퀘스트 항목에만 신규 카드 스타일을 추가하는 것이다.
4. **일일퀘스트 카드(`DailyQuestCard`) 신설**: `BossCard`와 동일한 카드 골격(80px 높이, `rounded-[14px]`, 레테 고정 배색, 왼쪽 38%까지 선명하고 76%부터 완전 투명해지는 일러스트 bleed·마스크)을 재사용하되 데이터 소스만 다르다 — 왼쪽엔 난이도 뱃지 없이 접두어를 제거한 퀘스트명만, 오른쪽엔 `quest_state` 3단 뱃지(결정 5). `ContentScreen.tsx`에서 `BossScreen.tsx`가 `BossCard`/`DifficultyBadge`를 export하는 것과 동일한 방식으로 `DailyQuestCard`를 export해 아래 디버그 프리뷰가 재사용한다.
5. **`quest_state` 뱃지 3종 (확정, 2026-07-14)**: 보스 카드는 미완료를 "빈 공간"으로 표현했지만, 이번엔 사용자가 "3가지로 구분해서 뱃지를 만들어달라"고 명시했으므로 3가지 모두 뱃지를 그린다.

**이유**: 보스 카드에서 이미 검증된 "카드에 일러스트를 bleed로 깔고 오른쪽에 상태 뱃지" 패턴을 그대로 재사용하면 새 시각 언어를 발명하지 않고도 일관된 스케줄러 UI를 유지할 수 있다. `quest_state`를 도메인 타입에 편입시키는 것도 이미 존재하지만 버려지던 데이터를 활용하는 것뿐이라 API 변경이 필요 없다.

**트레이드오프**: `kind === 'contents'`와 `kind === 'quest'`가 같은 일간 탭 리스트 안에서 서로 다른 두 카드 스타일로 섞여 보인다(일부는 진행률 바, 일부는 배경 일러스트+뱃지). 사용자가 "먼저"라고 표현했으므로 의도된 단계적 적용으로 보고, 나머지 `contents` 항목의 카드화는 이번 범위 밖으로 둔다.

### ADR-021: 주간 컨텐츠 카드 도입 — 4개 카테고리(에픽던전/지역 주간 퀘스트/무릉도장/길드)별 카드화 (설계, 구현 전)

**배경**: 컨텐츠 스케줄러 주간 탭은 지금까지 `weekly_contents`의 모든 항목을 하나의 카드 안에 `{이름} · {now}/{max}` 텍스트 행으로만 나열해왔다(일간 탭이 [[ADR-020]]으로 카드화되기 전과 동일한 구식 표시). 사용자가 일간 탭과 마�찬가지로 주간 탭도 카드 형태로 바꿔달라고 요청했다(2026-07-14). 실제 API 응답(`weekly_contents`)을 보면 항목들이 크게 4가지 성격으로 나뉜다: (1) "에픽 던전 : {던전명}" 형태의 3종 던전, (2) 아케인리버 6개 지역에서 주 1회 완료하는 6종 콘텐츠, (3) 무릉도장 1종, (4) "[길드]" 접두어가 붙은 3종(길드 관련 콘텐츠). 각각 표시 방식이 달라 카테고리별로 별도 카드를 설계한다.

**결정**:

1. **에픽 던전 카드(`EpicDungeonCard`)** — 대상: `content_name`이 `"에픽 던전 : "`로 시작하는 3종("하이마운틴"/"앵글러 컴퍼니"/"악몽선경").
   - 표시명은 접두어(`"에픽 던전 : "`)를 제거한 나머지(예: "하이마운틴").
   - 카드 왼쪽: "에픽 던전" 카테고리 뱃지 + 던전명(순서는 보스 카드의 `난이도 뱃지 → 이름`과 동일).
   - 카드 오른쪽: 완료 상태 뱃지 — `now_count`가 0이면 "시작 안함", 0보다 크면(사용자 확인상 최대값은 5) "완료". 두 상태 모두 [[ADR-020]]의 `QuestStateBadge`를 그대로 재사용한다(0→시작 안함, 2→완료 매핑, 1/진행중은 쓰지 않음 — 이 콘텐츠엔 중간 상태가 없다).
   - 배경: 던전별 전용 일러스트를 `src/assets/bosses/`에 추가로 넣어(보스 카드와 같은 폴더·같은 `import.meta.glob` 인프라 재사용) `lib/boss-icons.ts`의 `getBossPortraitUrl`/`getBossPortraitCrop`으로 그대로 조회한다(신규 lib 불필요, `src/data/boss-portrait-crops.json`에 슬러그 3개 추가). 매핑: 하이마운틴→`ancientGodMitra`(원본 `고대신미트라.webp`), 앵글러 컴퍼니→`senya`(원본 `세냐.png`), 악몽선경→`baekyeon`(원본 `백연.png`).
   - 카드 골격은 80px(일일퀘스트 카드와 동일, 진행률 바 없음).

2. **주간 지역 퀘스트 카드(`WeeklyRegionalContentCard`)** — 대상: 아케인리버 6개 지역에서 주 1회 완료하는 6종("에르다 스펙트럼"/"배고픈 무토"/"미드나잇 체이서"/"스피릿 세이비어"/"엔하임 디펜스"/"프로텍트 에스페라"). 이 6개는 [[ADR-020]]의 일일퀘스트 지역과 달리 `content_name`에 지역명이 전혀 포함되지 않으므로(예: "에르다 스펙트럼"은 텍스트상 "소멸의 여로"와 무관), 신규 참조 테이블 `src/data/weekly-regional-quests.json`(콘텐츠명 → [[ADR-020]]에서 이미 쓰는 지역 배경 슬러그)으로 정확히 1:1 매핑한다. 접두어 제거나 `startsWith` 매칭이 아니라 콘텐츠명 정확 일치다.
   - 매핑: 에르다 스펙트럼→`vanishingJourney`, 배고픈 무토→`chuchuIsland`, 미드나잇 체이서→`lachelein`, 스피릿 세이비어→`arcana`, 엔하임 디펜스→`morass`, 프로텍트 에스페라→`esfera`.

   - 카드 골격은 80px, 진행률 바 없음.

3. **무릉도장 카드(`MuLungDojoCard`)** — 배경·뱃지 없이 카드 껍데기(80px, 레테 고정 배경/보더)만 유지하고 이름만 표시한다(사용자 지시: "일단 배경과 뱃지 없이 카드만 표시해놔" — 추후 확장 여지 있음).

4. **길드 컨텐츠 카드(`GuildContentCard`)** — 대상: `"[길드] "` 접두어가 붙은 3종("주간 미션 포인트"/"지하 수로"/"플래그 레이스")을 하나의 카드로 묶는다.
   - `"[길드] "` 접두어는 삭제가 아니라 "길드" 카테고리 뱃지로 변환(에픽 던전과 동일한 취급).
   - 메인 타이틀은 "지하 수로"(카드 왼쪽: 길드 뱃지 + "지하 수로"). 오른쪽: 점수 뱃지 — "지하 수로"의 `now_count` 뒤에 "점"을 붙여 표시(예: "13416점"), 스타일은 [[ADR-020]] 몬스터파크 카드의 진행률 뱃지와 동일한 테마 토큰(`bg-primary/15 text-primary`).
   - 카드 하단에 "주간 미션 포인트: {now_count} · 플래그 레이스: {now_count}"를 한 줄로 이어 붙여 작은 글씨로 추가 표시(**정정, 2026-07-14** — 처음엔 두 줄로 나눴다가 사용자 지시로 한 줄로 통합).
   - 카드 높이는 몬스터파크 카드와 동일한 원칙(메인 행 80px 고정 + 늘어난 높이는 하단 확장 영역)으로 확대. **정정(2026-07-14)**: ~~두 줄 텍스트라 128px(`h-32`)까지 확대~~ → 한 줄로 통합하면서 몬스터파크와 동일한 112px(`h-28`)로 되돌림.
   - **폴백 조건**: "주간 미션 포인트"와 "플래그 레이스" 둘 다 `registration_flag`가 `false`(게임 내 스케줄러에 미등록)이면, 이 묶음 카드를 만들지 않고 등록된 길드 항목([길드] 접두어 포함 원문 그대로)을 기존 plain 카드와 완전히 동일한 형태로 표시한다 — 두 서브 항목을 추적하지 않는 사용자에게 억지로 진행률 문구를 보여주지 않기 위함.
   - 배경: `src/assets/bosses/아르카누스.png` → `arcanus.png`로 리네임, 결정 1과 동일하게 `boss-icons.ts`로 조회(`boss-portrait-crops.json`에 슬러그 추가).

   **정정(2026-07-14, 사용자 지시)**: 사용자가 실제 사용해보니 3종을 한 카드에 묶은 위 설계에 문제가 있다고 판단, 하나로 묶는 대신 다른 주간 카테고리와 동일하게 3종을 완전히 독립된 카드로 쪼갠다. `showGuildCard`/폴백 조건(둘 다 미등록이면 지하 수로만 plain 표시) 로직은 폐기하고, 3종 모두 각자의 등록 여부에 따라 개별적으로 표시한다(에픽 던전·지역 퀘스트와 동일한 취급).
   - **`[길드] 지하 수로` → `GuildUndergroundWaterwayCard`**: 배경은 그대로 `arcanus`(변경 없음, `boss-icons.ts`/`boss-portrait-crops.json` 유지). "길드" 뱃지 + "지하 수로" + 점수 뱃지("13416점")를 [[ADR-020]] `EpicDungeonCard`와 동일한 단일 행(80px) 레이아웃으로 표시 — 다른 두 종을 위해 확장했던 하단 텍스트 영역은 제거.
   - **`[길드] 주간 미션 포인트` → `GuildMissionPointsCard`**: 신규 배경 `hallOfHeroes`(원본 파일 `영웅의전당.webp`를 영문 파일명으로 리네임, `src/assets/maps/hallOfHeroes.webp`). `max_count`가 있는 콘텐츠(예: 10/10)라 [[ADR-020]] `MonsterParkCard`와 동일하게 "길드" 뱃지 + 이름 + `now/max` 뱃지 + 진행률 바 레이아웃(112px)을 재사용한다. 배경/크롭 조회는 지역 퀘스트와 같은 `daily-quest-backgrounds.ts`/`daily-quest-region-crops.json` 파이프라인을 쓴다(보스 파이프라인이 아님 — 지도 배경이므로).
   - **`[길드] 플래그 레이스` → `GuildFlagRaceCard`**: 신규 배경 `flagRace`(원본 파일 `플래그레이스.jpg`를 영문 파일명으로 리네임, `src/assets/maps/flagRace.jpg` — 포맷은 jpg 그대로 유지). `now_count`가 진행률로서 의미가 없는 콘텐츠라 `WeeklyRegionalContentCard`/`EpicDungeonCard`와 동일하게 "길드" 뱃지 + 이름 + 완료/시작 안함 뱃지(`now_count > 0` → 완료)만 표시하는 단일 행(80px) 레이아웃. 배경/크롭 조회는 미션 포인트와 동일하게 `daily-quest-backgrounds.ts`/`daily-quest-region-crops.json` 파이프라인을 쓴다.
   - **`daily-quest-backgrounds.ts` 확장자 처리 변경**: 지금까지 `import.meta.glob('../assets/maps/*.webp')`로 webp만 지원했는데, `flagRace.jpg`를 위해 `*.{webp,jpg}`로 확장한다(`boss-icons.ts`가 이미 `{webp,png}` 혼용을 지원하는 것과 동일한 전례). 조회 키도 고정 접미사(`${slug}.webp`) 방식에서 `boss-icons.ts`와 동일한 "확장자를 뗀 slug" 방식으로 바꿔 여러 확장자를 함께 지원한다.

5. **디버그 프리뷰 확장**: `/debug/quest-cards`(`DailyQuestCardPreview.tsx`)에 결정 1·4의 신규 배경 크롭 조정 카드를 추가한다. [[ADR-020]] 결정 9와 동일하게 방향키/줌 조정 UI를 이 파일 안에 그대로 복제한다(공용 추상화 신설 없음). **정정(2026-07-14)**: 길드는 이제 1개(`arcanus`, `boss-portrait-crops.json`)가 아니라 3개(`arcanus`/`hallOfHeroes`/`flagRace`) 조정 카드가 필요 — 뒤의 둘은 `daily-quest-region-crops.json` 기반. 최종 배경 크롭 값은 사용자가 이 프리뷰에서 눈으로 맞춘 뒤 확정([[ADR-006]]과 별개로, 크롭 좌표는 게임 수치 데이터가 아니라 순수 UI 값이라 AI가 초안을 채워도 무방하지만 시각 결과는 사용자가 최종 확인).

**이유**: 카테고리마다 실제 데이터 형태(단발성 완료/지역 반복/그룹 집계)가 달라 하나의 카드 컴포넌트로 억지로 통합하면 조건 분기가 과도해진다. 반면 [[ADR-020]]에서 이미 만든 배경·아이콘·뱃지 인프라(지역 슬러그 시스템, `QuestStateBadge`, 몬스터파크식 "메인 행 고정 + 하단 확장" 레이아웃)를 그대로 재사용하면 새 카테고리마다 새 시각 언어를 발명하지 않아도 된다.

**트레이드오프**: 길드 카드의 "둘 다 미등록이면 폴백" 조건은 `weekly_contents` 배열 전체(등록 여부 무관)를 미리 조회해야 판단할 수 있어, 기존에 `isRegistered`로만 필터링하던 렌더링 로직보다 조금 더 복잡해진다(주간 탭 렌더링에서 길드 3종만 특별 취급 후 나머지는 기존 필터링 로직 그대로 사용).

**정정 — 카드 내부 점수 배지·진행률 바를 `third`로(2026-07-14)**: 몬스터파크·길드 지하수로·길드 미션 포인트 카드의 점수/카운트 배지(`{nowCount}/{maxCount}` 또는 `{nowCount}점`)와 진행률 바 채움을 `primary`/흰색 대신 `third` 테마 토큰으로 바꿨다(사용자 지시 — "카드 내부 점수 뱃지, 프로그래스바를 third로"). 단, 이 카드들은 배경/보더/이름 텍스트가 [[ADR-018]]처럼 앱 테마와 무관하게 항상 레테(다크, `#1A1720`) 고정이라, `bg-third/15 text-third`(틴트+자기 색 텍스트) 패턴을 그대로 쓰면 렌 테마에서 `third`가 아주 밝을 때는 괜찮아도 레테 테마의 `third`(`#D8608F`, 중간 톤 핑크)로는 WCAG AA 미달(4.18:1)이 나왔다. 그래서 배지는 처음엔 `bg-black/20 text-third`(두 테마 모두 4.5:1 이상 — 렌 14.97:1, 레테 5.28:1)로 했다가, **재정정(2026-07-14, 사용자 지시)**: `bg-third/20 text-third`(자기 색 틴트)로 다시 바꿨다 — 계산해보면 렌은 8.18:1로 여전히 여유 있지만, **레테는 3.88:1로 WCAG AA(4.5:1) 미달**이다. 사용자에게 수치를 보고했고, 그대로 유지할지는 사용자 판단에 맡긴 상태(레테 테마의 이 배지 3곳은 현재 대비 기준 미달인 채로 남아있음). 진행률 바는 트랙을 기존 `bg-white/15` 그대로 두고 채움만 `bg-third`(솔리드라 텍스트 대비 문제가 없음)로 바꿨다. `third-text`([[ADR-009]] 정정 참고)는 애초에 앱의 `surface`(밝은/어두운 페이지 배경) 기준으로 계산한 값이라 이 "항상 어두운 카드" 맥락에는 안 맞아 여기서는 쓰지 않는다 — 카드 밖(앱 페이지 배경 위)에서 `third`를 텍스트로 쓸 일이 생기면 그때 `third-text`를 쓴다.

### ADR-022: Live Update(OTA) 도입 — Capgo 플러그인 + GitHub Releases 자체 호스팅 (구현 완료)

**배경**: 지금까지 JS/HTML/CSS 번들 변경(버그 픽스, `src/data/` 참조 데이터 보정 등)도 항상 앱스토어/플레이스토어 심사를 거쳐야 했다. 사용자가 스토어 심사 없이 웹 번들을 즉시 배포할 수 있는 Live Update(OTA) 도입을 요청했다(2026-07-14). WebView가 실행하는 JS/HTML/CSS는 Apple 3.3.2(인터프리터 코드) 조항상 심사 없이 교체 가능한 영역이라는 전제로 진행한다.

**결정**:
1. **플러그인**: `@capgo/capacitor-updater`(오픈소스, MIT, `^8.51.0` — Capacitor 8과 짝맞춘 버전)를 클라이언트 플러그인으로만 사용한다. Capgo의 매니지드 백엔드(계정 가입, 사용량 과금)는 쓰지 않는다 — 버전 체크·다운로드·스왑·크래시 감지 롤백 로직만 재사용하고 배포 인프라는 직접 소유한다. 이 플러그인은 기본값이 "Capgo 클라우드로 자동 체크·통계 전송"이라(`autoUpdate` 기본 `true`, `statsUrl` 기본 `https://plugin.capgo.app/stats`), `capacitor.config.ts`에서 `autoUpdate: 'off'`·`statsUrl: ''`로 명시적으로 꺼서 네이티브 쪽에서 Capgo 서버로 아무것도 나가지 않게 한다 — 버전 체크·다운로드·적용은 전부 아래 결정 3의 수동 흐름(JS에서 직접 fetch)으로만 이뤄진다.
2. **호스팅**: 처음에는 Cloudflare R2를 검토했으나, R2는 무료 한도 내에서도 결제 카드 등록이 필수라(활성화 시 소액 임시 승인이 걸릴 수 있음) 카드 등록 없이 쓸 수 있는 **GitHub Releases**(이미 쓰는 이 저장소 `jini520/maple-routine`, public repo라 인증 없이 다운로드 가능)로 변경했다(2026-07-14). 이 저장소에 고정 릴리스 태그 `live-update-latest` 하나를 만들어 두고, 배포할 때마다 그 릴리스에 자산을 덧붙이거나 덮어쓴다.
   - `latest.json`(버전 manifest, 배포마다 `--clobber`로 덮어씀): `{ version, url, checksum(sha256) }`
   - `{version}.zip`: 해당 버전의 `npm run build` 산출물(`dist/`) 압축본 — 버전마다 새 파일명이라 계속 누적되고, 롤백 시 과거 zip도 그대로 남아있다.
   이 프로젝트가 지켜온 "백엔드 없이 기기 로컬 저장소만"([[ADR-003]]) 원칙은 **사용자 데이터**에 대한 것이고, GitHub Releases는 순수 배포용 정적 자산(코드 배포 인프라)이라 이 원칙의 예외가 아니라 별개 층위로 취급한다.
3. **체크/적용 정책**: 앱 시작 시 `native/live-update`가 `latest.json`을 조회 → 로컬에 적용된 버전보다 최신이면 백그라운드에서 zip 다운로드 및 체크섬 검증 → 다음 재시작 시 적용(조용히, "업데이트 확인 중" UI 노출 없음). 적용 후 앱이 정상 기동하면 `notifyAppReady()`를 호출하며, 이 호출 전에 다시 크래시가 나면 플러그인이 자동으로 직전 정상 버전으로 롤백한다.
4. **무결성 검증**: `latest.json`의 `checksum` 필드를 다운로드한 zip과 대조해 불일치 시 적용을 거부하고 기존 번들을 유지한다.
5. **적용 범위**: JS/HTML/CSS 자산 전체(`src/data/`의 게임 레퍼런스 데이터 포함)가 OTA 대상이다. 다만 [[ADR-006]]에 따라 게임 데이터 값 자체의 변경은 여전히 사용자(도메인 전문가) 확인을 거쳐야 하며, OTA는 "심사 없이 반영하는 배포 경로"일 뿐 "AI가 임의로 데이터를 바꿔도 되는 권한"은 아니다. 네이티브 플러그인 추가/변경, 권한(Info.plist/AndroidManifest) 변경은 OTA 대상이 아니고 기존과 동일하게 스토어 심사를 거친다.
6. **아키텍처 배치**: 신규 `native/live-update.ts` 어댑터(기존 `native/notifications.ts`와 동일하게 플러그인을 감싸는 단일 파일 — 커스텀 네이티브 플러그인이 아니라 기존 오픈소스 플러그인 래핑이라 `hunting-timer/`류 폴더 분리는 하지 않음)가 플러그인 호출을 전담한다. `main.tsx` 부트스트랩에서 `notifyLiveUpdateReady()`(번들 실행 직후, 네트워크 요청 이전 — 타임아웃 안에 호출 안 하면 자동 롤백)와 `checkForLiveUpdate()`(백그라운드, 실패 시 조용히 무시)를 호출한다. `features/*`·`app/*`는 이 어댑터를 거치지 않고 플러그인을 직접 import하지 않는다([[ADR-003]]/[[ADR-005]]와 동일한 어댑터 원칙).
7. **배포 스크립트**: `scripts/publish-live-update.mjs`(빌드 → `dist/` zip → sha256 체크섬 계산 → `latest.json` 갱신 → `gh release upload`로 `live-update-latest` 릴리스에 업로드)를 신설한다. `src/` 바깥의 개발 도구로 취급하며 앱 번들에 포함되지 않는다. 이미 쓰는 `gh` CLI 인증만 있으면 되고, 별도 계정 가입·카드 등록이 필요 없다.
8. **채널/단계적 롤아웃은 범위 밖**: 지금은 "latest" 단일 채널만 존재하고, 퍼센트 기반 점진 배포나 dev/prod 채널 분리는 이번 범위에 포함하지 않는다.

**이유**: 스토어 심사 없이 버그 픽스·데이터 보정을 즉시 배포할 수 있게 되면 반영 지연(수일~수주)이 사라진다. Capgo 플러그인은 다운로드·스왑·크래시 롤백처럼 직접 구현 시 실수하기 쉬운 부분(적용한 새 번들이 깨져서 부팅 불가가 되는 상황의 안전망)을 검증된 오픈소스로 해결해주면서도, 매니지드 백엔드 없이 배포 인프라를 이미 쓰는 GitHub 저장소로 직접 소유해 이 프로젝트의 "불필요한 외부 의존 최소화" 기조를 유지한다. GitHub Releases는 이미 쓰는 계정 그대로라 신규 가입·결제 수단 등록이 전혀 없다는 것이 R2 대비 결정적 이유였다.

**트레이드오프**: 배포용 정적 파일 호스팅(GitHub Releases)이라는 이 프로젝트 최초의 "코드 배포 인프라" 의존성이 생긴다(사용자 데이터 백엔드는 아니지만, ADR-003이 전제해온 "인프라 제로" 이미지와는 다르다). GitHub Releases는 R2 같은 전용 CDN이 아니라 배포 자산 호스팅치고는 다운로드 속도·대역폭 정책이 R2보다 불리할 수 있다(트래픽이 늘어나면 재검토 대상). 배포 스크립트·체크섬 관리를 직접 해야 해서 Capgo 매니지드 서비스 대비 초기 셋업 비용이 더 든다. 조용한(silent) 업데이트 정책이라 사용자가 "왜 갑자기 동작이 바뀌었지"를 인지하기 어려울 수 있어, 추후 설정 화면에 "적용된 번들 버전 표시" 정도의 최소 가시성은 고려할 만하다(이번 범위 밖).
### ADR-023: 보스 수익 계산기 → "보스 수익" 페이지 개편 — 주간/월간 탭, 기간 네비게이터, 로컬 우선 캐싱 (설계, 구현 전)

**배경**: [[ADR-014]]가 확정한 캐릭터별 드롭다운 레이아웃은 유지하되, [[ADR-017]] 결정 5가 "화면명과 실제 표시 범위를 맞추기 위해" 당분간 제거했던 월간 보스 노출("미확정 — 월간 보스를 이후 어떻게 다시 보여줄지" 참고)을 이번에 정리한다. 이 화면이 "가계부"처럼 매주·매달 기록이 쌓이는 화면으로 발전해야 한다는 요구(2026-07-14, 사용자)에 따라, 실제 코드(`BossProfitScreen.tsx`)를 그대로 옮긴 아티팩트 와이어프레임으로 여러 라운드 검토를 거쳐 페이지 프레임 전체를 다시 설계했다.

**결정**:
1. 화면 제목을 "주간 보스 수익 계산기"에서 **"보스 수익"**으로 바꾼다 — 이 화면이 특정 주기에 묶이지 않고 주간·월간 보스 수익을 함께 다룬다는 걸 나타낸다.
2. 화면 상단에 **주간/월간 탭**을 추가한다([[ADR-017]] 결정 5의 "미확정"을 여기서 확정) — 기존 보스 스케줄러 탭과 동일한 pill 스타일을 재사용하고 새 스타일은 신설하지 않는다([[ADR-018]], `docs/UI_GUIDE.md` "탭 토글" 참고). 주간 탭은 cycle: weekly 보스만, 월간 탭은 cycle: monthly 보스(현재 검은마법사 1종)만 다룬다.
3. 탭 아래 **기간 네비게이터**(‹ 기간 라벨 ›)를 추가해 과거 기간을 탐색할 수 있게 한다.
   - 라벨은 가장 최근 두 기간까지만 상대 표현("이번 주"/"지난 주", "이번 달"/"지난 달")을 쓰고, 그 이전은 "N주 전"처럼 세는 표현 대신 **그 기간이 속한 달 안에서 몇 번째 주인지("OO월 N주차")** 또는 "OO년 O월"로 표기한다. 정확한 날짜/월은 그 아래 작은 글씨로 항상 함께 표시한다.
   - 한 주가 두 달에 걸치면(목요일 리셋 주가 월말에 걸리는 경우) **그 주가 시작하는 목요일이 속한 달** 기준으로 몇 주차인지 정한다.
   - 미래 방향 화살표는 최신 기간(오늘 기준 이번 주/이번 달)에서 비활성화한다 — 아직 일어나지 않은 미래 데이터는 없다.
4. **로컬 우선 캐싱 원칙**: 기간 이동(‹ ›)은 항상 로컬에 이미 저장된 기록(`boss_profit_records`)만 읽어 즉시 전환한다 — 매 이동마다 Nexon API를 다시 호출하지 않는다. 저장된 기록이 아직 없는 과거 기간으로 최초 이동했을 때만 예외적으로 로딩 스피너를 보여주며 그 시점을 스케줄러 API로 **자동으로 한 번** 재조회하고(버튼 트리거 없음), 결과를 로컬에 영구 저장해 다음부터는 다시 조회하지 않는다.
5. **월간 탭 구성 = 주차별 합계 + 월간 보스**: 월간 탭은 보스를 하나하나 나열하는 대신, 그 달 안에서 벌어들인 **주간 보스 수익을 주차별 합계**로 먼저 보여주고 그 아래에 그 달의 **월간 보스**(현재는 검은마법사 1종) 수익을 보스별 상세로 이어 붙인다. 캐릭터 합계는 이 둘을 더한 값이다. 이미 끝난 주는 확정 금액을, 지금 진행 중인 주는 "진행 중" 배지와 함께 잠정 합계를, 아직 시작하지 않은 주는 "예정"으로 흐리게 표시한다. 주차별 상세(그 주에 어떤 보스로 벌었는지)는 이 화면에서 다시 보여주지 않고, 사용자가 주간 탭에서 그 주로 이동해 확인한다.
6. **캐릭터 아코디언 헤더와 펼친 목록을 하나의 틀로 결합**한다 — 지금은 헤더도 그 아래 목록도 각각 독립된 카드라 간격이 다른 무관한 블록들과 똑같아 "펼친 결과"로 안 읽힌다. 헤더와 목록을 테두리 하나를 공유하는 단일 셸로 묶고, 그 경계는 얇은 구분선 하나로만 표시한다(접힌 상태의 헤더는 지금처럼 완결된 카드 그대로 둔다).
7. 캐릭터 아코디언 헤더의 표기를 **"{이름} · {합계} 메소"로 이어붙인 한 줄 텍스트**에서, **원형 아바타(이니셜) + 이름 + 우측 정렬 금액**으로 분리한다 — 보스 아이콘(사각형, `BossPortrait`)과 형태로 구분되는 캐릭터 표현(원형)을 도입한다.
8. 보스 행 리스트를 **행별 개별 카드에서 하나의 통합 리스트**(hairline 구분선)로 바꾸고, 각 행에서 이미 헤더에 있는 캐릭터명을 제거한다. 보스명·난이도는 자기 줄 전체를 쓰고(길어도 줄바꿈), 파티원 스테퍼·정산 금액은 그 아래 별도 줄에서 좌우로 벌어진다 — 이름 길이와 무관하게 금액이 항상 같은 위치에서 우측 정렬을 유지한다. "가격 미확정" 행도 같은 행 모양을 유지하고(기존 `bg-primary/15 text-primary` 배지를 금액 자리에 그대로 씀), 파티원 스테퍼는 비활성 처리한다.
9. 파티원 수 입력을 숫자 인풋에서 **−/+ 스테퍼**로 바꾼다(파티 관리 모달, [[ADR-019]]과 동일한 조작 방식으로 통일). 정산 금액은 **우측 정렬 + 등폭 숫자(tabular-nums)**로 표시해 세로로 훑으며 비교할 수 있게 한다.
10. 캐릭터 소계를 아코디언 헤더에만 두지 않고, **펼친 목록 최하단에 footer로도** 고정 표시한다.

**이유**:
- 화면 제목·탭 도입: [[ADR-017]]이 월간 보스를 임시로 뺐던 근본 원인(화면이 "주간"이라는 이름에 갇혀 있었던 것) 자체를 해소한다.
- 로컬 우선 캐싱: 기간을 넘길 때마다 API를 다시 부르면 매번 로딩이 걸려 사용성이 나빠진다는 문제 제기(2026-07-14, 사용자)에 따른 것 — 스케줄러 API가 `date`(YYYY-MM-DD) 쿼리 파라미터를 지원한다는 사실을 이번에 처음 확인했다(아래 "확인된 사실", [[ADR-007]]에도 추가). 이 파라미터로 과거 시점 스냅샷을 조회할 수 있어 "저장된 적 없는 과거 기간을 나중에 채워 넣는" 백필이 가능해진다.
- 헤더+본문 결합·아바타 분리·리스트 통합 등 아코디언 내부 리디자인은 실제 코드를 그대로 옮긴 와이어프레임으로 여러 라운드 검토를 거쳐 확정했다(2026-07-14) — "펼친 결과로 안 보인다", "한 줄에 다 밀어넣으면 보스 이름이 길 때 무리가 있다" 등 구체적 피드백을 반영.

**확인된 사실 (2026-07-14, `openapi.nexon.com` 공식 문서 확인)**: `GET /maplestory/v1/scheduler/character-state`는 문서화된 `date`(query, `YYYY-MM-DD`) 파라미터를 지원한다 — "조회 기준일, 미입력 시 오늘 날짜. 캐릭터가 해당 기준일에 접속하지 않은 경우, 응답 결과가 없을 수 있음"이 원문 설명이다. 지금까지 이 프로젝트의 어떤 문서에도 이 파라미터가 언급된 적이 없었고 `nexon/schedule/client.ts`도 `ocid`만 쿼리스트링에 넣는다 — [[ADR-007]]에 이 사실을 추가한다.

**추가 확인(2026-07-14, 사용자 실측)**: ~~실제 조회 가능한 최소 날짜는 `2026-06-25`다~~ → **정정(2026-07-14, 사용자 재실측)**: 실제 조회 가능한 최소 날짜는 `2026-07-01`이다 — 처음 확인했던 `2026-06-25`는 잘못된 확인이었다. 그 이전(2026-06-30 이하)으로 `date`를 지정해 조회하면 에러 상태코드(400 등)가 반환된다. 이 스케줄러 API 자체가 최근 신규 도입돼 그 이전 시점의 데이터가 아예 존재하지 않는 것으로 추정되는 **고정 하한선**이다(오늘 날짜를 기준으로 움직이는 롤링 윈도우가 아니다 — 위 #6·#7의 "최근 14일치" 제약과는 무관한 별개의 제약). 과거 기간 백필은 이 하한선 이전 기간에 대해서는 API를 호출하지 않고 즉시 "데이터 없음(체크 완료)"으로 처리해야 한다 — 재시도해도 영구히 실패하는 기간이므로 매 방문마다 헛되이 호출할 이유가 없다.

**트레이드오프**:
- 월간 탭의 "주차별 합계"는 보스 단위가 아니라 그 달 안의 weekly 기록을 주차로 묶어 다시 집계하는 새 로직이 필요하다 — 기존 `BossProfitRow`는 보스 1개 단위 row만 다뤄 이 집계를 아직 할 수 없다(아래 "미확정" 참고).
- 로컬 우선 캐싱은 "저장된 적 없는 과거 주"에 한해 최초 방문 시 스피너가 뜨는 지연을 감수해야 한다 — 다만 그 이후로는 다시 뜨지 않는다.
- `date` 파라미터로 과거 시점을 재조회해도 "그 날짜에 캐릭터가 접속하지 않았으면 응답 결과가 없을 수 있음"이라는 API 자체 제약은 그대로 남는다 — 완전한 임의 과거 조회를 보장하지 않는다.

**미확정 — 추후 확정 필요**:
- **과거 기록 편집 가능 여부**: 이번 라운드는 과거 기간을 읽기 전용(스테퍼 비활성)으로 잠정 설계했다 — 실제로 편집(파티원 수 정정)까지 허용할지는 결정하지 않았다. 허용한다면 반드시: (1) 과거 row의 `priceMeso`는 `buildBossProfitRow`(라이브 시세 조회)가 아니라 **그 기간에 저장된 기록에서 그대로 복원**해야 한다 — 현재 `mergeRecordsIntoRows`(`features/boss-profit/store.ts:71-88`)는 `partySize`/`payoutMeso`만 복원하고 `priceMeso`는 복원하지 않아, 그대로 과거 조회에 재사용하면 과거 기록이 조용히 오늘 시세로 재계산되는 위험이 있다. (2) 기록된 적 없는 과거 주를 `date` 파라미터 재조회로 채워 넣는 백필과, 이미 기록된 행을 고치는 편집은 별개 흐름으로 다뤄야 한다.
- **결정 시세의 시점별 이력화**: 결정 시세는 패치로 조정될 수 있어(사용자 확인, 2026-07-14), 과거 기록을 편집·백필하려면 `boss-crystal-prices.json`도 "언제부터 이 가격"인지 이력으로 관리해야 할 수 있다. 실제 과거 시세 값은 [[ADR-006]]에 따라 사용자(도메인 전문가) 확인이 필요 — 지금 이 수치를 임의로 채우지 않는다.
- **주차별 합계 집계 로직**: `getCurrentBossProfitPeriod`(`lib/boss-profit-period.ts`)는 "지금" 기준 기간만 계산한다 — 과거 임의 기간 계산, weekly 기록을 월 안에서 주차별로 묶어 합산하는 로직 둘 다 새로 필요하다.
- 캐릭터 아바타는 이번 라운드에서 이니셜 원형으로 잠정 설계했다 — `character/basic`의 `character_image`([[ADR-015]]에서 이미 다른 화면용으로 캐싱 중)를 재사용해 실제 캐릭터 이미지로 대체할지는 미정.
- 아코디언 펼침 상태가 캐릭터별로 완전히 독립적이라는 점, 파티원 수가 캐릭터마다 완전히 분리 저장된다는 점(같은 보스를 여러 캐릭터가 실제로 같은 파티로 돌아도 공유 안 됨), 소계 계산 로직이 화면 전체 합계와 캐릭터별 소계 두 군데 중복돼 있다는 점은 이번 라운드 조사 중 함께 발견했으나 범위 밖으로 남겨둔다.
