# 스플래시 다크모드 색 문제 트러블슈팅 — 결국 네이티브 액티비티로 (2026-07-16)

"다크모드 스플래시 배경이 어둡게 보인다"는 한 줄에서 시작해 [[ADR-025]]의 색 튜닝 한계 확인 → JS 레이어로 옮기려던 두 번의 실패([[ADR-028]]) → 착각으로 끝날 뻔한 "문제 없음" 결론 → 결국 Capacitor를 안 거치는 별도 네이티브 액티비티로 해결([[ADR-029]])까지의 전체 과정. **두 번의 실패와 한 번의 오판을 거쳐야 진짜 원인(WebView는 그려지는 시점 자체가 너무 늦다)에 도달했다**는 게 이 문서의 핵심이다.

검증 기기: Xiaomi 23129RAA4G(HyperOS, adb-tls 무선), iPhone 17 Pro(시뮬레이터), jinni iPhone(iPhone 17, 실기기).

## 1. 색 자체는 이미 한계였다

**증상**: MIUI/HyperOS 다크모드에서 스플래시 배경(`#D06100`)이 "칙칙하고 너무 어둡다."

**시도**: hue를 15°/22°/28°(브랜드 원색)/35°로 바꿔가며 실기기 빌드·설치·`adb shell screencap` 픽셀 샘플링으로 각 hue에서 force-dark를 통과하는 최댓값을 찾음.

**결과**: 빨간 쪽(15°, 22°)은 오히려 안전 임계값이 더 낮아지고, 노란 쪽(35°)은 같은 밝기를 유지하려면 더 낮은 V가 필요했다. **브랜드 원색(28°)이 이미 안전-밝기 곡선의 정점**이라, `#D06100`은 hue를 어느 쪽으로 틀어도 더 밝게 못 만든다. 자세한 데이터는 [[project_miui_forcedark_splash]] 메모리 참고.

**여기서 멈췄으면 좋았을 실수**: "색은 한계, 그럼 노출 시간이라도 줄이자"로 방향을 튼 것. 이 판단 자체는 합리적이었지만, 뒤에 나올 두 번의 실패의 시작점이 됐다.

## 2. "그럼 브랜드를 WebView로 옮기자" — 두 번의 실패

**아이디어**([[ADR-028]]): force-dark는 **네이티브 solid color 속성만** 건드리고 **WebView가 그리는 내용은 안 건드린다**(`capacitor.config`의 WebView 배경 `#FB8101`이 다크모드에서도 그대로 렌더링되는 걸로 확인됨). 그러면 네이티브 스플래시는 무채색+아이콘 없음으로 최대한 짧게 노출하고, 진짜 브랜드(로고+워드마크+주황)는 `index.html`의 HTML 레이어가 담당하면 되지 않을까?

### 2-1. 1차: OS 기본 타이밍에 맡기기

`MainActivity`의 커스텀 `setKeepOnScreenCondition`(고정 1초 유지)을 제거하고 안드로이드 SplashScreen API의 기본 동작(첫 프레임이 그려지면 자동 해제)에 맡김.

**검증 방법**: `adb shell screenrecord`로 콜드스타트를 녹화 → `ffmpeg -vsync 0`으로 전체 프레임 추출 → 각 프레임 픽셀을 스캔해 주황(`r > g+15 and r > b+30`)이 한 프레임이라도 있는지 확인.

**결과**: 112프레임(전체 구간) 중 **단 한 프레임도 주황이 없었다**. 무채색 회색(`28,27,32`)이 3.2초간 그대로 유지되다 앱 콘텐츠로 직접 크로스페이드. `ActivityTaskManager: Displayed +3s1ms` 로그로 재확인 — 이전(고정 1초, 그래도 브랜드는 보임)보다 명백한 회귀였다.

### 2-2. 2차: JS→네이티브 신호 방식

"고정 타이머가 아니라 WebView가 실제로 페인트된 시점에 신호를 주면 되지 않을까"로 재시도. `BootSplashPlugin`(커스텀 Capacitor 플러그인, `ready()` 메서드로 `AtomicBoolean` 플래그를 켬) + `index.html`의 인라인 스크립트(rAF 2회로 페인트 완료를 기다린 뒤 `BootSplash.ready()` 호출) + `MainActivity`의 `setKeepOnScreenCondition`이 그 플래그를 참조(상한 5초).

**검증**: `logcat`으로 `ready() called from JS`가 정확히 찍힘 — 신호 자체는 정상 동작. 그런데도 화면 녹화(113프레임) 재검증 결과 **여전히 주황이 한 프레임도 없었다**.

**진짜 원인**: logcat 타임스탬프 대조 결과, `ready()` 신호 자체가 스플래시 창이 뜨고 **약 1.8초 뒤에야** 발화했다 — Capacitor 브릿지 초기화 + 8개 플러그인 등록 + WebView 로딩이 그만큼 걸린다(이 앱 JS 코드가 실행되기도 전 구간). 신호 이후에도 실제 창이 그려지기까지 다시 ~1.2초가 더 걸렸다(`Displayed +3s127ms`). **HTML 레이어가 그려질 수 있는 시점 자체가 이미 콜드스타트 후반부**라, 네이티브 해제 조건을 아무리 정교하게 바꿔도 그 전 구간을 앞당길 수 없었다.

**교훈**: "WebView 콘텐츠는 force-dark 면역"이라는 전제는 맞았지만, **그 면역성을 살릴 수 있는 시점 자체가 오지 않는다**는 걸 놓쳤다. 신호 메커니즘을 아무리 정교하게 만들어도 병목(브릿지+플러그인 초기화)은 그 메커니즘보다 먼저 일어난다.

## 3. "3초 무브랜드"는 테스트 아티팩트였다 — 그리고 이 결론도 틀렸다

**재확인**: 2-1/2-2에서 측정한 "3초"는 전부 `adb install -r` 직후 `am force-stop` 후 곧바로 재실행하는 방식이었다(ART가 전혀 컴파일 안 된 최악의 콜드스타트). 실사용 패턴(`am kill`로 재현 — 앱이 이미 실행된 적 있는 상태에서 OS가 백그라운드 프로세스를 회수 후 재실행)으로 다시 녹화하니 전혀 다른 결과가 나왔다: 검정 전환(~1.3초) 이후 주황 네이티브 스플래시(`206,97,0` ≈ `#D06100`)가 약 1초 가까이 안정적으로 보이고 매끄럽게 콘텐츠로 넘어갔다.

**여기서 내린 결론(틀림)**: "타이밍은 실사용에서 문제없다 → 스플래시는 색도 타이밍도 이미 문제없다 → 추가 조치 불필요."

**사용자 정정**: "다크모드 때문에 색이 어두워지는 문제가 있어서 시킨 작업인데, 그렇게 결론을 내면 어떡해?" — 노출 **시간**이 괜찮다는 것과 노출되는 **색 자체**가 여전히 `#D06100`(진짜 브랜드 `#FB8101`이 아님)이라는 건 별개의 문제였다. "타이밍 문제 없음"을 "문제 해결됨"으로 뭉뚱그린 게 오판이었다.

**교훈**: 원래 문제(색이 어둡다)가 무엇이었는지 계속 추적해야 한다. 곁가지 조사(타이밍)에서 좋은 결과가 나왔다고 원래 질문에 답한 것으로 착각하면 안 된다.

## 4. 진짜 해결책 — Capacitor를 아예 안 거치는 네이티브 액티비티

**재정의된 요구사항**: 사용자는 타협안(무채색 다크 리디자인 후보 2종, `#0C080F`/`#2B1B0E`, 실기기 스크린샷까지 준비)도 거절 — "기존 밝은 오렌지색을 쓰고 싶은 거야."

**핵심 통찰**: 2번의 실패 원인은 "브랜드를 WebView로 옮기자"가 아니라 "**WebView를 거치자**"였다. force-dark가 비트맵 드로어블은 안 건드린다는 사실(아이콘 PNG가 안 깎이는 것과 동일 원리) 자체는 처음부터 맞았다. 이걸 WebView가 아니라 **Capacitor를 전혀 초기화하지 않는 별도의 초경량 네이티브 `Activity`**로 살리면, 브릿지/플러그인 초기화 지연이 원천적으로 없다.

**구현**([[ADR-029]]): `SplashActivity`(플레인 `Activity`)를 앱의 진짜 런처로 두고, OS가 강제하는 극히 짧은 무채색 첫 프레임 직후 `@drawable/splash`(전체 브랜드 이미지 비트맵)를 그린 뒤 1.2초 후 `MainActivity`로 넘긴다.

**검증**: 화면 녹화 → 프레임 스캔 결과 `(252,128,2)` ≈ `#FC8002`가 약 0.9초간 노출 — 원본 브랜드 `#FB8101`(251,129,1)과 채널당 오차 1(영상 압축 수준)로 사실상 동일. **2회 재현 확인.** 이 프로젝트 스플래시 작업 전체(ADR-025~029)에서 다크모드에 진짜 브랜드 오렌지가 뜬 첫 사례.

## 5. 구현 직후 발견된 버그 두 개

### 5-1. 이미지가 가로로 눌려 보임

**원인**: `android:windowBackground=@drawable/splash`(비트맵을 윈도우 배경으로 직접 지정)는 `Gravity.FILL`로 화면 실제 비율과 무관하게 드로어블을 늘려 채운다. 정사각형(2732×2732) 원본이 실기기(1080×2400, 훨씬 좁고 긴 비율)에서 늘어나 보였다.

**해결**: `activity_splash.xml`(전체화면 `ImageView`, `android:scaleType="centerCrop"`)로 교체. `windowBackground`는 `ImageView`가 인플레이트되기 전 찰나에만 보이는 폴백이라 단색(`@color/splash_background`)으로 낮췄다.

### 5-2. 상태바·제스처 바 색이 스플래시와 안 맞음

**원인**: `SplashActivity`는 시스템 바 처리를 전혀 안 해서 상단 상태바·하단 제스처 영역이 기본 검정으로 남았다.

**해결**: `MainActivity`의 `SystemBarsPlugin`(edge-to-edge + 투명 바)과 동일한 원리를 브릿지 없는 이 액티비티에 맞게 직접 구현 — `WindowCompat.setDecorFitsSystemWindows(false)` + 상태바/내비게이션 바 `Color.TRANSPARENT` + `setStatusBarContrastEnforced(false)`.

**곁가지 — MIUI가 아이콘 색을 무조건 강제**: `setAppearanceLightStatusBars(false)`(흰 아이콘 의도)를 걸었는데도 아이콘이 계속 어둡게 나옴. `true`/`false`를 바꿔 실기기로 재확인했더니 **결과가 완전히 동일**했다(스크린샷 두 장이 픽셀 단위로 같음) — MIUI/HyperOS가 배경 밝기를 자체 판단해 이 API를 완전히 무시하는 것으로 결론. 이번 세션 내내 본 force-dark류 OS 레벨 오버라이드와 같은 패턴이라 더 파고들지 않고, 의미상 올바른 값(`false`)만 코드에 남겼다.

## 6. 테스트 방법론 교훈

### 6-1. `adb screencap`은 빠른 전환을 못 잡는다

무선 adb로 `screencap` 1회 호출에 ~1.5초가 걸려, `sleep(N) + screencap`을 반복해도 실제 타이밍과 스크립트가 기대한 타이밍이 어긋난다(첫 테스트에서 "1.2초"라고 찍은 프레임이 실제로는 훨씬 늦은 시점이었음). **`adb shell screenrecord` + `ffmpeg -vsync 0`으로 전체 프레임을 추출해 픽셀 스캔**하는 방식으로 바꾸고서야 신뢰할 수 있는 타임라인을 얻었다.

### 6-2. fresh-install 콜드스타트 ≠ 실사용

`adb install -r` 직후 `am force-stop` → 재실행은 ART가 전혀 컴파일 안 된 **최악의 콜드스타트**를 매번 재현한다. 실사용에 가까운 조건을 보려면 `am kill`(백그라운드 프로세스만 회수, force-stop과 달리 dex/oat 캐시는 유지)로 재현해야 한다. 이 둘의 차이(3초대 vs 1.3초대)를 인지 못 하면 "왜 이렇게 느리지"를 잘못된 원인에서 찾게 된다.

### 6-3. 화면이 계속 잠들어(AOD) 자동 검증을 막음

반복 테스트 도중 기기가 여러 번 Doze/AOD 상태로 전환됐고, `adb shell input keyevent`(화면 깨우기)는 MIUI가 `INJECT_EVENTS` 권한 부족으로 차단한다([[reference_device_testing]]과 동일 제약). 자동화만으로는 못 풀고 **사용자에게 폰 화면을 한 번 깨워달라고 요청**해야 했다.

### 6-4. iOS 시뮬레이터도 스크린샷 타이밍을 못 잡을 만큼 빠르다

`xcrun simctl io booted screenshot`을 아무리 지연을 조절해도 스플래시 구간을 못 잡거나(검정 화면) 이미 콘텐츠로 넘어간 프레임만 잡혔다. `xcrun simctl io booted recordVideo` + `ffmpeg`로 영상을 찍어 프레임을 추출하니 바로 해결됐다 — Android와 동일한 처방.

### 6-5. `@capacitor/assets generate`가 요청 안 한 PWA 산출물을 만듦

이 프로젝트는 PWA를 안 쓰는데(`index.html`에 매니페스트 참조 없음) `npx @capacitor/assets generate`를 플랫폼 지정 없이 돌리면 `public/manifest.webmanifest`·`public/assets/icons/*`까지 생성한다. 스플래시 이미지를 교체할 때마다(이번 세션에서만 4회) 매번 생성됐다가 삭제해야 했다 — 아이콘 소스(`icon-*.png`)를 안 건드려도 매번 재실행되지만 icon 산출물 자체는 바이트 동일이라 git diff엔 안 잡히고, PWA 산출물만 매번 새로 생겨 치웠다.

## 요약 — 재발 방지 체크리스트

- **"WebView는 force-dark 면역"과 "WebView가 제때 그려진다"는 별개 명제다.** 전자가 참이어도 후자가 거짓이면 소용없다 — Capacitor 브릿지+플러그인 초기화가 끝나기 전엔 WebView에 뭘 그려둬도 안 보인다. 브랜드를 force-dark로부터 지키려면 **Capacitor를 안 거치는 네이티브 레이어**([[ADR-029]]의 `SplashActivity`)가 유일한 답이었다.
- **"부수 조사 결과가 좋다"를 "원래 질문 해결됨"으로 착각하지 말 것.** 타이밍이 실사용에서 괜찮다는 걸 확인했다고 "색 문제 없음"이라고 결론 내린 게 이번 세션의 가장 큰 오판이었다. 사용자가 애초에 뭘 물어봤는지 항상 재확인할 것.
- **고정밀 타이밍 검증은 스크린샷이 아니라 화면 녹화로.** Android(`screenrecord`)든 iOS(`simctl io recordVideo`)든 1초 미만 구간의 전환을 스크린샷 반복으로 잡으려 하면 반드시 놓친다.
- **콜드스타트 측정 방식(`force-stop` vs `am kill`)에 따라 결과가 3배 가까이 달라진다.** 실사용 조건을 재현하려면 후자를 써야 한다.
- **에셋 생성 도구가 플랫폼 지정 없이 돌면 안 쓰는 산출물까지 만든다.** `@capacitor/assets generate`를 돌릴 때마다 PWA 산출물 생성 여부를 확인하고 정리할 것.
- **MIUI/HyperOS의 OS 레벨 오버라이드는 이번에도 반복됐다** — force-dark(색)에 이어 상태바 아이콘 appearance(`setAppearanceLightStatusBars`)도 앱 요청을 무시했다. 이런 종류는 `true`/`false` 양쪽을 실기기로 찍어 "정말 안 되는지" 빠르게 확인하고, 안 되면 다른 OEM/AOSP를 위해 의미상 올바른 값만 남기고 더 파고들지 않는 게 효율적이다.
